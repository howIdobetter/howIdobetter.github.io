覆盖多播

覆盖多播简史

回想一下，IP 多播是在 20 世纪 90 年代和 21 世纪初开发的。21 世纪初，IP 多播的部署进展缓慢，部分原因是我们之前讨论过的那些问题。因此，涌现出了许多初创公司，包括 FastForward Networks（伯克利）、ProxyNet（伯克利）、Sightpath（MIT）和 Akamai（MIT）。它们的工作大体上彼此独立，但其解决方案都采用了基于覆盖网的多播（overlay-based multicast）这一相同的基本思想。

覆盖多播：定义

回想一下 IP 多播的一个主要问题：很难跨不同网络实现多播。在下图中，如果所有主机都是同一组的成员，不同网络中的路由器就很难协调并将数据包发送给整个组。

这里的解决方案是构建一个把主机直接彼此连接起来的虚拟网络拓扑（virtual network topology）：

我们在这里画出的虚拟链路是虚构的，在现实中并不真正对应物理链路。例如，如果 A 想沿这条虚拟链路向 D 发送数据包，该数据包仍然必须经过若干真实的路由器和链路。

不过，画出这些虚拟链路后，我们就可以假装所有主机都在一个小型局域网中彼此相连。随后，这些主机便可以运行多播路由算法，在彼此之间转发数据包。

例如，我们可以利用虚拟链路构建一棵以 D 为根的基于核心的树。然后，每个人都可以沿树的虚拟链路广播数据包，从而实现数据包的多播。

请记住，当一个数据包沿虚拟链路发送时，它仍然必须经过若干真实的路由器和链路。例如，如果 A 想把一个数据包同时转发给 D 和 B，它就必须发送两个单播数据包：“从 A 到 D”和“从 A 到 B”。这两个单播数据包都会经过若干真实的路由器和链路才能到达目的地。

在下面的例子中，A 正向 G 发送一个单播数据包。该数据包经过若干真实的路由器和链路到达 G。途中，中间主机 C 和 D 接收并转发该数据包。

直观地说，虚拟网络让我们产生一种错觉：所有主机都连接在一个小型局域网中，尽管现实中它们散布在世界各地。

从网络体系结构的角度来看，现在由终端主机（位于第 7 层）负责运行多播协议。终端主机此时充当虚拟路由器（virtual routers）。这意味着终端主机必须构建多播转发表，了解自己的出站虚拟链路（例如通过静态表项），并沿虚拟链路转发数据包。路由器完全不需要考虑多播（它们只需运行标准的单播协议）。

这与 IP 多播不同：在 IP 多播中，路由器负责运行多播协议，而终端主机不需要考虑这些协议（它们只需向组地址发送数据包）。

我们画出的这些虚拟链路构成覆盖网络（overlay network）。覆盖网络中的终端主机（虚拟路由器）彼此通信，以运行多播路由算法。覆盖网路由表以虚拟链路为基础（例如，B 的表中可能写着：如果我从 A 收到数据包，就把它转发给 C 和 D）。

负责沿虚拟链路发送数据包的真实链路和路由器构成底层网络（underlay network）。底层网络的路由器彼此通信，以运行标准的单播路由算法（例如距离向量、BGP）。底层路由表以物理链路为基础（例如，R1 的表中可能写着：到 G 的下一跳是 R2）。

为了实现覆盖网络和底层网络，我们将使用封装（encapsulation）。假设我们想把一个数据包多播到组地址。那么内层首部（覆盖网）会写着“从 A 到 G1”，主机将读取这个覆盖网数据包，以决定如何转发数据包。

假设主机 A 判定该数据包需要沿通往 C 的虚拟链路转发。那么主机 A 会用“从 A 到 C”的外层首部封装该数据包，并把这个数据包单播给 C。底层网络负责只使用外层首部，将单播数据包从 A 转发到 C。

实现覆盖网络

在最基本的模型中，覆盖网络中的节点就是终端主机（例如你的个人笔记本电脑）。这意味着终端主机需要理解多播路由协议、构建自己的转发表并转发数据包。

终端主机也可以是由某家公司部署的代理服务器（类似于 CDN 服务器）。这些机器仍然是运行多播路由协议的终端主机，但它们并非实际的用户机器（例如你的个人笔记本电脑），而是专门为了帮助支持多播路由而部署的。请注意，这些代理服务器仍然是在覆盖网中运行多播路由的终端主机。这些服务器仍需封装数据包，并通过底层网络单播这些数据包，因此它们不是第 3 层路由器。

覆盖网络的一般思想也可以用于多播以外的其他用途。例如，数据包也可以跨覆盖网络进行单播。你可以使用覆盖网拓扑构建点对点文件共享服务（peer-to-peer file sharing service）。（点对点服务是指组内任何用户都可以与任何其他用户共享文件，而不依赖存储所有文件的中央服务器。）

许多覆盖网络可以同时共存于同一个底层网络之上。从终端主机的角度看，终端主机会运行两个独立的应用。每个应用都有各自独立的转发表、相邻链路列表等。每个应用都可以提供不同的服务。

覆盖多播的优势

覆盖多播方法有什么优点？

最大的优势是易于部署。从底层路由器的视角来看，覆盖网络只是另一个收发单播数据包的应用。底层路由器和协议不需要进行任何修改。

IP 多播要求大多数或所有路由器都理解多播协议。相比之下，在覆盖多播中，只有某些参与节点（例如组内用户）需要理解该协议。其他所有终端主机都不需要进行任何修改。

每个覆盖多播应用都可以使用自己的实现或协议，因此不同应用（例如不同的组）之间不需要标准化。相比之下，在 IP 多播中，所有路由器都需要使用相同的协议，才能相互协调。

由于每个覆盖多播应用都能自行作出实现决策，这种方法也使应用可以自由定义自己的目标。

每个应用都可以决定如何绘制自己的虚拟拓扑、如何设置链路成本，以及如何计算穿过网络的路径。有些组可能更关心时延，另一些组则可能更关心吞吐量。

与 IP 多播相比，覆盖多播中的访问控制也更容易实现。可以定制路由协议的实现，只允许经过授权的用户参与协议。每个应用都可以自行决定用户获得授权意味着什么。

每个应用还可以决定自己的商业模式。可以定制路由协议的实现来跟踪使用情况，并据此向用户收费；每个应用也可以自行决定跟踪使用情况意味着什么。例如，由 CDN 服务器构成的覆盖网络可能用于向数百万用户直播体育比赛。应用本身可以跟踪哪些用户正在观看比赛，并据此向他们收费。

也存在更不寻常的商业模式。例如，点对点文件共享系统可能被用于非法传播受版权保护的材料。为了避免给用户惹上麻烦，这种系统可能希望不跟踪用户。

覆盖多播的性能

覆盖网络的性能高度依赖于你在终端主机之间绘制的虚拟拓扑。尤其是，虚拟拓扑中的链路和成本应当准确反映相应的底层拓扑。

在这个特定的覆盖网拓扑中，如果尝试计算从 A 到 G 的最短路径，我们会得到 A 到 C、再到 B、E、F，最后到 G 的路径。如果随后沿这条路径发送数据包，那么从 A 到 G 的数据包最终会在底层网络中沿一条糟糕得多的路径转发。

为了衡量覆盖网络的性能，我们可以定义伸展因子（stretch factor）。它是底层路径成本与覆盖网路径成本之比。

在上面的例子中，底层成本为 4，覆盖网成本为 1，因此伸展因子为 4。为了更好地模拟底层网络，为这条虚拟链路分配 4 的成本可能更合理。

高伸展值很糟糕，因为这意味着底层路径比相应的覆盖网路径长许多倍。理想情况下，我们希望伸展值更低（更接近 1），这意味着底层路径成本与覆盖网路径成本大致相同。

如何构建低伸展的覆盖网拓扑？有时，运营者会手动设计拓扑。

此外，也存在能够自动发现良好覆盖网拓扑的自组织协议（self-organizing protocols）。从较高层次来看，自组织协议可能像这样工作：起初，随机选择你的邻居（即从你到随机邻居绘制虚拟链路）。你会周期性地寻找新的候选邻居，并测量自己与这些新候选邻居之间的距离（例如发送一个数据包并测量往返时间）。如果最佳候选邻居的表现优于当前最差的邻居，就放弃当前最差的邻居（删除虚拟链路），并添加最佳候选邻居（添加一条新的虚拟链路）。

覆盖多播的缺点

覆盖多播会引入额外开销，从而影响性能。例如，封装和解封装数据包需要额外的时间和处理能力。

覆盖多播并非互联网的内置功能，这意味着应用开发者必须自行实现覆盖多播。相比之下，在 IP 多播中，应用开发者只需向组地址发送数据包，无需构建自己的转发表等。

尽管存在这些缺点，覆盖多播的性能已经足够好，因此如今在互联网中得到广泛部署。
