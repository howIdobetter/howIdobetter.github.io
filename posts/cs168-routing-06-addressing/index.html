寻址

扩展路由

到目前为止，我们的转发表中每个目的地都有一个条目。这种做法无法扩展到整个互联网。

如果在整个互联网上运行距离向量（distance-vector），我们就必须为互联网上的每台主机发送一则通告。如果在整个互联网上运行链路状态（link-state），每台路由器就必须知道完整的互联网网络图。在这两种情况下，只要有任何主机加入或离开互联网，我们都必须重新进行计算，才能收敛到新的路由状态。

让路由具备可扩展性的诀窍在于我们如何为主机寻址。到目前为止，我们一直用某个名称来称呼每台主机和路由器（例如 R1、R2、A、B），但在实际中，我们会采用一种更巧妙的寻址方案。

IP 寻址

回想一下，在邮政系统的类比中，我们针对不同情境采用了不同的寻址方案。邮递员使用的是诸如 2551 Hearst Ave. 这样的街道地址。楼内的秘书使用的是诸如 413 Soda Hall 这样的房间号。地址是按照某种结构化方式分配的。例如，三楼所有房间号都以数字 3 开头，四楼所有房间号都以数字 4 开头。

与邮政系统一样，互联网在每一层使用不同的寻址方案。在本节中，我们将重点讨论可用于网络层（第 3 层）路由的 IP 地址（IP address）。

网络上的每台主机（例如你的计算机、Google 的服务器）都会被分配一个 IP 地址。在本节中，你可以假定每台主机都有一个唯一的 IP 地址。

IP 地址是唯一标识一台主机的数字。与邮政系统一样，这个数字在选取时会包含一些关于主机所在位置的上下文信息。

请注意，IP 地址不一定是静态的。在类比中，如果你搬到另一所房子，你的地址就会改变。同样，如果你的计算机移动到另一个位置，它加入网络时可能会被分配一个不同的 IP 地址（而你的旧 IP 地址最终会过期）。

IP 地址的长度取决于所使用的 IP 版本。IPv4 地址为 32 位，IPv6 地址为 128 位。两个版本的路由概念相似，但我们会尽可能使用 IPv4，因为较短的地址更容易阅读。

分层寻址

回想一下，互联网是由网络组成的网络。互联网中有许多局部网络，我们在局部网络之间添加链路，从而形成更广阔的互联网。这为我们提供了一种自然的层次结构，可以用来组织寻址方案。

下面是一个直观的寻址示意。我们可以为每个网络分配一个数字。然后，在网络 3 内，可以为主机分配 3.1、3.2、3.3 等编号，其他网络中的主机也可以采用类似方式。

现在，考虑路由器 R9 中的转发表。以前，我们会为网络 1 中的每台主机分别设置一个条目，而这些条目的下一跳全都是 R6。采用分层寻址后，我们可以改为用一个条目表示整个局部网络，说明所有 1._ 地址（其中 _ 表示任意数字）的下一跳都是 R6。我们还可以说明，所有 2.* 地址的下一跳都是 R8。

这种使用通配符匹配来汇总路由的分层模型，可以让转发表变得更小。

此外，这种模型也能让转发表更加稳定。

如果网络 1 内部的拓扑发生变化，我们不需要更新 R9 的转发表（也不需要更新其他网络中的任何转发表）。实际中，局部网络内部的变化（例如新主机加入网络）远比网络之间的变化（例如铺设新的地下电缆）频繁，因此，让局部变化只影响局部转发表是一件好事。

更一般地说，我们的地址由两部分组成：网络 ID（network ID）和主机 ID（host ID）。这使域间路由协议（inter-domain routing protocol）能够关注网络 ID，以寻找网络之间的路由；也使域内路由协议（intra-domain routing protocol）能够关注主机 ID，以寻找网络内部的路由。随着网络发生变化，这也能让路由协议更加稳定。域间协议不关心网络内部的变化，而域内协议不关心其他网络中的变化。

请注意，R9 中的转发表仍然需要为其自身网络（即网络 2）内的每台主机设置条目。

同样，R4 是一台不与其他网络直接相连的内部路由器，它既需要为网络 3 内的各台主机设置条目，也需要为其他网络设置聚合条目（例如，2.* 的下一跳是 R9）。转发表的规模取决于同一网络中内部主机的数量与外部网络的数量之和。

默认路由

现在我们知道，条目可以表示整个地址范围，而不必总是表示单个地址。我们还可以进一步扩展这个想法，以提高可扩展性。

考虑 R4。它为每个外部网络（1.*、3.* 和 4.*）都设置了一个条目，而这些条目的下一跳全都是 R9。我们可以把所有外部网络聚合成一个条目。我们仍然会为每台内部主机（2.1、2.2 等）保留条目，但在最后注明：对于转发表中没有列出的所有其他主机，下一跳都是 R9。

在 R2 上，我们可以进行更为激进的聚合。同样，所有外部网络的下一跳都是 R3。不过，2.1、2.2、2.3、2.6 和 2.7 的下一跳也都是 R3。因此，转发表只需要为 2.4 和 2.5 设置静态条目。然后我们可以注明：对于转发表中没有列出的所有其他主机（包括一些内部主机和一些外部主机），下一跳都是 R3。

为了表示表中没有列出的所有主机，我们可以使用匹配一切的通配符 .。向给定目的地转发时，路由器首先检查具体主机（例如 3.1）或地址范围（例如 2._）是否匹配。如果路由器找不到任何匹配项，最终就会匹配 _.* 通配符。这称为默认路由（default route）。

大多数主机只有一条硬编码的默认路由。例如，主机 2.4 的转发表只有一个条目，说明所有流量都发送给 R2。实际中，你的家用计算机只有一个条目，说明所有流量都发送给家用路由器。这就是主机不需要参与路由协议的原因。

分配分层 IP 地址：早期互联网

为了让路由更具可扩展性，我们需要以某种分层方式分配地址。地址需要包含一些与其位置有关的信息（例如，邻近主机的地址需要有某些相同部分）。

在早期互联网中，IPv4 地址有 8 位网络 ID 和 24 位主机 ID，与我们直观模型中的情况一样。

例如，AT&T 的网络 ID 是 12，Apple 的网络 ID 是 17，而美国国防部拥有 13 个不同的网络 ID。

8 位网络 ID 意味着我们只能分配 256 个不同的网络 ID，但现实中，可能运营自己局部网络的组织远不止 256 个。此外，24 位主机 ID 意味着每个网络都会得到 2^24 = 16,777,216 个地址。小型网络（例如只有 10 名员工的公司）很可能不需要 1600 万个地址。随着互联网不断扩大，我们需要一种新的寻址方法。

分配分层 IP 地址：分类寻址

首次解决这一问题的尝试是分类寻址（classful addressing），它根据需要分配不同规模的网络。在这种方法中，地址分为 3 类，每一类分配给网络 ID 和主机 ID 的位数各不相同。开头的 1 至 3 位用于标识所使用的类别。

A 类地址以起始位 0 开头。接下来的 7 位是网络 ID（128 个网络），再接下来的 24 位是主机 ID（1600 万台主机）。

B 类地址以起始位 10 开头。接下来的 14 位是网络 ID（16,000 个网络），再接下来的 16 位是主机 ID（65,000 台主机）。

C 类地址以起始位 110 开头。接下来的 21 位是网络 ID（200 万个网络），再接下来的 8 位是主机 ID（256 台主机）。

采用这种方法后，我们可以拥有 200 万 + 16,000 + 128 个不同的局部网络。主机较多的大型组织可以获得一个 A 类网络，较小的组织则可以获得 B 类或 C 类网络。与之前一样，在同一个网络内，开头的类别位和网络 ID 位都相同，而每台主机都会获得不同的主机 ID。

分类寻址的一个主要问题是各类别的规模。A 类（1600 万台主机）对大多数组织而言过大，而 C 类（256 台主机）对大多数组织而言又过小。因此，大多数网络都需要使用 B 类。

遗憾的是，B 类网络 ID 只有 16,000 个；到 1994 年时，B 类网络即将耗尽。我们又一次需要新的寻址方法。

注意：分类寻址在现代互联网中已经过时。

注意：严格来说，每个网络的主机数要少 2，因为全零地址和全一地址保留用于特殊用途。例如，在 C 类网络中，每个网络实际上有 254 台主机，而不是 256 台。

分配分层 IP 地址：CIDR

我们的第三种分层寻址方法，也是现代互联网仍在使用的方法，是 CIDR（无类别域间路由，Classless Inter-Domain Routing）。在 CIDR 中，我们仍然采用可变长度的网络 ID，但不再只允许 3 种不同的网络 ID 长度（A 类、B 类、C 类），而是允许固定任意数量的位。

例如，考虑前面那家只有 10 名员工的小公司。在分类寻址中，它会得到一个拥有 256 个主机地址的 C 类网络。如果它只需要 10 个主机地址，我们可以给它分配更长的网络 ID，从而分配更少的地址。

如果分配 28 位网络 ID，主机 ID 的长度就是 4 位（16 个可能的地址）。如果分配 29 位网络 ID，主机 ID 的长度就是 3 位（8 个可能的地址）。我们无法恰好分配 10 个地址，但 28 位网络 ID 足以满足这家公司的需要。这里存在少量浪费（6 个未使用的地址），但仍然比分配 256 个地址好得多。

再举一个例子，考虑一个需要 450 个主机地址的组织。在分类寻址中，C 类（256 个地址）不够用，因此它会获得一个拥有 65,000 个主机地址的 B 类网络，而其中大多数地址都不会被使用。有了任意长度的网络 ID，我们可以分配 23 位网络 ID，从而留下 9 位用于主机寻址（512 个地址）。这既能满足该组织的需要，又会少浪费许多地址。

多层次的分层分配

现实中的层次结构可能有多层。例如，在一个网络内部，组织可以选择将特定的地址范围分配给特定的下属组织（例如公司或大学中的院系）。

实际中，我们会利用现实里的多层组织和地理层次结构来分配地址。ICANN（互联网名称与数字地址分配机构，Internet Corporation for Names and Numbers）是拥有全部 IP 地址的全球性组织。

ICANN 将地址块发放给代表特定国家或大洲的区域互联网注册管理机构（Regional Internet Registry，RIR）。例如，RIPE 获得欧盟的所有地址，ARIN 获得北美的地址，APNIC 获得亚洲／太平洋地区的地址，LACNIC 获得南美洲的地址，AFRINIC 获得非洲的地址。示例：ICANN 将所有以 1101 开头的地址交给 ARIN。

然后，每个 RIR 会把自己地址范围的一部分发放给大型组织（例如公司、大学）或互联网服务提供商（Internet Service Provider，ISP）。这些组织或 ISP 称为本地互联网注册管理机构（Local Internet Registry）。示例：ARIN 控制所有以 1101 开头的地址，并将所有以 1101 11001 开头的地址交给 AT&T。

最后，每个本地互联网注册管理机构会把单独的 IP 地址分配给特定主机。为了增加层次，本地注册管理机构也可以将 IP 地址范围分配给小型组织，再由这些小型组织分配单独的 IP 地址。

在每一层，新增固定的位数由待分配的地址数量决定。例如，ARIN 可能想给 AT&T 800 万个地址，并计算出固定 9 位可得到 800 万个主机地址。ARIN 已经固定了 4 位，因此它再固定 5 位，并将所有以这 9 位开头的地址分配给 AT&T。随后，AT&T 可能将前缀 1101 11001 110100010 交给加州大学伯克利分校（UC Berkeley），为其提供 16,000 个地址。随着我们向下属组织分配地址，固定的位数会越来越多，同时始终保留上级组织固定的位。

IP 地址的书写方式

我们可以把 IP 地址写成由 1 和 0 组成的 32 位序列，也可以写成一个大整数。实际中，为了便于阅读，我们把每组 8 位写成一个整数（介于 0 和 255 之间）。例如，IP 地址 00010001 00100010 10011110 00000101 可以写成 17.34.158.5。这有时称为点分四组表示法（dotted quad representation）。

到目前为止，我们一直以比特形式书写地址范围（例如所有以 1101 开头的 IP 地址）。要书写一个地址范围，可以使用斜线表示法（slash notation）。我们先写出固定前缀，然后将所有剩余的未固定位写为 0，再把得到的 32 位值转换成点分四组 IP 地址。之后，在斜线后写上固定位的数量。

例如，如果前缀是 11000000，我们在所有未固定位上补零，得到 11000000 00000000 00000000 00000000。作为 32 位地址，它是 192.0.0.0。然后，因为固定了 8 位，所以我们把这个范围写成 192.0.0.0/8。

要把单个地址写成一个范围，可以写成 192.168.1.1/32，这表示全部 32 位都是固定的。此外，默认路由 . 可以写成 0.0.0.0/0。

斜线表示法有时看起来会有些令人困惑，因为我们任意地按 8 位分组，并用十进制书写数字。例如，8 位前缀 11000000 和 12 位前缀 11000000 0000 会分别写成 192.0.0.0/8 和 192.0.0.0/12（同一个 IP 地址表示不同的范围）。再举一个例子，如果我拥有 4 位前缀 1100，就可以分配 5 位前缀 11001。作为范围，它们分别写成 192.0.0.0/4 和 200.0.0.0/5。乍看之下，很难看出第二个范围实际上是第一个范围的子集，我们必须把比特位写出来才能确认。

斜线表示法中的斜线部分（例如 /16）也可以替换为网络掩码（netmask）。与斜线后的数字一样，网络掩码告诉我们哪些位是固定的。书写网络掩码时，我们将所有固定位写成 1，把所有未固定位写成 0，再将结果转换为点分四组形式。例如，如果地址范围是 192.168.1.0/29，可以写出 29 个 1（固定位）和 3 个 0（未固定位）。11111111 11111111 11111111 11111000 的点分四组形式是 255.255.255.248。这个范围用网络掩码表示法写作 IP 地址 192.168.1.0、网络掩码 255.255.255.248（即用网络掩码替换了斜线）。

在这些讲义中，我们通常会使用斜线表示法，因为它阅读起来更加方便。实际中，网络掩码可能很有用，因为给定一个具体 IP 地址后，如果在该 IP 地址和网络掩码之间执行按位与（bitwise AND），所有主机位都会被清零，只留下网络位。

使用 CIDR 聚合路由

在最初使用网络 ID 和主机 ID 的模型中，我们可以把同一网络内的所有主机聚合为转发表中的一条路由（例如，用 2.* 表示网络 2 中的一切）。

多层次的分层寻址意味着，我们也可以把多个网络聚合为一条路由。

考虑这张网络示意图。在最初的模型中，R6 需要分别为 AT&T、UCB 和 Stanford 设置转发条目。

然而，如果采用分层寻址，那么 UCB 的地址范围（4.12.0.0/16）和 Stanford 的地址范围（4.29.0.0/16）都是 AT&T 地址范围（4.0.0.0/8）的子集。如果 AT&T 将这些地址范围分配给其下属客户 UCB 和 Stanford，就会出现这种情况。

现在，R6 只需要用一个条目表示 AT&T、UCB 和 Stanford。我们已经把两个较小的范围聚合进二者共同所属的较大范围中。

多宿主

地址范围聚合并非总是有效。假设我们添加一条从 R6 直接连接到 Stanford 的链路。

我们的聚合路由说明，所有发往 AT&T（及其下属组织）的数据包都以 R2 为下一跳。我们需要增加一个条目，说明 Stanford 的下一跳是 R7。

请注意，现在转发表中的地址范围出现了重叠。一个目的地可能匹配多个范围。为了选择路由，我们会执行最长前缀匹配（longest prefix matching），也就是使用与目的 IP 地址相匹配的最具体范围。例如，如果有一个数据包要发往 UCM 的主机，我们会使用 UCM 专用条目，因为它具有更长的 19 位前缀。尽管 9 位的 AT&T 条目也与目的地匹配，但它的前缀更短，因此我们不会使用这条路由。

如果有一个数据包要发往 UCB 的主机，我们就不能使用 Stanford 专用条目，因为它的 16 位前缀与 UCB 主机不匹配。不过，我们仍然可以使用 8 位的 AT&T 条目，它会与目的地匹配。

IPv6 简史

IPv4 地址为 32 位，这意味着我们大约有 40 亿个可用地址。这够用吗？

这张图绘制了各区域注册管理机构随时间推移剩余的未分配 IP 地址数量（y 轴）。

到 2017 年，每个机构可用的地址都不足一个 /8 地址块（即少于 2^24 = 1600 万个地址）。每个区域注册管理机构都保留了一个备用的 /8 地址块以防万一，但到 2017 年，所有机构都不得不开始使用自己的备用地址。到 2021 年，就连备用地址也即将耗尽。

趣闻：2011 年 2 月，最后一个 /8 地址块获分配时还举行了一场现场仪式，甚至还颁发了一张特别的纸质证书。

随着互联网的发展，我们开始意识到地址终有一天会耗尽。幸运的是，人们很早就发现了这个问题，并于 1998 年开发出 IPv6，以应对 IP 地址耗尽。

从根本上说，IPv6 的寻址结构与 IPv4 相同。IPv6 确实需要一些细微的实现改动，但这些改动与此处无关。

IPv6 的主要新特性是地址更长。IPv6 地址长 128 位，这意味着大约有 $$3.4 \times 10^{38}$$ 个可能的地址。这是一个天文数字，所以我们永远不会用尽地址。宇宙已经存在了 $$10^{21}$$ 秒，因此，即使每秒分配一个地址，我们仍然只会用掉全部可用地址的 0.000000000000001%。

IPv6 开发于 20 世纪 90 年代，但并没有立即得到所有计算机的采用。即使到了 2010 年，也基本上没有人使用 IPv6。截至 2024 年，约 45% 的最终用户使用 IPv6，其中大多数用户位于互联网普及程度较高的发达国家。IPv6 得到越来越广泛采用的主要原因，是 IPv4 地址即将耗尽。

为什么 IPv6 的采用速度如此缓慢？用户、服务器和互联网运营商必须升级软件和硬件（例如路由器、链路、计算机上的设备驱动程序），以支持 IPv6。路由器现在需要两张转发表，一张包含 IPv4 地址，另一张包含 IPv6 地址。

IPv6 升级必须向后兼容。如果服务器只有 IPv6 地址，那些仅支持 IPv4 的旧计算机用户便无法使用这台服务器。IPv4 和 IPv6 本质上是两个独立的寻址系统，IPv4 地址与 IPv6 地址之间无法相互转换。截至 2024 年，许多计算机仍不支持 IPv6，因此许多服务需要同时支持 IPv4 和 IPv6。

同时支持 IPv4 和 IPv6 的计算机还必须考虑使用哪一种。一个是否比另一个更好？实际中，IPv6 更快，但许多其他实现细节也可能影响你的选择。

IPv6 地址表示法

IPv6 地址通常使用十六进制而不是十进制书写。例如：

2001:0D08:CAFE:BEEF:DEAD:1234:5678:9012

是一个 IPv6 地址（32 个十六进制数字 = 128 位）。为了便于阅读，我们在每 4 个十六进制数字（16 位）之间添加冒号。

为了便于阅读，可以省略一个 4 位数字块内的前导零。例如：

2001:0DB8:0000:0000:0000:0000:0000:0001

可以缩写为 2001:DB8:0:0:0:0:0:1。

为了便于阅读，我们还可以省略一长串零，例如 2001:DB8::1。双冒号表示用 0000 补齐所有缺失的 4 位数字块。每个地址中只能这样做一次。（省略两个范围会产生歧义，因为我们不知道每个范围中分别有多少个零。）

斜线表示法仍然可以用于 IPv6。单个地址使用 /128（所有位均固定）。一个 32 位前缀可能写成 2001:0DB8::/32。

由于地址空间如此庞大，在 IPv6 中，可以将网络 ID 固定为 64 位、主机 ID 设为 64 位，而且仍然永远不会用尽网络 ID 或主机 ID。事实上，存在一些特殊协议，让网络和主机可以自行选择 64 位网络 ID 和主机 ID（并检查是否有其他对象正在使用它），而不需要由某个组织分配具体 ID。

实际中，区域注册管理机构通常向 ISP 分配 32 位前缀，而 ISP 通常向组织分配 48 位前缀。随后，组织可以向更小的子网分配 64 位前缀。在 IPv6 中，我们通常看不到长于 /64 的前缀。即便是组织内部最小的子网，也拥有 64 位前缀，以及用于寻址特定主机的 64 位。使用这些标准化的前缀长度，可以让前缀包含更多信息。例如，在 IPv6 中，很难看出 /19 前缀代表什么，但在 IPv6 中，我们知道 /32 前缀通常代表一个 ISP。TODO 再次确认这一点
