TLS：安全字节流

安全字节流

TCP 本身对于网络攻击者来说是不安全的。网络上的某人（例如恶意路由器、在线路上嗅探数据包的攻击者）可能会在传输过程中读取甚至修改您的 TCP 数据包。

此外，使用 TCP，您可能会连接到攻击者而不是真实服务器。假设您要连接到银行网站，并且您对 www.bank.com 进行 DNS 查找。攻击者（例如侵入解析器或路由器的人）更改 DNS 响应，以便将 www.bank.com 映射到攻击者的 IP 地址 6.6.6.6。现在，当您与银行网站建立 TCP 连接时，您正在与攻击者交谈。您最终可能会将您的银行密码发送给攻击者！

为了解决这些安全问题，我们在 TCP 之上添加了一个新协议传输层安全性 (TLS)。

TLS 可以被视为第 4.5 层协议，位于 TCP 和 HTTP 等应用程序协议之间。 （我们使用像 4.5 这样奇怪的数字，因为过时的第 5 层和第 6 层与安全性无关。）TLS 依赖于 TCP 的字节流抽象，因此它不会考虑单个数据包或数据包丢失/重新排序。 TLS 为应用程序提供了与 TCP 完全相同的字节流抽象，但字节流现在可以安全地抵御网络攻击者。这就是为什么 HTTP 和 HTTPS 在语义上是相同的协议。唯一的区别是 HTTPS 在 TLS-over-TCP 的安全字节流上运行，而 HTTP 在没有 TLS 的原始 TCP 上运行。

为了区分 HTTPS 和 HTTP，我们对 HTTP 连接使用端口 80，对 HTTPS 连接使用端口 443。服务器可以通过回复所有端口 80 请求并重定向到使用端口 443 来强制用户使用 HTTPS。

TLS 握手

在较高级别上，TLS 使用加密技术来加密通过字节流发送的消息。 TLS 还使用其他加密协议（消息身份验证代码）来防止攻击者在通过网络发送消息时更改消息。

为了加密流量，TLS 必须从额外的握手开始，以交换密钥并验证服务器的身份（例如真正的银行，而不是冒充银行的人）。

由于 TLS 构建在 TCP 之上，因此 TCP 三向握手首先正常进行。这会创建一个（不安全的）字节流，允许所有未来的消息（包括 TLS 握手）继续进行，而无需考虑各个数据包。

现在可以继续进行 TLS 握手：

  1. 客户端和服务器互相打招呼。 hello 包含随机数，这确保每次握手都会产生不同的密钥。 （如果我们每次都使用相同的密钥，并且攻击者攻击了我们并了解了该密钥，那就糟糕了。）hello 还允许客户端和服务器就使用的特定加密协议达成一致。客户端的 hello 列出了客户端支持的所有加密方案，服务器的 hello 选择一种来使用。

  2. 服务器发送真实性证书。这将允许客户端验证它是否正在与真实服务器而不是冒充者进行通信。客户端如何实际验证此证书有点复杂，我们在此不讨论。

  3. 客户端和服务器导出一个只有他们两个知道的秘密。由于此时字节流仍然不安全，因此他们需要一个加密协议来支持通过不安全的通道共享秘密。我们不会在这里讨论细节，但如果您熟悉 RSA 公钥加密（例如来自加州大学伯克利分校的 CS 70），那么这是这里可以使用的一种可能的加密方案。客户端使用服务器的公钥加密秘密并将其发送到服务器。只有服务器知道相应的私钥，并且能够解密消息并获悉秘密。

  4. 客户端和服务器根据共享密钥和来自 hello 的随机值导出密钥。使用秘密可确保攻击者无法获悉秘密密钥。使用随机值可确保我们每次都派生出不同的密钥。此推导由客户端和服务器在本地独立完成。秘密密钥实际上从未通过网络发送，因此攻击者没有机会了解它们。

  5. 客户端和服务器交换一些确认以确认它们派生了相同的秘密，并且到目前为止没有人篡改通过网络发送的消息（因为字节流仍然不安全）。

至此，握手完成，以后的所有消息都用密钥加密（消息验证码也用于防止篡改）。我们现在已经在 TCP 连接之上建立了安全字节流，应用程序可以在我们的安全字节流之上交换数据。
