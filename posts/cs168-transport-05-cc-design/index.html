拥塞控制设计

基于主机的算法概览

大多数基于主机的算法都采用相同的总体思路，它们的差异体现在三个关键选择点上。

每个源端都独立地在循环中反复执行以下逻辑：尝试以速率 R 发送一段时间。然后询问：在这段时间内是否经历了拥塞？如果是，就降低 R；如果不是，就提高 R。

其中还缺少一个环节：最初应该以什么速率开始发送？我们需要某种方法来选择初始速率 R。

三个关键选择点分别是：如何选择初始速率？如何检测拥塞？每次应该增加和减少多少？

检测拥塞

发送方如何检测网络是否拥塞？有两种常见方法。

发送方可以检查是否发生丢包。这是 TCP 常用的方法。其优点是信号明确无歧义：每个数据包要么被标记为丢失（超时或重复确认），要么没有丢失。此外，TCP 为了重发数据包，本来就会检测丢包，因此无需从头重新实现这一机制。

这种方法也有缺点，因为有时丢包源于数据损坏（校验和错误），而非拥塞。事实上，当链路并不拥塞、却频繁损坏数据包时，TCP 会产生误判并表现不佳。数据包重排序也可能迷惑 TCP：一个迟到的数据包可能被误认为已经丢失。

这种方法的另一个主要缺点是发现拥塞太晚。等到数据包开始被丢弃时，路由器队列已经满了，数据包也已受到延迟。

除了检查丢包，发送方还可以通过检查数据包延迟来检测拥塞。发送方可以测量从发送数据包到收到该数据包确认（ack）之间的时间。如果发送方发现延迟正在增加，这可能就是拥塞的迹象。

过去，人们一直认为准确测量延迟很困难。数据包延迟会随队列大小和其他流量而变化。许多年来，基于延迟的方法并未得到广泛部署；不过近年来，Google 的 BBR 协议（2016 年）证明了基于延迟的算法是可行的，一些服务（例如 Google 的服务）也已采用这类算法。

发现初始速率

连接刚开始时，我们必须确定发送数据包的初始速率。可以通过一个探测过程来了解初始速率：尝试几种不同的速率，以估计可用带宽。

我们希望这个探测过程是安全的，因此应该从较低的速率开始。我们不想一开始就用数据包淹没网络。

同时，为了提高效率，我们也希望探测过程能够迅速发现可用带宽。为此，每次后续尝试都要快速提高速率。如果探测过程耗时很久，就会浪费本可用于以最优速率发送数据包的时间。例如，假设每 100 ms 将速率增加 0.5 Mbps，直到检测到拥塞（丢包）。如果可用带宽为 1 Mbps，探测阶段需要 2 次迭代，即 200 ms。然而，如果可用带宽为 1 Gbps = 1000 Mbps，那么探测阶段需要 2000 次迭代，即 200 秒，才能提升到合适的速率，这显然太久了。互联网上的链路速率差异很大，因此这两种情况在现实中都可能出现。

为了既能缓慢起步又能快速提升，我们每次将带宽乘以一个倍数（而不是增加一个固定值）。这种方案称为慢启动（slow start），尽管这个名称可能不太直观。在慢启动中，我们从一个很小、几乎总是远低于实际带宽的速率开始。随后以指数方式提高速率（例如每次翻倍），直到遇到丢包。可安全使用的速率是发生丢包之前的速率（不能使用曾经发生丢包的速率）。形式化地说，如果在速率 R 时发生丢包，那么安全速率就是 R/2。

调整：对拥塞作出反应

回想一下，在探测阶段之后，我们仍将不断调整带宽，因为网络本身在变化，可用带宽并非常数。

最后一个选择点是：如果检测到拥塞，应该将带宽减少多少；如果没有检测到拥塞，又应该将带宽增加多少。

我们的决定会影响主机适应可用带宽变化的速度，进而决定带宽利用的效率。如果需要很长时间才能适应变化并找到合适速率，我们就会长时间以次优带宽运行，造成效率低下。适应缓慢还可能引发公平性问题。例如，如果我正在使用一条链路的全部带宽，而另一条连接刚刚建立，我就需要迅速适应并降低自己的带宽，以便分享链路。

回想一下，拥塞控制算法的主要目标是效率（使用全部可用带宽）和公平（各连接平等分享带宽）。我们需要选择能够同时实现这两个目标的增加和减少规则。

可以选择哪些规则？从高层看，我们可以快速或缓慢地作出反应。更具体地说，快速变化采用乘法方式，例如每次迭代将速率翻倍或减半；缓慢变化采用加法方式，例如每次迭代给速率加 1 或减 1。这些选择形成四种可能方案：

AIAD：加性增、加性减（additive increase, additive decrease）

AIMD：加性增、乘性减（additive increase, multiplicative decrease）

MIAD：乘性增、加性减（multiplicative increase, additive decrease）

MIMD：乘性增、乘性减（multiplicative increase, multiplicative decrease）

在这四种方案中，事实证明 AIMD（缓慢增加、快速减少）最有利于实现效率和公平。

直观地说，AIMD 是一个合理的选择，因为发送过多比发送过少更糟。速率过高时会造成拥塞，数据包会被丢弃；速率过低时虽然无法用满带宽，但至少不会造成拥塞。

AIMD 的行为是：没有拥塞时缓慢提高速率，逐步逼近最大带宽；一旦超过最大带宽并检测到拥塞，就迅速降低速率。这样，大多数时间里速率会略低（更可取）；而当速率过高（不可取）时，又能迅速降低以避免拥塞。

调整：模型

为什么 AIMD 最有利于实现效率和公平？下面进行更详细的分析。

首先，请注意，四种方案在实现效率方面都相当不错。低于最优速率（不拥塞）时提高速率，高于最优速率（拥塞）时降低速率，长期来看，速率应始终在最优速率附近波动。

然而，事实证明，四种方案中只有 AIMD 能实现公平。为了解释原因，考虑一个简单模型：两条连接经过一条容量为 C 的链路，它们分别以速率 X1 和 X2 发送。我们知道，如果 X1+X2 大于 C，网络就会拥塞；如果 X1+X2 小于 C，网络就处于负载不足状态。

为实现效率，我们希望链路得到充分利用，即 X1+X2 = C。为实现公平，我们希望 X1 = X2，使两条连接平等分享容量。

为了将各种可能性可视化，考虑一张二维图，其中 x 轴表示 X1（用户 1 的速率），y 轴表示 X2（用户 2 的速率）。图中的每一点都代表一种可能情形，即每个用户分别以某个特定速率发送。

假设 C=1。为了实现最大效率，我们希望 X1+X2 = 1。可以在图中画出这条直线。直线上的每一点都用满了全部可用带宽。

我们知道，当 X1+X2 大于 1 时，网络拥塞。在图中，这个不等式对应直线上方的半平面。我们还知道，当 X1+X2 小于 1 时，网络未得到充分利用，对应直线下方的半平面。因此，直线上方的所有点都表示拥塞状态，直线下方的所有点都表示利用不足状态。

为了实现公平，我们希望 X1 = X2。也可以画出这条直线。该直线上的每一点都表示公平状态，即两个用户使用相同数量的带宽。不在这条直线上的点都不公平。

理想状态位于两条直线的交点，即 X1 = X2 = 0.5。这个点同时位于两条直线上，因此既公平又高效。

点 (0.2, 0.5) 效率低，因为只使用了 0.7 的带宽；从图形上看，它位于效率线下方。点 (0.7, 0.5) 表示拥塞，因此位于效率线上方。点 (0.7, 0.3) 是高效的（位于效率线上），但不公平（不在公平线上）。

回想一下，在动态调整算法中，每个发送方都独立运行相同的算法来决定自己的速率。这意味着，如果两个用户检测到利用不足，二者都会以相同方式提高速率（采用加法还是乘法，取决于所选规则）。同样，如果两个用户检测到拥塞，二者都会以相同方式降低速率。

如果两个用户都以加法方式增加或减少速率，会发生什么？如果二者都给速率加上 b，状态 (x1, x2) 就变为 (x1+b, x2+b)。如果二者都从速率中减去 a，状态 (x1, x2) 就变为 (x1-a, x2-a)。

在图中，进行加法变化时，表示状态的点会沿斜率为 1 的直线移动。

如果两个用户都以乘法方式增加或减少速率，会发生什么？乘以 c 会把 (x1, x2) 变换为 (cx1, cx2)，除以 d 会把 (x1, x2) 变换为 (x1/d, x2/d)。

在图中，进行乘法变化时，表示状态的点会沿斜率为 x2/x1 的直线移动。等价地说，这就是连接 (x1, x2) 与原点 (0, 0) 的直线。

现在可以把这个模型应用于四种增减方案，看看它们会让点靠近公平线还是远离公平线。我们的目标是让点在调整速率时逐渐接近公平线。

调整：AIAD 的动态过程

考虑每次增加时加 1，每次减少时减 2。假设容量 C = 5。那么从某个起点开始，点会按如下方式移动：

X1 = 1, X2 = 3（起点，4 小于 5，增加）

X1 = 2, X2 = 4（6 大于 5，减少）

X1 = 0, X2 = 2（2 小于 5，增加）

X1 = 1, X2 = 3

我们回到了起点！最初的分配并不公平，几次迭代之后，又回到了同一个不公平分配。

事实上，如果观察 X1 与 X2 之差（公平时差值为 0），会发现每次迭代中的差值始终相同（均为 2）。迭代没有让分配变得更公平，也没有让它变得更不公平。

可以从图形上看到这种行为。从某个起点开始，如果以加法方式增减，就总会沿斜率为 1 的直线移动，永远不会更接近公平线。

不过请注意，点确实会按预期在效率线附近振荡。四种方案都会有这种行为。

也可以从代数上理解这种行为。假设 X1 和 X2 相差 5（分配不公平）。如果给 X1 和 X2 加上相同的数，所得 X1’ 和 X2’ 仍相差 5（同样不公平）。如果从 X1 和 X2 中减去相同的数，结果也一样。

总之，这种方法无法缩小公平性差距。如果分配最初不公平，它就会一直不公平。

你可能会问：如果给 X1 增加得多一些（例如 +2），给 X2 增加得少一些（例如 +1）呢？请记住，去中心化的方法意味着每个人都在运行相同的算法。从实际角度说，一台主机也无法知道，相对于其他主机，自己应该增加多少。

调整：MIMD 的动态过程

考虑增加时翻倍，减少时除以 4。容量仍为 C = 5。从某个起点开始，前几次迭代如下：

X1 = 0.5, X2 = 1（1.5 小于 5，增加）

X1 = 1, X2 = 2（3 小于 5，增加）

X1 = 2, X2 = 4（6 大于 5，减少）

X1 = 0.5, X2 = 1

我们又回到了起点，公平性没有得到任何改善！

可以在图中看到这种行为。以乘法方式增减速率时，我们沿当前点与原点之间的直线移动，永远不会更接近公平线。

从代数上说，可以考虑 X2 与 X1 的比值，即 X2/X1（公平时比值为 1）。在上面的例子中，该比值始终为 2，也就是说，X2 的带宽始终是 X1 的两倍。即便把 X1 和 X2 同时乘以或除以某个常数，这一比值也不会改变。调整无法让我们更接近公平比值 1。

调整：MIAD 的动态过程

这一种稍微复杂一些。考虑增加时翻倍，减少时减 1。令 C = 5，前几次迭代如下：

X1 = 1, X2 = 3（4 小于 5，增加）

X1 = 2, X2 = 6（8 大于 5，减少）

X1 = 1, X2 = 5（6 大于 5，减少）

X1 = 0, X2 = 4（4 小于 5，增加）

X1 = 0, X2 = 8

此时，X1 的带宽为零。以后每次增加时即使翻倍，X1 的带宽仍为零。我们实际上制造了最不公平的情形：X2 占据全部带宽，而 X1 一无所有。

更一般地说，如果从不公平的分配开始，MIAD 会让分配变得更加不公平，最终达到一个人占据全部带宽、另一个人带宽为零的状态。

从代数上看，可以考虑 X1 与 X2 之间的差距。增加时翻倍，差距也会翻倍，从 (X2 - X1) 变为 (2 X2 - 2 X1) = 2(X2 - X1)。但同时从 X1 和 X2 中减去 1 时，差距保持不变。差距只会增大或保持不变；经过足够多次增减迭代，它将达到最不公平的状态（一个人的带宽永远为零）。

调整：AIMD 的动态过程

最后，考虑增加时加 1，减少时减半。令 C = 5，前几次迭代如下：

X1 = 1, X2 = 2（3 小于 5，增加）

X1 = 2, X2 = 3（5 不大于 5，增加）

X1 = 3, X2 = 4（7 大于 5，减少）

X1 = 1.5, X2 = 2（3.5 小于 5，增加）

X1 = 2.5, X2 = 3（5.5 大于 5，减少）

X1 = 1.25, X2 = 1.5（2.75 小于 5，增加）

X1 = 2.25, X2 = 2.5（4.75 小于 5，增加）

X1 = 3.25, X2 = 3.5（6.75 大于 5，减少）

X1 = 1.625, X2 = 1.75（小于 5，增加）

X2 = 2.625, X2 = 2.75

可以看到 X1 和 X2 正在逐渐接近；事实上，它们正在逼近公平分配 X1 = X2 = 2.5。

从代数上也可以看到，X1 与 X2 之间的差距正在缩小。具体来说，给两个数加上同一个常数时，差距保持不变；但将两个数都减半时，差距也会减半，从 (X1 - X2) 变为 (X1 / 2 - X2 / 2) = (X1 - X2) / 2。随着增加与减少交替进行，差距会不断减半并趋近于 0。

也可以从图形上看到这一点。进行乘性减时，我们沿经过原点的直线移动。这条直线朝公平线倾斜，沿它向下移动意味着正在接近公平线。与之前一样，加性增不会让我们更接近公平线，因为我们沿斜率为 1 的直线移动（与公平线平行）。但关键在于，加法也不会让我们离公平线更远。两种操作要么使我们更接近，要么既不接近也不远离。经过多次迭代，点会缓慢移向公平线。

总结来说：AIAD 和 MIMD 会保留不公平状态，无法改善公平性；MIAD 会加剧不公平；AIMD 则会收敛到公平状态。
