路由器辅助的拥塞控制

借助路由器进行拥塞控制

前面我们看到了基于主机的拥塞控制算法存在的一些问题。借助路由器，其中许多问题都可以得到解决！

TCP 会混淆拥塞与数据损坏。TCP 会填满队列，速率起伏不定，并且在短流上的表现很差；这一切都是因为主机需要不断调整速率来检测拥塞。如果路由器能够把拥塞情况告知发送方，甚至直接告诉发送方理想速率，那么其中许多问题都能得到解决。

此外，如果由路由器强制执行公平共享，主机作弊就会困难得多。

从更偏哲学的角度看，让路由器参与拥塞控制也是一种自然的设计选择。拥塞发生在路由器上，因此路由器掌握的拥塞信息往往比主机更多。

路由器辅助的拥塞控制（router-assisted congestion control）可以非常有效，并带来近乎最优的性能（高链路利用率、低时延），但部署这些协议可能颇具挑战。路由器现在需要支持额外功能，而且这些功能有时会相当复杂。有些协议甚至可能要求每一台路由器都同意添加相应功能。

强制执行公平排队

路由器怎样才能确保每条连接都获得自己的公平份额？

到目前为止，路由器接收数据包，必要时将其排队，然后按照先进先出（FIFO）的顺序发送。路由器并不关心数据包来自哪条连接。

在新模型中，路由器需要按连接对数据包进行分类。（暂且假设所有连接都是 TCP 连接。）这意味着路由器必须查看数据包内部，获知源 IP 地址、目的 IP 地址以及端口。

为了对公平性作出形式化定义，路由器可以为每条连接维护一个单独的队列。数据包到达时，路由器将其加入对应的队列。随后，每次发送时，路由器只需选取一个队列，并发送该队列前端的数据包。只要路由器以某种公平方式选择队列，就能在各连接之间强制实现公平。

如果所有数据包大小相同，路由器可以用轮询（round-robin）方式选择队列：先从第一个队列发送，再从第二个队列发送，以此类推。事实证明，即使并非所有连接都需要相同带宽，这种方法仍然有效。有些连接将数据包放入队列的速度可能比其他连接慢。如果对带宽需求不同的连接采用轮询服务，如何计算分配给每条连接的带宽？例如，假设我们每秒可以发送 10 个数据包，而 A、B、C 每秒分别送来 8、6、2 个数据包。

如果采用轮询方式发送，每种数据包每秒会分别发出多少个？我们可以把它建模成资源分配问题并加以求解。

例如，假设链路容量为 10。连接 A 请求 8，B 请求 6，C 请求 2。应该如何在这三条连接之间分配容量？如果试图做到公平，每条连接都应获得 3.33。但 C 只请求了 2，所以就把它请求的 2 分给 C，不再多给。

现在还剩下 8，而 A 和 B 仍需要分配。如果讲求公平，它们各自会获得 4。这个数值低于它们的请求，但我们无法满足其全部请求，因此给每条连接分配公平份额 4。

形式化地定义最大最小公平（max-min fairness）：假设 CCC 是路由器可用的总带宽。每条连接 rir_iri​ 都有一个带宽需求，我们必须为每条连接分配带宽 aia_iai​。最大最小公平的带宽分配为 ai=min⁡(f,ri)a_i = \min(f, r_i)ai​=min(f,ri​)，其中 fff 是满足 ∑ai=C\sum a_i = C∑ai​=C 的唯一值（对所有连接都取同一个值）。在这个方程中，最小值项保证任何连接获得的带宽都不超过其请求，而求和约束则保证没有带宽闲置。直观地说，fff 是平均分配给每条连接的公平份额（所以所有连接都使用同一个 fff 值）。

还可以这样理解这个方程：存在某个神奇的公平份额数值，可以平均分给每条连接。如果你的请求低于公平份额，你就得到自己请求的数量，不会额外多得；如果你的请求高于公平份额，你获得的数量会被限制在公平份额，但其他任何连接获得的数量都不会超过你。

在前面的例子中，fff 为 4。A 和 B 得到 fff（它们想要更多），C 得到 2（它想要的更少）。

采用最大最小公平时，方程保证：如果你没有获得全部所需带宽，那么其他任何连接获得的带宽都不会超过你。在假设数据包大小相等的情况下，轮询方法符合最大最小公平。

如果不假设数据包大小相等呢？现实中，数据包大小可能相差悬殊，例如 40 字节与 1500 字节。理想情况下，我们希望执行逐比特轮询（bit-by-bit round robin），轮流从每条连接的队列中发送一个比特。这并不实用（我们不会一次只发送一个比特），但如果在理论上这样做，就可以为每个数据包写下其最后一个比特发出的时间。我们把这个时间称为该数据包的截止时间（deadline）。于是，一种公平的近似方法就是按照截止时间的先后顺序发送数据包，也就是按照其最后一个比特在理想情况下发出的时间排序。

实践中的公平排队

公平排队（fair queuing）有什么优点？它能确保连接之间相互隔离，并防止作弊连接获得更多带宽。连接不必实现 TCP（或某种 TCP 友好的替代方案），而可以选择自己的拥塞控制算法，即便这种算法可能并不友好。

从根本上说，公平排队的优势在于，它能够抵御作弊和 RTT 差异等外部因素的影响。无论发生什么，每条连接都能获得给定链路的公平份额。不过，我们仍需要端系统发现并适应自己的公平份额，例如请求过多时主动减速。

公平排队有什么缺点？它比 FIFO 排队复杂得多。计算截止时间的过程很棘手，而这里并未给出相应算法。此外，路由器还需要维护多个队列，并对每个数据包执行额外的解析工作。

实践中，我们无法在路由器上实现完美的公平排队，因为它过于复杂，难以在高速下运行；不过，确实存在近似算法，例如亏缺轮询（Deficit Round Robin）。现代路由器通常会实现近似方案，但使用的队列数量更少。队列较少意味着不会为每条连接分别设置队列，隔离的粒度也就更粗，例如为每位客户设置一个队列。

公平排队无法消除拥塞，它只是另一种管理拥塞的方式。例如，考虑这条瓶颈链路：它可能为每条连接分配 0.5 Gbps，从而挫败作弊。但是，如果上方连接以 0.5 Gbps 运行，那么其中 0.4 Gbps 的流量会在紧接着的下一条链路被丢弃。更好的分配方式是沿上方连接发送 0.1 Gbps，沿下方连接发送 0.9 Gbps。

根本问题在于，这条瓶颈链路不知道后续（下游）链路将发生什么。解决这个问题的唯一方法是让发送方主机减速，路由器排队对此无能为力。

公平排队为我们提供了按连接的公平性，但从更偏哲学的角度看，我们仍要追问这是否是正确的公平模型。正如前面所见，按连接实现公平意味着拥有更多连接的人仍会获得更多带宽。我们是否应该改为对每个源—目的地对执行公平，或许按每个源执行公平？是否应该惩罚使用更多拥塞链路的连接，因为它们占用了更多稀缺资源？

路由器辅助的拥塞控制

公平排队会在特定链路上强制实现公平，却不会向发送方传递任何信息。如果路由器把信息传回发送方，帮助发送方调整速率，又会怎样？

一种方案是让路由器直接告诉发送方应该使用的速率。我们可以在数据包中添加一个速率字段，让路由器用该连接的公平份额填写这个字段。当数据包返回发送方时，发送方可以读取首部，并按照路由器给出的数值设置速率。这样，发送方就不必通过动态调整来寻找合适的速率。

另一种方案是让路由器向发送方通知拥塞，而不指定确切速率。这种方案已经以 IP 首部中的显式拥塞通知（Explicit Congestion Notification，ECN）位得到部署。如果一个数据包经过拥塞的路由器，路由器就把该位设为 1。当接收方收到 ECN 位已开启的数据包时，确认回复也会设置 ECN 位，于是发送方得知发生了拥塞。

路由器何时设置这个位，有许多种选择。路由器可以非常谨慎，频繁设置该位，这会降低时延，但可能导致链路利用不足；路由器也可以更为冒进，很少设置该位，这会增加时延，却能带来很高的链路利用率。

当这个位被设置时，主机如何响应也有许多种选择。例如，主机可以假装数据包已经丢失，并据此作出调整。

ECN 有什么优点？它解决了混淆数据损坏与拥塞的问题。它使路由器可以更早地向主机发出拥塞警告，例如在队列尚未填满时就发出警告，从而降低时延。它的实现也很轻量。

实践中，要让 ECN 有效，大多数或所有路由器都必须支持这一协议，并在必要时开启该位。在现代互联网中，部分路由器部署了 ECN 位，但并非全部。不过，在所有路由器都同意启用该位的小型网络中，例如数据中心的局域网，ECN 位可以发挥作用。
