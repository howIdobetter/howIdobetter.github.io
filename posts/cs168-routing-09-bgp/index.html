边界网关协议（BGP）

BGP 简史

最低成本路由协议与图论中的最短路径问题密切相关，而早在互联网出现之前，计算机科学领域就已经开始研究这个问题。Dijkstra 算法诞生于 1956 年，Bellman-Ford 算法诞生于 1958 年。在开发早期路由协议时，设计者可以借鉴这些算法的思想。

早期的互联网是一个由政府资助的项目，网络由美国国防部集中控制。当时还不存在自治系统（autonomous system，AS）的概念，并且最低成本算法能够扩展到早期互联网那样的小规模。后来，随着互联网不断发展，政府将控制权移交给不同的商业实体，而这些实体不得不在实践中逐步开发域间路由协议（inter-domain routing protocol）。

与早期的最低成本路由协议不同，让每个自治系统都拥有自己的私有策略这一概念，在计算机科学中没有先例。域间路由协议背后的思想只能随着这些新兴互联网公司的需求，在实践中逐步形成。

BGP 创建于 1989 至 1995 年，其临时应需式的开发过程意味着这个协议并不完美。如果今天能够从头重写该协议，结果可能会有所不同。不过，该协议已被证明既有效又有韧性，至今仍是正在使用的域间路由协议。（请记住，每个人都必须同意使用同一种域间路由协议，因此这样的协议只有一个。）

BGP 基于距离向量

回想一下，我们见过两类域内路由算法：距离向量算法（distance-vector algorithm）和链路状态算法（link-state algorithm）。在设计 BGP 时，哪一类算法更适合作为设计的起点？

请记住，在 BGP 中，我们需要尊重各个 AS 的隐私。如果使用链路状态协议，那么每个 AS 都必须将自己的策略告知整个网络，让所有人都掌握完整信息，以便自行计算路由。

此外，在 BGP 中，我们还需要尊重自治性，允许每个 AS 自行作出策略决策。然而，链路状态协议要求所有人以某种一致的方式计算路由（例如，所有人都同意使用最低成本路径）。

链路状态算法并不尊重 AS 的隐私或自治性，因此以链路状态算法为基础设计 BGP 并不是一个好选择。相比之下，距离向量允许每个 AS 自行决定接受或拒绝哪些路由，以及通告哪些路由。此外，由于距离向量并非全局协议，每个 AS 无须了解其他所有 AS 的策略，也能计算出有效路由。

距离向量协议中的许多核心思想仍然适用于 BGP。我们发送和接收的通告（advertisement）仍然针对单个目的地。与前几节一样，我们将考虑单个目的地的通告与路由，但要知道，该协议实际上会同时针对多个目的地运行。

在距离向量协议和 BGP 中，每个 AS 都只使用从收到的通告中获得的信息来计算路由，而看不到网络拓扑的全貌。此外，在这两类协议中，AS 都会持续不断地发送和接收通告，直到所有 AS 都收敛（converge）到一组路由。

BGP 遵循与距离向量协议相同的核心思想，但术语略有变化。我们不再说每个 AS 宣布或通告路由，而是说该 AS 正在 导出（exporting） 路由。随后，每个 AS 监听通告并选择自己偏好的路由，我们称之为 导入（importing） 路由。

距离向量是一个很好的起点，但还缺少什么？

距离向量协议旨在寻找最低成本路由，但在 BGP 中，我们希望根据各个 AS 自身的策略来决定路由。

基于策略的导入与导出

从高层来看，为了支持策略，我们将改变导入和导出路由的规则。每个 AS 只会导出（通告）自己喜欢的路由（依据其策略）。此外，在导入（选择）路由时，AS 将依据策略而非距离来选择最佳路由。

当一个 AS 收到针对同一目的地的多个通告时，它不再选择最短路由，而是依据策略选择（导入）一条路由。

请记住，通告从目的地向外传播，而报文则被转发得越来越靠近目的地（方向与通告相反）。导入决策决定了一个 AS 将其出站流量发送到哪里。例如，如果 S 从 A、B 和 C 处听到关于同一目的地的通告，那么 S 的导入决策（A、B 或 C）就决定了发往该目的地的数据包会被转发到哪里。

在距离向量协议中，当我收到一个通告并安装一条新路由时，总会把这条新路由通告给我的所有邻居。

现在，AS 拥有自己的策略，可以选择是否参与一条路由。如果某个 AS 有一条自己可能不喜欢的路由，它现在可以选择不将该路由导出给某些邻居。

例如，假设我的策略是不想承载 C 的流量。这可能是出于经济原因，也可能是我作出的其他策略决定。当我接受一个通告并安装一条路由后，不把该路由通告给 C 也是可以的。

再次提醒，数据流动的方向与通告相反。导出决策决定了一个 AS 愿意承载哪些入站流量。如果我导出一条路由，就表示我同意参与这条路由，并允许其他人沿这条路由把数据包转发给我。

这条规则带来的一个后果是：即使底层图是连通的（任意两个节点之间都存在一条路径），也不能保证每个 AS 都能到达其他所有 AS。在实践中，我们可以通过对 AS 的策略和 AS 图的结构建立一些约定来保证可达性。

实施 Gao-Rexford 规则

一般而言，BGP 支持任意策略，但任意策略无法保证互联网完全连通（数据包能够从任意源到达任意目的地）。

回想一下，Gao-Rexford 规则（Gao-Rexford rules）基于常见的、以经济关系为基础的导入与导出策略，规定了一组限制更严格的策略。没有人强制要求 AS 必须遵守这些规则。不过，如果各个 AS 同意遵守这些规则，我们就能对互联网连通性作出更有力的假设。

简史：这些规则以 20 世纪 90 年代供职于 AT&T 的 Lixin Gao 和 Jennifer Rexford 命名。当时，每个 AS 都在实践中自行制定策略。Gao 和 Rexford 调查了各个 AS 的策略，据此归纳出这些规则，并用它们证明了关于互联网的若干保证。

在导入路由时，Gao-Rexford 规则规定，AS 优先导入由客户通告的路由，其次是由对等方（peer）通告的路由，最后是由提供商（provider）通告的路由。

在实践中，除 Gao-Rexford 规则外，AS 还会实施额外的决胜规则（tiebreaking rule）。例如，如果我收到来自两个客户的通告，就需要某种额外的决胜规则来偏好其中一个。性能是一种常见的决胜因素，我们会选择带宽更高或路径更短的路由。

根据 Gao-Rexford 规则，我们应该如何导出路径？回想一下，如果一条路由中至少有一侧的邻居是客户，AS 就会同意参与该路由。因此，AS 应当只通告这样的路由：如果生成的路由被接受，那么该路由的一侧有一个客户邻居。

让我们逐一考察所有具体情况。

我从一个客户处收到并安装了一条路由。这意味着该路由的下一跳（next hop）就是这个客户。我应该把这条路由导出给谁？我已经能够保证一侧有一个客户向我付费，因此可以将这条路由导出给所有人（客户、提供商和对等方）。

我从一个对等方处收到并安装了一条路由（下一跳是一个对等方）。我应该把这条路由导出给谁？目前还没有人向我付费，所以我应该只把这条路由导出给客户。如果我把这条路由导出给一个对等方或提供商，而对方接受了它，那么我就创建了一条两侧都没有人向我付费的路由。

类似地，如果我从一个提供商处收到并安装了一条路由，就应该只把这条路由导出给客户，因为我需要至少有一侧向我付费，而提供商并不会付费给我。

Gao-Rexford 规则使我们能够严格证明以下陈述成立：假设 AS 图是分层且无环的，并且所有 AS 都遵守 Gao-Rexford 规则，那么我们就能保证稳态下的可达性与收敛性。

逐一解释该陈述中的具体术语：可达性（reachability）意味着图中的任意两个 AS 都能通信。收敛性（convergence）意味着所有 AS 最终都会停止更新其路径，网络将进入任意两个 AS 之间都存在有效路径的稳态（steady state）。“在稳态下”意味着，如果网络拓扑发生变化，路径可能需要一段时间才能完成改变并再次达到稳态。

回想一下，分层（hierarchical）意味着从任意 AS 出发，沿层次结构向上移动（从客户移向提供商），最终都会到达一个一级 AS（Tier 1 AS）。无环（acyclic）意味着客户—提供商关系（有向边）中不存在环。

该陈述的证明要求所有 AS 都遵守 Gao-Rexford 规则。如果各个 AS 运行各自的任意策略，这些保证就不再成立。

修改：BGP 聚合目的地

我们还需要对距离向量协议作出两项修改。

在距离向量协议中，我们说明了每个目的地都有唯一地址，而转发表（forwarding table）将每个目的地映射到一个下一跳和距离。

在 BGP 中，每个 AS 都用一个前缀（prefix）来寻址，该前缀表示该 AS 内的所有机器共享同一个前缀。

这些转发表可能变得非常庞大（设想一个提供商有数百个客户），而且每个目的地都需要在单独的通告中描述。有没有办法更简洁地表达这个转发表？

为提高可扩展性，BGP 允许 AS 将多个目的地聚合（aggregate）到单个转发表项中，并通告一个包含所有这些目的地的、更一般化的前缀。

请注意，在实践中，BGP 对所通告前缀的大小有约定。例如，AS 不会为单个 IP 地址发布通告。24 位前缀（包含 256 个地址的地址块）通常是通告地址时采用的最小单位。

修改：路径向量协议

在距离向量等最低成本协议中，我们无须担心环路。每台路由器都在尝试寻找最低成本路由，而根据定义，最低成本路由不会包含环路。

现在，每个 AS 都依据自己的偏好选择路由，我们便失去了不存在环路这一保证。例如，假设 B 喜欢经过 C 的路径，而 C 喜欢经过 B 的路径。这样就产生了路由环路（routing loop）！

为解决这个问题，BGP 通告不再包含到目的地的距离，而是包含通往目的地的完整 AS 路径。这使协议从距离向量协议变成了路径向量（path-vector）协议。

例如，在距离向量协议中，A 会通告：“我能以成本 1 到达目的地。”随后，B 会通告：“我能以成本 2 到达目的地。”

在路径向量协议中，A 会通告：“我能通过路径 [A] 到达目的地。”随后，B 会通告：“我能通过路径 [B, A] 到达目的地。”

经过这项修改，AS 可以通过逐一查看通告中的路径，判断所通告的路径是否包含环路。具体来说，如果我收到一个通告，只需检查路径是否包含我自己。如果包含，数据包就会被发回给我，从而形成环路，因此我会忽略该通告，既不接受也不通告这条带有环路的路由。

注意：如果所有人都同意丢弃带有环路的路由，就能保证通告不会包含环路。所通告的路由能够形成环路的唯一情形，就是我看到一条已经包含我自己的路由，而再次加入我自己正是形成环路的原因。

从距离向量改为路径向量，也使 AS 能够实施任意策略。在距离向量协议中，我可能有一项策略，比如“尽可能避开 AS#2063”。如果我收到一条“我能以成本 12 到达目的地”的通告，就完全无法知道所通告的路径是否经过 AS#2063。相反，如果通告包含完整路径，我就可以先检查路径是否经过 AS#2063，再决定接受还是拒绝它。

注意：我们之前看到的传统 BGP 导入策略（相比经过对等方和提供商的路由，优先选择通往客户的路由）只取决于下一跳，而不取决于整条路径。尽管如此，改用路径向量仍有助于检测环路，并让我们能够将协议推广到任意策略。

桩 AS 使用默认路由

有些 AS 无须运行 BGP 来确定如何在网络中转发数据包。特别是，如果一个桩 AS（stub AS）只连接到一个提供商，那么所有以其他 AS 为目的地的数据包都应发送给这唯一的提供商。该桩 AS 可以为其他 AS 中的所有目的地安装一条硬编码的默认路由（default route）。

那么，其他 AS 尝试向这个桩 AS 发送数据包时怎么办？桩 AS 可以请求提供商安装一条静态路由（static route），告诉提供商如何把数据包发送到这个桩 AS。这样，提供商就可以运行 BGP，并向互联网的其余部分通告这条静态路由。桩 AS 可以请求提供商把静态路由硬编码下来，而桩 AS 本身永远不必运行 BGP，因为提供商会代表桩 AS 通告通往它的路由。

互联网中的大多数小型 AS 都是使用默认路由和静态路由的桩 AS。

桩 AS 类似于域内路由中的端主机（end host）。它们为自己的 AS 发送和接收数据包，但不替其他 AS 转发数据包，也不参与路由过程。与域内路由中的处理方式一样，我们通常会忽略桩 AS，只考虑真正参与 BGP 的传输 AS（transit AS）。
