链路状态协议

链路状态协议简介

回想一下，根据底层算法的不同，路由协议可以分为不同类别。在上一节中，我们了解了距离向量（distance-vector）这一类协议。本节将讨论另一类主要协议——链路状态（link-state）。

回想一下，协议还可以分为外部网关协议（exterior gateway protocol，在网络之间运行）和内部网关协议（interior gateway protocol，在网络内部运行）。与距离向量协议一样，链路状态协议通常也是内部网关协议。

IS-IS（Intermediate System to Intermediate System，中间系统到中间系统）和 OSPF（Open Shortest Path First，开放最短路径优先）是链路状态协议的两个主要例子。如今，两者都得到了广泛部署。

链路状态概述

距离向量执行的是一种分布式协同计算。每个节点根据其邻居计算出的结果，计算出属于自己的那部分解。所有节点的计算共同构成完整的解。在计算过程中，每个节点只需要来自邻居的局部信息（节点并不知道完整的网络图）。

相比之下，链路状态协议执行本地计算。每个节点都独立地从头计算完整的解，不使用邻居的任何计算结果。不过，要做到这一点，每个节点都需要来自网络各处的全局信息。

用一句话概括链路状态协议：每台路由器都会获知完整的网络图，然后在图上运行最短路径算法（shortest-path algorithm），以填充转发表（forwarding table）。

我们必须实现两个主要步骤。首先，路由器需要以某种方式获知完整的网络图，包括每条链路的状态（连通或断开）、每条链路的成本，以及每个目的地的位置。

然后，路由器需要在该图上运行某种算法，以了解如何将数据包转发到每个目的地。

我们先考虑第二步（最短路径），再考虑第一步（获知网络图）。

计算路径

一旦路由器拥有网络的全局视图，就可以使用某种最短路径算法轻松计算穿过网络的路径。

具体来说，路由器应当计算到每个目的地的最短路径。然后，对于每个目的地，路由器记录最短路径上的下一跳（next hop），就像距离向量协议中所做的那样。转发时不需要路径的其余部分。

这一步可以使用许多单源最短路径（single-source shortest-path）算法。例如，贝尔曼–福特算法（Bellman-Ford algorithm，串行版本，不包含距离向量协议中的任何改动）和 Dijkstra 算法（Dijkstra’s algorithm）都能高效计算从单一源点到所有目的地的最短路径。我们也可以考虑广度优先搜索（breadth-first search）等替代方案，或能够并行运行的算法。

我们必须留意的一件事是路由器之间的不一致。

请记住，每台路由器都在独立计算最短路径，并据此决定下一跳。每台路由器只能控制自己的下一跳，无法影响下一跳会做什么。

例如，假设 R3 计算出这条到 A 的最短路径，并决定将数据包转发给 R2。随后，R2 计算出这条到 A 的最短路径，并决定将数据包转发给 R3。两台路由器计算出的都是有效的最短路径，但它们的决定却造成了路由环路（routing loop）。

为了避免这一问题，我们必须确保所有路由器做出的转发决定彼此兼容。要让所有路由器做出兼容的决定，需要满足哪些条件？

  1. 所有路由器都必须对网络拓扑（network topology）达成一致。假设某条链路发生故障，但只有一台路由器知道。那么，不同路由器实际上是在完全不同的图上计算路径，因而可能产生不一致的结果。

  2. 所有路由器都在寻找路径中的最低成本路径。如果某台路由器出于某种原因偏好成本更高的路径，我们就会得到不一致的结果。

  3. 所有成本都为正。负成本可能产生负权环。

  4. 所有路由器都使用相同的平局决胜规则。如果我们假设最短路径是唯一的，那么前两个条件足以确保每台路由器都选择相同的路径。这个条件还进一步保证：如果有多条路径并列为最短路径，所有路由器都会选择同一条。

满足这四个条件后，路由器即使使用不同的最短路径算法，仍会计算出相同的路径并做出兼容的决定。不过在实践中，为了简单起见，路由器通常都使用相同的算法。

获知图拓扑

路由器如何获知完整的网络图？首先，我们需要知道自己的邻居是谁（包括路由器和目的地）。然后，我们需要将这些信息分发到整个网络。路由器还需要将收到的所有信息拼接成一幅图拓扑。

为了发现邻居，每台路由器都会向自己的所有邻居发送一条问候消息（hello message）。

例如，在这个网络中，R2 向它的两个邻居发送：“你好，我是 R2。”现在，R1 知道自己连接到了 R2，R3 也知道自己连接到了 R2。类似地，R1 向 R2 问好，于是 R2 现在知道了 R1。同样，R3 向 R2 问好，因此 R2 也知道了 R3。

这样一来，每台路由器现在都知道自己的直接邻居是谁。请注意，R1 并不知道 R3，因为 R1 和 R3 不是邻居。

我们还希望知道链路是否断开。为了支持这一点，我们会定期重新发送问候消息。如果某个邻居停止问候（例如漏掉了一定数量的问候消息），我们就认为它已经消失。

既然我们已经知道了自己的邻居，就应该向所有路由器通告这一事实。为了进行全局通告，我们将通告发送给所有邻居。此外，每当我们收到一条通告时，也应将它发送给自己的所有邻居。这可以确保每条消息都传播到整个网络。这称为在网络中洪泛（flooding）信息。如果有任何信息发生变化（例如某个邻居消失），我们也应洪泛该信息。

我们还需要确保消息不会丢失。否则，其他路由器可能错过一次更新，并在错误的图上计算路径。为了解决这个问题，我们采用与距离向量协议中相同的办法，定期重新发送消息。只要链路正常工作，在经过足够多次尝试后，我们的消息就应该能够发出。

避免无限洪泛

我们必须谨慎处理通告在网络中的洪泛方式。

R2 获知某些信息后，将其通告给邻居 R3。当 R3 收到这些信息时，它会向邻居 R2 发出通告。当 R2 收到这些信息时，它又会向邻居 R3 发出通告。这两台路由器陷入了相互通告的循环，浪费带宽，尽管其中并没有任何新信息。

请注意，这与为了可靠性而定期重新发送消息并不相同。为了保证可靠性，我们可能每 5 秒重新发送一次消息。在这个无限循环中，路由器以最大速率接收并重新发送重复通告（例如每秒数百万次）。

如果网络中包含环路，问题会更加严重：

时间步 1：R1 向 R2 和 R3 广播。

时间步 2：R2 向 R1 和 R3 广播。R3 向 R1 和 R2 广播。

时间步 3：R1、R1、R2 和 R3 分别向 (R2, R3)、(R2, R3)、(R1, R3) 和 (R1, R2) 广播。请注意，R1 在时间步 2 收到了两条消息，因此它进行了两次广播。

时间步 4：R1、R1、R2、R2、R2、R3、R3、R3 分别向 (R2, R3)、(R2, R3)、(R1, R3)、(R1, R3)、(R1, R3)、(R1, R2)、(R1, R2)、(R1, R2) 广播。

时间步 5：R1 进行 6 次广播，R2 进行 5 次广播，R3 进行 5 次广播。

所有新信息都在时间步 1 被获知。但是，每台路由器仍不断重新发送相同的信息，重复通告呈指数级增长，并最终压垮网络。

为了解决这个问题，我们需要确保路由器不会将相同的信息发送两次。

第一次看到一条消息时，将该消息发送给所有邻居，并记下我们已经见过这条消息。（无论如何，我们都必须记下这条消息，因为我们正试图利用这些信息构建网络图。）然后，如果以后再次看到同一条消息，就不要再发送第二次。

为了唯一标识一条消息，我们可以引入时间戳（或某种对每条消息都唯一的其他计数器）。

现在，如果回到之前的例子：

时间步 1：R1 向 R2 和 R3 广播。

时间步 2：R2 向 R1 和 R3 广播。R3 向 R1 和 R2 广播。

时间步 3：此时，R1、R2 和 R3 以前都已经见过这条消息，因此不会再次发送它。之后不再发送更多重复消息。

请注意，经过这项修改后，有时仍会发送重复消息，但我们已经避免了无限发送重复消息。

收敛

在每台路由器都获知完整的网络拓扑并据此计算其转发表之后，链路状态协议就会收敛（convergence）到有效的最低成本路由状态。收敛依赖于每个节点使用相同的图。收敛后，只要网络拓扑不发生变化，路由状态就会保持有效。

一旦网络拓扑发生变化，网络可能需要一段时间才能再次收敛。我们必须等待变化被检测到（例如链路故障）。然后，还必须等待新信息在网络中传播，并等待路由器重新计算转发表条目。在网络收敛期间，我们可能处于无效的路由状态，因为一些路由器使用旧图，而另一些路由器使用更新后的图。路由状态中可能存在死路、环路，或者并非最低成本的路径。

例如，假设 R3-A 链路发生故障。R3 知道这一点，但其他路由器并不知道。R3 会将数据包转发给 R1。然而，R1 仍会将数据包转发给 R3。

链路状态协议的大部分复杂性都体现在细节上。为了确保更快地收敛，并尽可能避免无效路由，我们可以对协议进行一些小的优化和调整。

链路状态与距离向量

与距离向量协议相比，链路状态协议有哪些优点和缺点？

在距离向量协议中，当我们收到一条通告时，并不一定知道自己所接受路径的全部细节。我们必须相信邻居在通告中的说法。相比之下，在链路状态协议中，我们知道完整的图拓扑，因此对数据包所经过的路径了解得更多。

视实现方式而定，距离向量协议的收敛速度可能较慢。如果网络发生变化，我们必须等待邻居重新计算并再次通告一条路径，之后才能更新自己的转发表。然后，我们的所有邻居都必须等待我们，如此继续。相比之下，在链路状态协议中，每台路由器都可以快速洪泛新信息，并同时重新计算。

链路状态协议很适合小型局部网络，但无法很好地扩展到全球互联网。具体来说，链路状态协议要求每台路由器都了解整个网络。在全球互联网中，运营商可能不愿向竞争对手透露自己的网络拓扑（例如路由器的位置、链路的带宽）。

在实践中，大多数网络会组合部署距离向量协议和链路状态协议。
