DNS

DNS 有什么用途？

本节将介绍一些常见的应用（第 7 层）；它们运行在我们目前构建的各层之上。

我们首先介绍的应用是 DNS。它是一种运行在第 1～4 层之上的协议，提供名称解析这一重要的网络功能。

互联网通常用两种不同方式建立索引。人类使用 google.com 和 eecs.berkeley.edu 这样便于阅读的名称来指代网站，计算机则使用 172.217.4.174 和 23.195.69.108 这样的 IP 地址。DNS，即域名系统（Domain Name System），就是在二者之间进行转换的协议。

DNS 简史

要理解 DNS 为何采用这样的设计，回顾它的发展历史会很有帮助。

在最初的互联网及其前身 ARPANET 上，有三种关键应用。当时万维网和 Web 浏览器尚未出现，大多数应用都在命令行中运行。

远程终端（telnet）允许用户远程连接另一台机器，并在那台远程计算机上运行命令。你可能听说过 SSH，它是这一协议的现代安全版本。

文件传输允许用户在本地计算机和远程计算机之间传输文件。你可能听说过 FTP，它是用于文件传输的应用层协议。

电子邮件允许用户互相交换消息。现代电子邮件在浏览器中带有 Web 客户端，但最初用户必须在终端输入类似 mail alice@46.0.1.2 的命令，其中 46.0.1.2 是收件人的 IP 地址，alice 是那台机器上的收件人用户。

在这三种情况下，执行操作都需要指定远程主机。但正如我们所说，记住远程主机的 IP 地址很困难，也不方便用户使用。

解决这个问题的第一次尝试，是为每个 IP 地址分配一个主机名（hostname）。每台计算机都有一个名为 hosts.txt 的文件，把每个主机名映射到其 IP 地址。例如，可以把主机名 ucb-arpa 映射到 10.0.0.78，并把 mail mosher@10.0.0.78 改写成 mail mosher@ucb-arpa。

这个概念实际上沿用至今。如果你曾在自己的计算机上启动服务器，它的 IP 地址很可能是 127.0.0.1，主机名是 localhost。在浏览器中输入二者中的任何一个，结果都相同。

我们必须确保不同计算机上的 hosts.txt 完全相同。如果你换到另一台计算机并输入 mail mosher@ucb-arpa，理应仍然把邮件发给同一个人。主机文件最初由一个人（Elizabeth Feinler）维护，并通过实际复印纸质文档在用户之间传递。

最初的纸质主机文件可供人阅读。它把主机名映射到地址，还包含用户全名、所运行的协议（例如 TCP、FTP），甚至电话号码等信息。

所有人都认同（1973 年的 RFC606 中也提到）这种情形十分荒唐。拿到一份纸质文件后，你还得手动把它输入计算机。此外，文件以纸张形式传阅，所以你拿到的可能已经过时。

第一项改进是让列表可供机器读取。这样至少可以不再传递纸质副本，而使用 FTP 等协议共享文件。但这仍无法扩展：我们不能永远要求一个人维护该文件；而且随着文件变大，下载可能非常缓慢，网络连接一旦中断，你还可能只得到不完整的文件。

DNS 于 1983 年首次被提出（RFC882），用来解决这些问题。此后它经历了一些修改，但其基本系统一直沿用至今。

趣闻：第一个用于在 Unix 上运行 DNS 服务器的软件是 BIND（1984 年，加州大学伯克利分校），如今它仍然相当常见。

DNS 的设计目标

这段简史揭示了 DNS 的一些设计目标，我们应当牢记这些目标。

DNS 必须能够扩展。互联网上有数量庞大的主机，每秒都会执行海量查询，主机还可能频繁加入或移除。

DNS 必须具备高可用性、轻量且快速。几乎每个互联网连接都以一次 DNS 查询开始，把主机名转换为 IP 地址。因此 DNS 必须极快，否则每个连接都会被拖慢。此外，它不能存在单点故障，否则一处故障就会让互联网停止工作。

名称服务器

如果能有一台集中式服务器保存每个域到每个 IP 地址的映射，并供所有人查询，那当然很好。遗憾的是，没有任何服务器既大到足以存储互联网上每个域的 IP 地址，又快到足以处理全世界产生的 DNS 请求量。因此，DNS 使用许多名称服务器（name server）组成的集合；这些服务器专门负责响应 DNS 请求。

每台名称服务器负责一个特定的域区域（zone），因此没有任何一台服务器需要存储互联网上的所有域。例如，负责 .com 区域的名称服务器只需回答以 .com 结尾的域查询，不必存储任何与 wikipedia.org 有关的 DNS 信息。同样，负责 berkeley.edu 区域的名称服务器无需存储任何与 stanford.edu 有关的信息。

名称服务器虽然用途特殊（响应 DNS 请求），但和互联网上可联系的任何其他服务器一样：每台都有便于人类阅读的域名（例如 a.edu-servers.net）以及便于计算机读取的 IP 地址（例如 192.5.6.30）。注意不要混淆域名和区域。例如，这台名称服务器的域名中含有 .net，但它响应的是 .edu 域的 DNS 请求。

名称服务器层级

你可能会注意到这个设计存在两个问题。第一，.com 区域虽然比整个互联网小，但让一台名称服务器存储所有以 .com 结尾的域仍不现实。第二，如果名称服务器很多，计算机怎么知道应该联系哪一台？

DNS 引入了一个新思路来同时解决这两个问题：查询名称服务器时，它不必总是返回所查询域的 IP 地址，也可以把你引向另一台名称服务器以获取答案。这样，负责 .edu 等大区域的名称服务器就能把查询重定向到负责 berkeley.edu 等较小区域的名称服务器。于是，.edu 区域的名称服务器不必存储 eecs.berkeley.edu、math.berkeley.edu 等信息；它只需存储 berkeley.edu 名称服务器的信息，并将这些请求重定向到一台 berkeley.edu 名称服务器。

DNS 根据名称服务器负责的区域，把所有名称服务器组织成树状层级：

树最顶层的根服务器（root server）把所有域都纳入自己的区域（该区域通常写作 .）。树中较低层级的名称服务器负责更小、更具体的区域。

DNS 查询（概念）

DNS 查询总是从根开始。根会把查询引向它的某个子名称服务器；随后你查询该子服务器，它再把你重定向到自己的某个子服务器。这个过程不断重复，直到你查询到一台知道答案的名称服务器，它会返回与你的域对应的 IP 地址。

为了把你重定向到子名称服务器，父名称服务器必须提供子服务器的区域、便于人类阅读的域名和 IP 地址，以便你联系该子服务器获取更多信息。

例如，对 eecs.berkeley.edu 的一次 DNS 查询可能包含以下步骤。（howdns.works 提供了这一查询的漫画版本。）

  1. 你对根名称服务器说：请告诉我 eecs.berkeley.edu 的 IP 地址。
  2. 根服务器对你说：我不知道，但可以把你引向另一台掌握更多信息的名称服务器。它负责 .edu 区域，便于人类阅读的域名是 a.edu-servers.net，IP 地址是 192.5.6.30。
  3. 你对 .edu 名称服务器说：请告诉我 eecs.berkeley.edu 的 IP 地址。
  4. .edu 名称服务器对你说：我不知道，但可以把你引向另一台掌握更多信息的名称服务器。它负责 berkeley.edu 区域，便于人类阅读的域名是 adns1.berkeley.edu，IP 地址是 128.32.136.3。
  5. 你对 berkeley.edu 名称服务器说：请告诉我 eecs.berkeley.edu 的 IP 地址。
  6. berkeley.edu 名称服务器对你说：好的，eecs.berkeley.edu 的 IP 地址是 23.185.0.1。

得到答案后，我们可以把它存入缓存，这样以后再次需要这条记录时就不必重新询问。

存根解析器与递归解析器

最初，终端主机（例如你的计算机）会自行执行 DNS 查询，逐一联系各名称服务器。

如今，本地计算机通常把 DNS 查询任务委托给DNS 递归解析器（DNS Recursive Resolver），由它代为查询名称服务器。执行查询时，计算机上的DNS 存根解析器（DNS Stub Resolver）向递归解析器发送查询，让后者完成全部工作，再从它那里接收响应。

我们怎样得知解析器的 IP 地址？第一次连接互联网时，某个服务可以告诉你解析器地址。你也可以手动输入自己希望使用的解析器地址。

互联网上一些知名解析器包括 Cloudflare 运营的 1.1.1.1 和 Google 运营的 8.8.8.8。它们往往使用容易记忆的 IP 地址，免得我们还要用名称指代它们；否则就得通过 DNS 查询来寻找它们的 IP 地址，而这些服务器存在的目的恰恰是替我们执行 DNS 查询。

除科技公司外，互联网服务提供商（ISP）也会运营解析器，把它作为销售给客户的互联网服务的一部分。（如果你付费购买互联网服务，却仍不得不在浏览器中输入 IP 地址，那就太糟了。）

家用路由器也可以充当解析器。使用物理距离较近的路由器可以加快 DNS 查询，因为你和路由器之间的延迟更低。

解析器的一大优势是缓存效果更好。它会处理许多不同终端主机的查询（不只来自你的计算机），因此能建立大得多的缓存。如果你向解析器询问 eecs.berkeley.edu，而不久前有人问过同样的问题，解析器无需联系其他名称服务器便能直接给出答案。

请注意，尽管递归解析器保存着更大的缓存，存根解析器仍可维护自己独立的缓存。有些查询甚至不用询问递归解析器，就能由存根解析器的缓存回答。

冗余

到目前为止，我们一直在说“那台 berkeley.edu 名称服务器”，但实际上每个区域都有多台名称服务器。一个区域的所有名称服务器在功能上相同，每台都能回答该区域内的任何查询。

这保证了该区域的可用性：一台名称服务器宕机后，其他服务器仍能继续回答查询。按照惯例，一个区域必须有两台名称服务器，不过实际中大多数区域至少有三台。

通常会指定其中一台为真正管理该区域的主服务器，其余是辅助镜像服务器，只存储并提供主服务器信息的副本。

因此在 DNS 查询中，名称服务器可能这样回复：“我不知道，但你应该询问 .edu 区域。该区域有 13 台名称服务器，下面是每台的域名和 IP 地址。”随后你可以选择下一台要联系的名称服务器。

DNS API

现在我们已经对 DNS 有了概念性认识，接下来看看它实际如何实现。

首先，程序员怎样利用 DNS 执行查询？

执行 DNS 查询的应用程序接口（API）相对简单、常见且稳定，在不同编程语言中也相当类似。

在 C 标准库中，可以用 gethostbyname("foo.com") 查询 foo.com 对应的 IP 地址。不过，该函数仅支持 IPv4，因此现已弃用（尽管你仍会看到有人使用它）。

更新后的版本同样位于 C 标准库中：getaddrinfo("example.com", NULL, NULL, &result) 会查询 example.com 的 IP 地址，并把答案存入 result 结构体。不必太在意具体细节和额外参数（这里设为 null）。这个替代 API 不只支持 IPv4，也支持 IPv6 等。

作为程序员，你无需操心递归解析器或具体名称服务器等 DNS 的复杂细节，只需调用标准库函数。这些函数通常会向操作系统中的存根解析器发出请求。

DNS 使用 UDP，而非 TCP

从根本上说，DNS 是一种客户端—服务器协议。一方（客户端）发送问题，另一方（服务器）返回答案。客户端通常是用户主机或递归解析器，服务器通常是名称服务器。客户端和服务器应当如何组织消息格式？

DNS 使用 UDP（尽力而为的数据包，不进行 TCP 握手）来保持轻量和快速。我们不必等待 TCP 三次握手完成，也不关心数据包是否按序到达，因为查询和响应通常都能装入一个 UDP 数据包。

使用 UDP 时，服务器无需为每个连接保存状态（与 TCP 不同，TCP 服务器必须维护缓冲区）。每个数据包都可独立处理。这也有助于保持 DNS 轻量，因为名称服务器会收到大量请求，为每个请求打开新连接代价很高。

UDP 不可靠且数据包可能丢失，该怎么办？一个简单的重试机制即可解决：如果在某个时限内没有收到回复，就重新发送查询。超时时间因操作系统而异，有时会相当慢。

正因为 UDP 不可靠且超时缓慢，使用距离近、联系可靠的解析器非常重要。例如，家用路由器中的解析器离你很近，而且很可能相当可靠（家庭网络中不会太拥塞）。还可以设置多个备用解析器来提高可靠性（例如家用路由器和 8.8.8.8）。

如前所述，DNS 查询和响应通常能放入一个数据包。一个明显例外是主名称服务器与辅助名称服务器之间传输区域时。辅助服务器必须说：“把你的所有记录给我，这样我才能帮你提供它们。”响应很可能非常大，所以这类传输通常通过 TCP 完成。

DNS 近来的进展加入了安全功能（例如阻止攻击者修改传输途中的记录），这也可能要求从 UDP 切换到 TCP。

回想一下，UDP 通过端口支持同一台服务器上的多个应用。按照惯例，所有名称服务器都在 UDP 端口 53 上监听 DNS 查询。这个端口号众所周知且固定不变，所有人才能联系名称服务器上的正确端口。

DNS 消息格式

第一个字段是 16 位的标识字段（identification field），每次查询随机选择，用于匹配请求和响应。发送 DNS 查询时，ID 字段会填入随机位。由于 UDP 无状态，DNS 响应必须在 ID 字段中原样返回这些位，原查询发送方才能知道该响应对应哪次 DNS 查询。

接下来的 16 位留给标志。QR 位指明消息是查询（该位为 0）还是响应（该位为 1）。RD 位表示你希望解析器执行递归查询，还是只返回名称服务器给出的内容（即使内容是“我不知道”）。

理论上可以在标志中指定查询类型，不过 IQUERY 类型已经废弃，STATUS 类型也没有真正定义，所以几乎所有情况都使用 QUERY 类型。

标志还可以表明查询是否成功（例如查询成功时，回复会设置 NOERROR 标志；查询一个不存在的名称时，回复会设置 NXDOMAIN 标志）。

下一个字段指定所问问题的数量（实际中始终为 1）。再之后的三个字段用于响应消息，指定消息中所含资源记录（resource record，RR）的数量。稍后会深入说明各类 RR。

消息的其余部分包含 DNS 查询或响应的实际内容。这些内容始终组织为一组 RR，每条 RR 都是带有关联类型的键值对。

完整地说，DNS 记录的键正式定义为三元组 <Name, Class, Type>，其中 Name 是实际的键数据，Class 对互联网而言始终是 IN（获取 DNS 自身信息的特殊查询除外），Type 指定记录类型。DNS 记录的值包含 <TTL, Value>，其中 TTL 是生存时间（记录可缓存多少秒），Value 是实际的值数据。

DNS 中有三种主要记录。A 类型记录（A type record）把域映射到 IPv4 地址，键是域，值是 IP 地址。类似地，AAAA type records 把域映射到 IPv6 地址。NS 类型记录（NS type record）把区域映射到域，键是区域，值是域。

你有时也会遇到 CNAME 类型记录，用于别名或重定向。其名称和值都是域，该记录表示两个域拥有相同的 IP 地址。

本节要点：每个 DNS 数据包都有一个 16 位随机 ID 字段、一些元数据和一组资源记录。每条记录属于四个类别之一（问题、答案、权威、附加），并包含类型、键和值。记录类型包括 A 和 NS。

DNS 查询（实现）

现在来逐步完成一次真实的 DNS 查询，寻找 eecs.berkeley.edu 的 IP 地址。你可以在家使用 dig 工具 尝试；记得设置 +norecurse 标志，以便自行展开递归过程。

每次 DNS 查询都从根服务器开始。根服务器的名称和 IP 地址通常以根提示文件的形式硬编码进解析器。感兴趣的话，可以查看这份根提示文件：https://www.internic.net/domain/named.root。

第一台根服务器的域名是 a.root-servers.net，IP 地址是 198.41.0.4。我们可以用 dig 向该地址发送 DNS 请求，询问 eecs.berkeley.edu 的 IP 地址。

$ dig +norecurse eecs.berkeley.edu @198.41.0.4

;; Got answer:
;; ->>HEADER<<- opcode: QUERY, status: NOERROR, id: 26114
;; flags: qr; QUERY: 1, ANSWER: 0, AUTHORITY: 13, ADDITIONAL: 27

;; QUESTION SECTION:
;eecs.berkeley.edu.          IN   A

;; AUTHORITY SECTION:
edu.                172800   IN   NS   a.edu-servers.net.
edu.                172800   IN   NS   b.edu-servers.net.
edu.                172800   IN   NS   c.edu-servers.net.
...

;; ADDITIONAL SECTION:
a.edu-servers.net.  172800   IN   A    192.5.6.30
b.edu-servers.net.  172800   IN   A    192.33.14.30
c.edu-servers.net.  172800   IN   A    192.26.92.30
...

答案的第一部分显示了头部信息，包括 ID 字段（26114）、返回标志（NOERROR），以及各部分返回的记录数。

问题部分（question section）包含 1 条记录（可查看头部的 QUERY: 1 验证）。它的键是 eecs.berkeley.edu，类型是 A，值为空。这表示我们查询的域（值为空是因为我们不知道对应的 IP 地址）。

答案部分（answer section）为空（头部中为 ANSWER: 0），因为根服务器没有直接回答查询。

权威部分（authority section）包含 13 条记录。第一条以 .edu 为键、NS 为类型、a.edu-servers.net 为值。这是根服务器给出下一批可联系名称服务器的区域和域名。该部分的每条记录都对应一台接下来可能询问的名称服务器。

注意：名称最靠前（按字母或数字顺序）的服务器通常是主服务器（这里为 a.edu-servers.net），其余是镜像。另外，多条记录使用同一名称（.edu）并无问题；这只表示有多台名称服务器都能回答该区域的查询。

附加部分（additional section）包含 27 条记录。第一条以 a.edu-servers.net 为键、A 为类型、192.5.6.30 为值。这是根服务器通过把权威部分中的域映射到 IP 地址，向我们提供下一台名称服务器的 IP 地址。

权威部分和附加部分合在一起，提供下一台名称服务器的区域、域名和 IP 地址。为保持 DNS 消息的键值结构，这些信息被分散在两个部分中。

完整地说：172800 是每条记录的 TTL（生存时间），这里设为 172,800 秒，即 48 小时。IN 是互联网类，基本可以忽略。有时你会看到类型为 AAAA 的记录，对应 IPv6 地址（通常的 A 类型记录对应 IPv4 地址）。

合理性检查：接下来查询哪台名称服务器？我们怎么知道它位于哪里？要向它查询什么？\footnote{查询 a.edu-servers.net；由于附加部分中的记录，我们知道它的位置。仍像之前一样查询 eecs.berkeley.edu 的 IP 地址。}

$ dig +norecurse eecs.berkeley.edu @192.5.6.30

;; Got answer:
;; ->>HEADER<<- opcode: QUERY, status: NOERROR, id: 36257
;; flags: qr; QUERY: 1, ANSWER: 0, AUTHORITY: 3, ADDITIONAL: 5

;; QUESTION SECTION:
;eecs.berkeley.edu.           IN   A

;; AUTHORITY SECTION:
berkeley.edu.        172800   IN   NS   adns1.berkeley.edu.
berkeley.edu.        172800   IN   NS   adns2.berkeley.edu.
berkeley.edu.        172800   IN   NS   adns3.berkeley.edu.

;; ADDITIONAL SECTION:
adns1.berkeley.edu.  172800   IN   A    128.32.136.3
adns2.berkeley.edu.  172800   IN   A    128.32.136.14
adns3.berkeley.edu.  172800   IN   A    192.107.102.142
...

下一次查询的答案部分同样为空；权威部分中的 NS 记录和附加部分中的 A 记录给出了负责 berkeley.edu 区域的名称服务器域名和 IP 地址。

$ dig +norecurse eecs.berkeley.edu @128.32.136.3

;; Got answer:
;; ->>HEADER<<- opcode: QUERY, status: NOERROR, id: 52788
;; flags: qr aa; QUERY: 1, ANSWER: 1, AUTHORITY: 0, ADDITIONAL: 1

;; QUESTION SECTION:
;eecs.berkeley.edu.         IN   A

;; ANSWER SECTION:
eecs.berkeley.edu.  86400   IN   A   23.185.0.1

最后一次查询终于在答案部分以一条 A 类型记录给出了 eecs.berkeley.edu 对应的 IP 地址。

实际中，递归解析器会尽可能缓存答案，所以大多数查询都可以跳过最初几步，使用缓存的记录，而不必每次询问根服务器和 .edu 之类的高层名称服务器。缓存能减少把域名转换成 IP 地址时必须通过网络发送的数据包，从而加快 DNS；它也能减轻最高层名称服务器的请求负载。

了解 DNS 如何实现查询后，我们可以看看 DNS 在实践中究竟如何运作，并探讨 DNS 在现实世界中的商业层面。

DNS 权威层级

我们画出的树状层级实际上体现了 DNS 的三种不同层级。

如前所见，DNS 名称具有层级性，因此 eecs.berkeley.edu 这样的域名由点号分隔的多个词组成。

我们也已经看到 DNS 基础设施具有层级性。可以把名称服务器组织成树状层级，每台名称服务器只了解树中自己负责的部分。

第三种层级是权威，它告诉我们由谁定义树中存在的名称。例如，运营 .edu 名称服务器的组织负责 .edu 区域中的所有域。随后，.edu 组织可以把自己区域中部分范围的权威委托给树中的下级组织。

例如，.edu 组织可以说：“berkeley.edu（及其所有子域）属于我的区域，我把这部分区域的控制权移交给加州大学伯克利分校。”这样就创建了一个由该校拥有的新区域，而 .edu 组织无需了解这个新 berkeley.edu 区域中的更新。加州大学伯克利分校有权在自己的区域中创建新域，或把区域中的某些部分继续委托给下级组织。

把这三种层级都纳入考虑后，可以更准确地说，树中每个节点都表示一个区域。区域正式定义为负责层级中某一部分的管理权威。

从命名角度看，每个区域包含一个或多个名称—IP 映射，其中名称是该区域的子域。eecs.berkeley.edu 区域可以包含名称 eecs.berkeley.edu 或 iris.eecs.berkeley.edu，但不能包含 math.berkeley.edu。

从基础设施角度看，该区域由一台或多台名称服务器支撑，它们回答有关区域内域的查询。

从权威角度看，区域由某个组织控制，该组织从父级获得管理该区域名称的权威。例如，控制 berkeley.edu 区域的加州大学伯克利分校，可以把 eecs.berkeley.edu 区域委托给 EECS 系。

实际中的 DNS 区域

现实中这些组织是谁？

根区域由互联网名称与数字地址分配机构（Internet Corporation for Assigned Names and Numbers，ICANN）控制。它负责把根区域（代表整个互联网）的各部分委托给特定顶级域。

树中根节点正下方的所有区域都是顶级域（top-level domain，TLD）。互联网最初在美国发展，所以早期 TLD 按用途把互联网划分为不同区域，例如 .com（商业）、.gov（政府）、.edu（教育）和 .mil（军事）。后来又创建了国家 TLD，例如 .fr（法国）和 .jp（日本）。

历史上 TLD 相对很少。近年 ICANN 开始以 $150,000 的注册价格（另加持续维护费用）出售新 TLD，引发了新 TLD 的激增。截至 2024 年，已有超过 1,500 个 TLD，包括 .google 之类公司专用 TLD，以及 .travel 或 .pizza 等更特别的 TLD。

每个 TLD 都由某个组织运营，该组织可以自行决定如何继续划分这个 TLD。例如，.uk TLD 由 Nominet 管理；他们决定按用途划分 .uk 区域，创建了 .co.uk（商业）、.ac.uk（学术）等区域。

树中更低层的区域可以按深度称呼。example.com 是二级域，blog.example.com 是三级域。这些区域通常由不同组织和公司控制，它们从父区域购买相应区域。购买区域时，还必须把所用名称服务器告知父区域，使父区域能把用户重定向到你的名称服务器。

树中更低层的名称服务器通常由域名注册商运营。注册商是拥有特定区域并允许用户在该区域中特定域上托管服务的公司。例如，我可以按月付费，把网站托管在 blog.foo.com。注册商通常会提出把相应的域—IP 映射加入其名称服务器。

除了注册商，Google 等公司也可能为自己的应用运营名称服务器（例如提供 maps.google.com 的记录）。Amazon Web Services 也有名为 Route 53 的名称服务器，用户可以向其中添加要提供的记录。

通过任播实现根服务器可用性

如果根服务器不可用，后果会非常严重。缓存为空的人无法发起任何 DNS 请求。随着 TTL 到期，每个人的缓存最终都会清空，互联网也将停止工作。

为提供冗余，世界各地共有 13 台根服务器。我们可以查询这些公开且众所周知的根服务器 IP 地址。

考虑到整个互联网都依赖它们，13 这个数字似乎仍然很少。为了提供极高可用性，实际上的根服务器远不止 13 台，但列表只包含 13 个 IP 地址，这是因为采用了名为任播的巧妙技术。

在任播（anycast）技术中，我们部署一台根服务器的许多镜像，并让它们全部使用同一个 IP 地址。联系域 k.root-servers.net 或其对应 IP 地址时，你实际可能正在与它的任意一个镜像通信。

13 台根服务器中的每一台实际都由一组镜像组成。例如，f.root-servers.net 有 3,000 多个实例。这些镜像可以由不同网络运营商运行（例如 Google 和 Cloudflare 可能协助运营根镜像）。

实现任播时，每个镜像都会在路由协议中宣告同一个 IP 地址，路由协议的其余部分照常工作。你可能收到许多通往 k.root-servers.net 的路由宣告；接受其中任何一条，最终都会把数据包路由到某个镜像。一处镜像宕机后，其余镜像仍在发送宣告，你可以接受另一条路由来维持可用性。

任播还意味着即使不断增删镜像，根 IP 也很少改变。因此，根提示文件中根名称服务器记录的 TTL 很长（42 天）。

这是 k.root-servers.net 所有镜像的地图。它们都宣告同一个 IP 地址，你的路由器很可能会选择与最近的镜像通信。k.root-servers.net 由 RIPE（位于欧洲）运营，这或许能解释为何欧洲有如此多镜像。

Google 的 8.8.8.8 等公共解析器同样可以使用任播来实现高可用性。

DNS 与电子邮件

DNS 不只可以存储并提供域—IP 映射。例如，当你要向 evanbot@berkeley.edu 这样的地址发送电子邮件时，计算机仍需知道该把数据包发到哪里。

可以使用 MX 类型记录把电子邮件地址转换为邮件服务器。这些记录把 berkeley.edu 等域映射到 aspmx.l.google.com 等邮件服务器。邮件服务器位于不同区域也没问题（例如这里加州大学伯克利分校的邮件服务器由 Google 运营）。

历史上，一个电子邮件地址对应特定机器上的一个用户，因此邮件服务器就是那台机器。如今，你很可能希望从任何地方访问邮件。所以 MX 记录会映射到 aspmx.l.google.com 之类的邮件服务器，由它们接收邮件，并允许你从任何计算机访问。

MX 记录的一个关键区别是，其值还包含优先级。如果收到多条 MX 记录，它们把同一个域映射到不同邮件服务器，应当首先尝试优先级最高（数字最小）的邮件服务器。

DNS 与负载均衡

演示中最后一台名称服务器返回一条 A 类型记录，但实际上可以收到多条 A 类型记录，把一个域映射到多个 IP 地址。这些记录没有顺序，服务器还可能在返回前打乱次序。任意一条都有效，客户端可以选择任何一条（通常是第一条）。提供多个 IP 地址有助于负载均衡和冗余。

名称服务器还可以根据查询发送者及发送位置，返回不同的 A 类型记录。如果要把 www.google.com 这样的热门域的负载分散到多台服务器，这会很有帮助。例如，可以尝试把用户送往最近的服务器。

此时名称服务器需要额外的（可能是专有的）逻辑来决定回复哪些记录。它可以检查递归解析器是谁，也可以检查终端客户端是谁，不过这要求扩展 DNS，让解析器的查询包含客户端地址。它甚至可以检查终端用户的地理位置，但这需要 IP 地址到物理位置的映射。MaxMind 等商业数据库就提供这种映射。

基于用户地理位置的负载均衡并不完美。即使知道服务器和用户的物理位置，我们仍需猜测从网络角度看哪台服务器最近，也不了解用户与不同服务器之间的性能（例如网络带宽）。

下面是一项检验 Google 地理负载均衡效果的实验。我们分别在旧金山和俄勒冈查询 www.google.com，得到两个不同的 IP 地址。

随后执行反向查询，把每个 IP 地址匹配到特定服务器名称（每台服务器不同，并非 www.google.com）。结果是一条 PTR 类型记录，它把 IP 地址映射到名称（与 A 类型记录相反）。可以看到，Google 在旧金山给出的 IP 地址映射到名称中含有 sfo03s25 的服务器。假设 sfo 代表旧金山，那确实相当准确！

再看延迟：从旧金山的计算机连接到为旧金山提供的 IP 地址需要 20 ms，连接到为俄勒冈提供的 IP 地址需要 35 秒。Google 很好地为旧金山的计算机分配了更近的服务器！
