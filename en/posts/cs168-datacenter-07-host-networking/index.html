主机网络

什么是主机网络？

传统上，网络的瓶颈位于网络基础设施内部，而不是终端主机。然而，在现代高性能数据中心中，随着网络性能需求不断增加，终端主机正在努力满足需求。

特别是，运行 TCP 等网络协议的 CPU 不再能够提供数据中心所需的高性能。 CPU 价格昂贵，而提供高性能意味着 CPU 将所有时间都花在运行网络协议上，分配给运行实际应用程序的资源较少。

此外，我们一直在运行的实际协议（例如 IP 和 TCP）已不再能够满足现代的高性能需求。

为了解决这两个问题，我们转向主机网络，它涉及终端主机（而不是网络内部）的优化。

优化：用户空间共享内存

回想一下，在终端主机上，第 1 层和第 2 层是在网络接口卡 (NIC) 的硬件中实现的。第 3 层和第 4 层通过操作系统（在 CPU 上）的软件实现。第 7 层是应用程序本身。

回想一下先决课程（例如加州大学伯克利分校的 CS 61C），现代计算机是采用虚拟内存设计的，因此每个应用程序都有自己的专用地址空间，与其他应用程序隔离。特别是，每个第 7 层应用程序在用户空间中都有自己的专用地址空间。相比之下，操作系统本身运行在内核空间，这是用户空间中的应用程序无法访问的特殊内存部分。

这种内存管理模型意味着当我们通过堆栈传递数据包来发送数据时，我们不断地将数据从用户空间复制到内核空间。此外，当我们将数据包传递到堆栈以接收数据时，我们不断地将数据从内核空间复制到用户空间。在内核空间和用户空间之间复制位是昂贵的并且毫无意义。

这种内存管理模型的另一个问题是，在内核空间中编程很困难。如果我们想修改 TCP 并对其进行优化以达到我们的目的，我们将必须深入到操作系统和程序的非常低的级别。在内核空间中的部署和测试比在用户空间中更难、更慢。

为了解决这两个问题，我们可以将网络堆栈（例如第 3 层和第 4 层协议）从内核空间移至用户空间。现在，第 3、4 和 7 层都可以访问共享地址空间，并且不需要来回复制。此外，用户空间中的迭代和创新现在变得更加容易。

在用户空间中使用共享内存可以帮助我们消除一些无关的工作，例如来回复制，但这仍然不足以使我们的主机满足现代性能要求。

优化：卸载到 NIC

CPU 的速度不够快，无法以现代性能速度运行网络协议（例如 IP、TCP）。此外，使用 CPU 运行网络协议会减少应用程序本身使用的 CPU 资源。

为了解决这个问题，我们可以将网络堆栈从 CPU（软件）卸载到 NIC（硬件）。

NIC 是卸载操作的天然场所。每个数据包都必须经过 NIC，因此 NIC 可以进行一些额外的处理，从而节省 CPU 执行该工作。

网络驱动程序是操作系统中的一个软件，用于对 NIC 进行编程和管理。该驱动程序提供了一个API，允许操作系统中的更高级别的程序与NIC进行交互。您可以将驱动程序视为硬件和软件之间的桥梁。

卸载有什么好处？它释放 CPU 资源供应用程序使用。此外，硬件中的专用处理比通用 CPU 上的处理更高效。这里，效率指的是速度和功耗。最后，在硬件中运行操作不仅可以降低延迟，还可以提高延迟的可预测性和一致性。在软件中运行应用程序时，CPU 必须调度不同的进程，这可能会增加不可预测的延迟。 （例如，如果我有一个数据包要处理，CPU 可能必须先完成当前的工作，然后才能切换到处理我的数据包。）

卸载简史：Epoch 0

将操作从操作系统（软件）卸载到网卡（硬件）是一个活跃且持续的研究领域。经历了三个发展阶段，越来越复杂的操作已经被卸载到 NIC 上。

Epoch 0：在进行任何卸载之前，让我们看看 NIC 在我们迄今为止看到的标准网络堆栈中做了什么。

NIC 有一个中央控制器处理器，用于管理卡上的操作。

对于传入数据包，收发器将电信号转换为数字信号（1 和 0），并将这些位放入缓冲区中。然后，NIC 从缓冲区读取位，将其解析为以太网帧，处理该帧（例如验证校验和），并删除第 2 层标头。最后，NIC 生成一个中断，告诉 CPU 停止正在执行的操作并收集生成的第 3 层数据包以进行进一步处理。

对于传出数据包，来自网络驱动程序的数据包被放置在缓冲区中。 NIC 从缓冲区读取位并对其进行处理以构造以太网帧。然后，该帧被传递到收发器，该收发器将数字位转换为电信号。

在标准网络堆栈中，您可以将 NIC 视为一个门垫，它将传入数据包传递到操作系统，并向操作系统发送传出数据包，但对这些数据包执行非常少的处理。

卸载简史：Epoch 1

我们尝试卸载到 NIC 的第一个操作是简单的无状态操作。这些无状态操作可以针对每个数据包独立完成，并且 NIC 不必记住多个数据包之间的任何状态。

我们可以卸载的一项无状态操作是校验和计算，不仅在第 2 层，而且在第 3 层和第 4 层。NIC 可以验证这些校验和（针对传入数据包）并计算这些校验和（针对传出数据包），这样 CPU 就不需要这样做。

我们可以卸载的另一个无状态操作是分段。在我们的标准模型中，如果应用程序有一个巨大的文件要发送，那么操作系统负责将文件分割成较小的数据包。然后，在接收方，操作系统负责重新组装这些数据包。作为一种优化，我们可以让 NIC 处理数据包的拆分和重组。现在，操作系统不再需要处理大量的小数据包，而是可以处理一些大数据包，这样效率更高（例如，需要处理的标头更少）。

通过分段，需要在平滑连接和 CPU 效率之间进行权衡。如果应用程序将大数据包交给 NIC，则 CPU 要做的工作就会减少。然而，现在网卡获取大量突发数据，连接更加突发。相比之下，如果应用程序将较小的数据包交给 NIC，CPU 就会有更多的工作要做，但 NIC 会获得更稳定的数据流，并且最终的连接也会更顺畅。

聚合小数据包存在一些挑战。如果中间数据包丢失怎么办？那么网卡可能不得不放弃一堆小数据包，并且无法将它们组合成一个大数据包。如果某些数据包设置了标志（例如用于拥塞的 ECN），而其他数据包没有设置怎么办？生成的聚合数据包是否应该设置标志？

我们将讨论卸载的第三个无状态操作是多队列支持。在我们的标准模型中，NIC 有一个用于传出数据包的队列，以及一个用于传入数据包的队列，并且所有应用程序共享这些队列。网络驱动程序（软件）负责负载平衡，以防多个应用程序或多个 CPU 发送和接收数据。

我们可以将此负载平衡作业卸载到 NIC。现在，NIC 具有多个传输队列和多个接收队列。例如，在多处理器系统中，每个CPU都可以有自己的专用发送/接收队列。 NIC 并行维护所有队列，确保不同 CPU 之间的隔离和负载平衡。 NIC 还可以将某些队列优先于其他队列。

即使 NIC 有多个队列，它最终仍然必须沿着一根线路发送所有数据包。因此，NIC 需要一些数据包调度程序来决定下一个从哪个队列发送。可以对调度程序进行编程以实现所需的负载平衡行为（例如，如果我们想要优先考虑一个队列而不是另一个队列）。

多个队列面临的一项挑战是将数据包映射到队列。当CPU有一些数据要发送时，它使用哪个队列？特别是，我们希望确保单个流中的所有数据包最终都位于同一队列中（并且不会分布在多个队列中）。这有助于我们确保流中的数据包按顺序发送。回想一下，在 TCP 中，乱序发送数据包是可行的，但对性能不利（例如接收方必须缓冲乱序数据包）。

当处理来自各个接收队列的传入数据包时，NIC 可以对数据包进行哈希处理，以决定哪个 CPU 将处理该传入数据包。然后，NIC 中断该 CPU 并告诉它处理数据包。基于哈希的行为类似于 ECMP（等价多路径路由），有助于我们确保同一流中的所有数据包都由同一 CPU 按顺序处理。

卸载简史：Epoch 2

后来，我们开始将更复杂的有状态操作卸载到 NIC。

Epoch 2 的开发是由数据中心虚拟化推动的，其中多个虚拟机在同一台物理服务器上运行。例如，在虚拟化中，我们需要一个虚拟交换机将传入的数据包转发到适当的虚拟机。我们展示了在软件中运行的虚拟交换机，但虚拟交换机也可以在硬件中实现。

防火墙和带宽管理是有状态卸载的另一个示例。在软件中，我们可以实施防火墙来执行安全策略（例如，丢弃来自该恶意 IP 的所有传入数据包）。我们还可以强制执行策略来管理用户之间的带宽（例如，用户 A 每分钟只能发送 100 个数据包，超出的数据包将被丢弃）。这些安全策略可以通过硬件来检查。

为了实现这些有状态操作，我们可以使用匹配操作对表，类似于 OpenFlow 表（来自 SDN 部分）。该API允许软件将不同的策略编程到硬件上，以便硬件可以根据这些策略处理数据包。正如我们之前看到的，匹配可能针对 5 元组或其他一些标头字段。这些操作可能是丢弃数据包、将数据包转发到特定的下一跳或修改标头。

卸载简史：Epoch 3

这是当前卸载的时代。人们正在不断努力将整个协议（例如 TCP）从操作系统中卸载到 NIC 上。这个时代是由更高的性能需求推动的，特别是像 AI/ML（人工智能、机器学习）这样具有高性能要求的应用程序。

理想情况下，我们希望让应用程序直接将数据交给硬件，并让硬件在第 4、3、2 和 1 层执行所有必要的网络处理。操作系统完全不在考虑范围内，所有网络协议都直接在硬件中实现。

虽然已经进行了一些将 TCP 等标准网络协议卸载到 NIC 上的实验，但尚未大规模部署。相反，我们设计了 RDMA 等新的网络协议，这些协议经过专门设计，可以直接在硬件中实现。

RDMA：远程直接内存访问

RDMA 提供了一种抽象，其中服务器 A 可以直接访问服务器 B 中的内存，而无需任何服务器中操作系统或 CPU 的参与。 RDMA 可以直接在硬件中实现，取代标准 TCP/IP 软件网络堆栈。

假设服务器 A 想要向服务器 B 发送一个 10 GB 的文件。在标准网络堆栈中，CPU 从内存中读取文件，对其进行处理（例如 TCP/IP），然后将生成的数据包传递到 NIC。在接收方，NIC 将数据包传递给 CPU，CPU 处理数据包，并将生成的文件有效负载写入内存。请注意，CPU 参与处理 10 GB 文件的每个数据包。

在 RDMA 抽象中，NIC 从内存中读取文件并将其发送出去，无需 CPU 参与。在接收方，NIC 处理传入的字节并将其写入内存，同样无需 CPU 参与。请注意，CPU 仍然需要在开始时设置传输，并在结束时完成传输。但大部分 10 GB 文件传输是在没有 CPU 的情况下完成的。

为了使用 RDMA，程序员不再使用套接字抽象。相反，我们使用的主要抽象是队列对。发送工作队列包含所有待处理的作业，其中数据需要从我传输给其他人。接收工作队列包含我需要从其他人接收数据的所有待处理作业。单个 NIC 可以有多个队列对，每个队列对为程序员提供不同的服务。例如，一对可能提供可靠的、按顺序的交付，而另一对可能提供不可靠的交付。配置为可靠且有序的队列对最接近传统的 TCP 连接。

队列中的每个元素称为工作队列元素（WQE）。 WQE 让应用程序描述需要完成哪些工作。在英语中，接收队列中的 WQE 可能会说：“从远程服务器上的地址 0xffff1234 开始取出 100 MB，并将它们写入本地内存中的地址 0xffff7890。”在代码中，WQE 是一个包含这些指令的结构，例如指向我们写入接收到的数据的位置的指针。

请注意，WQE 抽象为 RDMA 协议提供了应用程序的更高级别视图。在 TCP/IP 堆栈中，网络仅看到字节流，但在 RDMA 中，WQE 允许应用程序更详细地描述作业（例如，指定正在传输的数据块的开始和结束）。

当作业完成时，WQE 将从队列中删除，并且 NIC 创建一个名为 完成队列元素 (CQE) 的新结构，描述作业发生的情况（例如成功或失败）。此 CQE 存储在完成队列中，并在那里等待，直到应用程序准备好读取 CQE 并了解作业发生的情况。

请注意，RDMA 是异步的。应用程序可以随时将作业（WQE）添加到队列对中，NIC 将按顺序处理作业。同样，当作业完成时，CQE 会被放入完成队列中，应用程序可以随时读取 CQE。 （将此与 TCP/IP 堆栈进行对比，其中传入数据会触发 CPU 处理该数据的中断。）

RDMA 示例

RDMA 可用于服务器之间的多种不同操作。每个操作都有自己的性能规范（例如不同的延迟）和不同的语义（例如不同的错误消息）。作为示例，让我们看一下 RDMA 发送操作，其中服务器 A 从其内存中读取文件，传输该数据，而服务器 B 将该文件写入其内存中。

  1. 每个服务器指定其内存的某些部分可供 NIC 访问以进行 RDMA 传输。服务器A指定该文件对应的内存为NIC可读。服务器 B 指定一个空白缓冲区，它将在其中接收 NIC 可读的文件。

2、每个服务器设置队列。两个 NIC 现在都有一个发送队列、一个接收队列和一个完成队列。请注意，此步骤可以在带外完成，使用 TCP 等传统协议在两个服务器之间进行协调。

  1. 服务器A 在发送队列中创建WQE。该 WQE 包含一个指向文件的指针，指示要发送的数据。另一方面，服务器 B 在接收队列中创建一个 WQE。该 WQE 包含一个指向空白缓冲区的指针，指示应将接收到的数据写入何处。
  1. 一旦传输在双方排队，数据传输就可以发生，无需软件参与。 NIC 处理一切事务，包括可靠性、拥塞控制等等。
  1. 传输完成后，WQE 将从队列中删除。两个 NIC 都会生成 CQE，指示传输已完成，并包括任何相关的状态消息（例如错误消息）。服务器A的CQE表示数据发送成功，服务器B的CQE表示数据接收成功。
  1. 最终，应用程序读取 CQE 以了解传输发生了什么。

RDMA 的优点、缺点与应用

RDMA 为我们提供了高性能数据传输（低延迟、高带宽），并为应用程序释放了 CPU。然而，RDMA 并不是免费的。 RDMA 需要专门的硬件和软件，并且通常比传统的网络堆栈更复杂。请记住，RDMA 正在取代 TCP/IP 堆栈，因此它必须直接在硬件中实现所有 TCP/IP 功能，例如可靠性和拥塞控制。

RDMA 也有一些限制，通常在两台服务器物理距离较近的数据中心中效果最佳。如果两个服务器距离很远，则主要延迟来自通过网络发送数据，并且 RDMA 节省的时间可以忽略不计。相比之下，如果两台服务器位于附近，则主机处理数据包可能是主要延迟，因此 RDMA 可以显着节省时间。

RDMA 已应用于许多需要高性能、低延迟计算的不同环境中。示例包括科学研究、金融建模、天气预报、机器学习和搜索查询。在云计算中，RDMA可用于将大型虚拟机从一台物理服务器迁移到另一台物理服务器，从而释放CPU供客户使用。在 AI/ML 训练中，RDMA 不仅可以释放 CPU 并为我们提供低延迟，而且还为我们提供了可预测的延迟，这在不同服务器需要协调训练 AI/ML 模型时非常重要。

RDMA 的实现

请记住，RDMA 取代了 TCP/IP 网络堆栈，因此 RDMA 负责可靠性、拥塞控制等。如何实现这一点有两种广泛的理念。

一种选择是在网络本身中实现这些功能，例如开关的可靠性。这就是 Nvidia InfiniBand 背后的想法。

另一种选择是在 NIC 中的队列对抽象之下实现这些功能。这就是谷歌目前正在追求的想法。

在这两种情况下，软件中的应用程序和操作系统都会通过队列对抽象获得可靠、按顺序交付的错觉。这里的区别在于 RDMA 如何实际实现这些服务保证。
