端到端连接

动机

在本节中，我们将逐步介绍当我们打开计算机、将其插入以太网并在网络浏览器中键入 www.berkeley.edu 时会发生什么情况。在此过程中，我们将看到网络的所有不同部分如何协同工作来处理用户的请求。

我们假设我们不需要从头开始打开互联网。例如，路由器已经在主动运行路由协议并相应地填充了它们的转发表。

步骤 1：DHCP

我们打开计算机并将其插入以太网。我们还没有任何有关网络的信息，因此我们广播 DHCP 请求。

我们假设家庭路由器是 DHCP 服务器，这在家庭网络中很常见。路由器/服务器将报价单播回给我们。该报价包含有关网络的信息：子网掩码、默认网关的 IP 地址以及 DNS 服务器的 IP 地址。该优惠还为我们提供了一个可以使用的 IP 地址。

为了完成 DHCP 协议，我们发送请求消息，确认我们想要使用所提供的配置，然后路由器/服务器以确认进行响应。

步骤 2：查找第 2 层路由器

从 DHCP 中，我们了解了路由器的 IP 地址，并且我们的转发表现在表明所有非本地数据包都应转发到该路由器。我们将向 DNS 服务器（查找 www.berkeley.edu 的 IP 地址）和 Berkeley 服务器本身发送一些数据包，这两个服务器都可能是非本地的。

不过，在将 IP 数据包转发到路由器之前，我们需要找出路由器的第 2 层 MAC 地址，以便我们可以将数据包发送到本地网络内的路由器。

首先，我们可以验证路由器的 IP 地址 192.168.1.1 属于本地子网 192.168.1.2/24。这告诉我们路由器位于本地网络中，通过向路由器的 MAC 地址发送以太网数据包，我们将到达路由器。

为了找到路由器的 MAC 地址，我们广播一个 ARP 请求，询问 MAC 地址 192.168.1.1（路由器的 IP 地址）。路由器听到此请求并回复说：“我是 192.168.1.1，我的 MAC 地址是 01:ab:cd:ef:42:01。”

我们现在可以缓存此 IP 到 MAC 的映射，并且现在我们知道路由器的 MAC 地址。只要该条目保留在缓存中，我们就不必再次发出相同的 ARP 请求。以后所有对外部 Internet 的请求都可以转发到路由器的 MAC 地址。

步骤 3：DNS 查找

接下来，我们需要查找 www.berkeley.edu 的 IP 地址。这一切都是在浏览器代码调用 getaddrinfo 之类的内容来触发 DNS 查找之后在操作系统中完成的。

从 DHCP 中，我们获知 DNS 服务器的 IP 地址 8.8.8.8。我们还了解到我们位于子网 192.168.1.2/24 中。 DNS服务器不在我们的本地网络中，因此我们需要将DNS数据包转发到路由器。

我们现在可以从上到下构建 DNS 请求数据包。

第 7 层：在问题部分，我们添加一条 DNS 记录，请求带有 www.berkeley.edu 的 IP 地址的 A 记录。我们添加包含 ID、记录数等的 DNS 标头。

第 4 层：DNS 运行在 UDP 之上。我们选择任何随机源端口，因为我们是客户端。我们选择端口 53 作为目标端口，因为这是解析器和名称服务器侦听 DNS 查询的地方。

第 3 层：源 IP 是我们自己的 IP，由 DHCP 分配。目标IP是8.8.8.8，这是我们从DHCP获知的DNS服务器的IP地址。

第2层：源MAC是我们自己的MAC地址，它被烧录到我们的硬件中。目的MAC是我们从ARP中获知的路由器的MAC地址（下一跳）。

数据包完全构建后，我们可以沿线路发送位（第 1 层）。

当数据包到达路由器时，如果网络使用 NAT，路由器可能会重写 UDP/IP 标头，以将我们的私有 IP 地址转换为公共 IP 地址。然而，作为最终主机，我们不必担心NAT。路由器应该为我们完成所有转换，让我们产生可以使用自己的 IP 地址（来自 DHCP）的错觉。

当我们的数据包到达 8.8.8.8 处的递归解析器时，如果解析器尚未缓存我们的答案，则可能需要执行一些额外的查找并向权威名称服务器询问记录。最终，递归解析器找到答案，并将 A 记录发送回给我们。现在我们有了 www.berkeley.edu 的 IP 地址。

步骤 4：连接到网站

现在我们有了 www.berkeley.edu 的 IP 地址，我们可以将数据包发送到 Berkeley。我们使用的是 Web 浏览器，因此我们的目标是向该服务器发出 HTTP 请求。

HTTP 运行在 TCP 之上，因此我们首先必须进行 TCP 握手以打开与 Berkeley 服务器的连接。浏览器将在特定套接字上调用类似 connect 的内容来打开此连接，操作系统（运行 TCP 的地方）将执行握手并将数据包传递到浏览器或从浏览器传递数据包。

执行 TCP 握手：我们发送 SYN，Berkeley 发送 SYN-ACK，然后我们发送 ACK。现在我们的计算机和伯克利服务器之间有一个字节流。

现在，我们可以从上到下构建 HTTP 数据包。

第7层：HTTP方法是GET。我们想要的资源是/（主页）。版本是HTTP/1.1。

第 4 层：HTTP 运行在 TCP 之上。浏览器可以选择任何源端口，因为它是客户端。一般来说，该端口可以由应用程序手动指定，或者应用程序可以指定“端口 0”，这是要求操作系统选择当前未使用的随机临时端口的简写。 （顺便说一句，回想一下 NAT，允许应用程序手动指定端口是两个用户可能选择相同源端口的原因。）目标端口是 80，这是 HTTP 的固定端口号。

第 3 层：源 IP 是我们自己的 IP，由 DHCP 分配。目标IP是141.193.213.21，这是我们之前从DNS查询返回的www.berkeley.edu的IP地址。

第 2 层：这与我们之前的 DNS 数据包相同。源 MAC 是我们自己的（刻录到硬件中），目标 MAC 是路由器的（从 ARP 发现并缓存）。

HTTP 响应返回状态码 200 OK，响应内容包含网站的 HTML 代码。浏览器在套接字上调用 read 来获取 HTTP 有效负载的字节以及状态代码和响应，并相应地处理它们。

在字节流中，HTTP 可以添加一些分隔符（如换行符）来表示请求或响应的结束。此外，HTTP 标头（例如 Content-Length）可以指定有效负载的长度。这也允许浏览器分配足够的内存来接收响应。

返回的 HTTP 响应可能会触发进一步的请求。如果响应中的 HTML 具有类似 <img src="/logo.png"> 的语法，这会告诉浏览器发出另一个 HTTP 请求来获取 /logo.png 资源。或者，用户可能单击网站上的链接（例如 www.berkeley.edu/about.html），这也会触发对同一服务器的另一个 HTTP 请求。

回想一下，为了提高效率，对同一服务器的多个 HTTP 请求可以通过同一个 TCP 连接进行管道传输，因此我们可以保持 TCP 连接打开并继续将其用于后续的 HTTP 请求和响应。

最终，经过一些管道操作后，客户端或服务器选择关闭连接。发生正常的拆卸握手，其中每一方都发送 FIN，并且两个 FIN 数据包都被确认。我们都完成了！

请注意，HTTP 请求/响应不一定包含在单个数据包中。 HTTP 构建在 TCP 字节流之上，因此单个 HTTP 请求或响应可能会分为多个 TCP/IP 数据包，其中每个数据包在第 1-3 层具有相同的标头，而第 4 层标头的序列号不同。整个 HTTP 请求/响应只有一个标头，即使请求/响应跨数据包拆分也是如此。使用 HTTP，一个请求/响应与一个数据包之间不再存在一对一的关联。

套接字

如果您是通过浏览器访问网站的用户，则无需编写任何代码即可通过 Internet 运行应用程序 (HTTP)。但是，如果您是编写自己的应用程序的程序员，您可能需要编写一些代码来与网络交互。

套接字抽象为程序员提供了一种与网络交互的便捷方式。套接字抽象完全存在于软件中，程序员可以运行五个基本操作：

我们可以创建一个新的套接字，对应一个新的连接。在像 Java 这样的面向对象语言中，这可能是构造函数调用。

我们可以调用 connect，它会启动到某个远程计算机的 TCP 连接。如果我们是客户端-服务器连接中的客户端，这非常有用。

我们可以在特定端口上调用listen。这不会启动连接，但允许其他人在指定端口上发起与我们的连接。

连接打开后，我们可以调用 write 在连接上发送一些字节。我们还可以调用 read，它接受一个参数 N，从连接中读取 N 个字节。

这种套接字抽象为程序员提供了一种编写应用程序的方法，而无需考虑 TCP、IP 或以太网等较低级别的抽象。

从操作系统的角度来看，每个套接字都与一个第 4 层端口号相关联。进出单个套接字的所有数据包都具有相同的端口号，操作系统可以使用该端口号进行多路分解并将数据包发送到正确的套接字。

操作系统中的层

在硬件中，第 1 层和第 2 层是在计算机的硬件网络接口卡 (NIC) 上实现的。第 3 层和第 4 层在操作系统的网络堆栈中实现。第 7 层应用程序是通过软件实现的。将第 3 层和第 4 层放入操作系统的好处是，应用程序不必担心每次都重新实现它们。

通过这种分工，应用程序只需要考虑数据即可。 NIC 只需要考虑数据包。操作系统中的网络堆栈在连接和数据包之间进行转换。

查看数据包

如果您想查看通过网络发送的数据包，可以使用 tshark 和wireshark 等工具。这些工具在调试代码的网络部分时非常有用。

在浏览器中，您还可以使用检查元素控制台的“网络”选项卡来查看正在发送和接收的数据。

如果您实际上查看了通过网络发送的原始数据包，您将看到一些我们在端到端演练中未涵盖的现实世界的复杂性。例如，数据包可能会被加密并通过 TLS 发送。此外，如果我们使用 HTTP/3.0，数据包可能通过 QUIC（针对 HTTP 优化的 UDP 变体）而不是 TCP 发送。

重新审视分层

完整的端到端图片让我们了解为什么分层是构建网络的有用原则。我们能够在单个层解决特定问题，而无需同时考虑所有层。

事实上，我们在这堂课上根本没有讨论过第 1 层。我们没有讨论通过电线发送信号所需的电气工程或物理学。然而，我们仍然能够在 Layer 1 之上构建其他层，而不需要确切知道 Layer 1 是如何工作的。

在本课程中，我们讨论了作为主要第 7 层协议的 HTTP，但 HTTP 是一个相对简单的协议。多个应用程序可能希望在 HTTP 之上构建相同的复杂功能，但他们不希望每个应用程序独立地编写该功能的代码。为了支持这一点，我们实际上可以在 HTTP 之上构建更多协议，这样程序员就不必总是从头开始使用 HTTP。

第 7 层之上的协议的一个示例是远程过程调用 (RPC) 库。这允许程序员编写一些代码，其中一些功能实际上在网络中其他位置的另一台计算机上执行。如果每个人都必须从头开始在 HTTP 之上编写 RPC，那会很烦人，因此，Apache Thrift 和 gRPC 等库的存在可以从程序员手中抽象出更多细节。

这是程序员可能编写的一些网络代码的示例。它对客户端进行编程，向某个远程服务器打招呼。

请注意，我们讨论的所有网络协议都完全隐藏在对网络库的两行调用后面。程序员不必考虑 HTTP、TCP、IP、以太网、ARP、DHCP 或任何其他较低级别的协议。如果这些协议出了问题，了解它们仍然很有用，并且了解这些协议可以帮助您针对特定协议优化代码，但最终，分层是一个非常强大的抽象工具。
