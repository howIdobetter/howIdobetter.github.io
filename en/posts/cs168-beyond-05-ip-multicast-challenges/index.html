IP 多播面临的挑战

跨域路由

我们到目前为止介绍的协议（IGMP、DVMRP、CBT）可用于域内多播路由（intra-domain multicast routing）。然而，它们很难扩展到跨域多播路由（inter-domain multicast routing）。

这里的一个主要问题是可扩展性。例如，如果我们在全球范围内使用 DVMRP，那么当剪枝状态被周期性删除时，数据包就会泛洪到整个互联网，这并不现实。

此外，回想一下，跨域路由还带来了自治系统（AS）自治性与隐私方面的额外挑战。例如，如果我们在全球范围内使用 CBT，那么核心路由器可能位于另一个网络中，这就要求你信任其他人来控制核心路由器。

跨域多播路由是一个难题，人们已经为开发解决方案做了大量工作。例如，可以通过设置多个相互通信的核心（每个网络一个）来解决 CBT 的核心选择问题。然而在实践中，跨域多播路由几乎没有得到采用。

计费

IP 多播的服务模型与现代互联网服务提供商（ISP）采用的商业模式存在根本冲突。例如，请看下面的 AS 图，其中 AS A 与 AS B 是对等方（peers）：

作为对等方，AS A 和 AS B 应当能够交换等量的流量，但多播使“等量流量”很难定义。举例来说，假设 AS A 向 AS B 发送一个多播数据包。AS B 可能有许多属于该组的下游子节点。这意味着 AS B 只接收了一个数据包，却不得不发送许多个数据包。AS B 在这里使用的带宽远多于 AS A。AS A 是否需要因此向 AS B 额外付费？（这是一个开放问题，没有明确答案。）

AS B 正在付费购买 AS A 的服务。如果 AS B 发送一个多播数据包，而 AS A 必须把该数据包的副本转发到许多其他目的地，会怎么样？与单播数据包相比，AS A 是否应该对这个数据包收取更多费用？如果应该，又该多收多少？（这是一个开放问题，没有明确答案。）

IP 多播模型不会显式记录组的规模，这使商业模式的设计更加困难。如果你想根据目的组的规模向用户收费，就没有明确的方法来确定任意给定目的组的规模。转发表会告诉你在不同分发树上的父节点和子节点，但不会告诉你总共有多少终端主机将收到这个数据包。

拥塞控制

设想一个源沿分发树向许多接收方发送多播数据包。源需要选择合适的发送速率，以免网络过载。源应当选择什么速率？

流量将沿许多不同路径传输，而每条路径的容量可能各不相同。源可以按 1 Mbps 发送，以避免任何链路过载，但这会使其他路径上的容量闲置。另一方面，源也可以按 100 Mbps 发送以最大化性能，但这会导致一些链路过载。源应该选择什么速率，并没有明确答案。

在实践中，一种可能的解决方案是按照性能定义不同的组。例如，我们可以定义四个不同的多播组，每个组都接收同一个视频流，但视频质量不同。这样，任何感兴趣的接收方都可以尝试加入不同的组，看看哪个组能提供最佳性能。

可靠性

与 IP 单播一样，IP 多播采用尽力而为（best-effort）服务，这会带来一些额外的复杂性。例如，你发送一个数据包后，它可能到达部分组成员，却没有到达所有组成员。

我们可以尝试添加确认（ack）来解决这个问题，但这样做也可能产生问题。如果组内有数百万个成员，单个发送方将无法为每个数据包处理数百万个确认。

另一种可能的方法是使用否定确认（negative acknowledgement，nack）：组成员收到数据包时什么也不发送；没有收到数据包时（例如其计时器到期）则发送 nack。同样，如果组内有数百万个成员，单个发送方可能会不堪重负。

在 nack 方法中，如何从故障中恢复也并不明确。如果有人没有收到数据包，我们是否要再次向整个组多播这个数据包？这会浪费带宽，因为有些组成员已经收到了该数据包，接下来收到的将是一个重复副本。

另一种方法是只向发送 nack 的组成员单播该数据包。如果许多组成员都没有收到数据包，这种做法可能很浪费，因为我们必须单播同一个数据包的许多份副本。例如，考虑第一条链路就丢弃数据包的情况，这意味着所有组成员都没有收到该数据包。

哪种重传方法更好？答案并非一目了然，而且可能取决于有多少组成员收到了数据包。

在实践中，一些现代 IP 多播应用根本不实现可靠性。或者，它们通过在数据流中编码一定的冗余（想想纠错码）来实现可靠性，使丢失和损坏能够通过数据本身得到纠正，而不需要网络的帮助。

编码冗余确实意味着需要更多比特来编码相同的数据。例如，如果你想发送相当于 5 个数据包的数据，可以发送 10 个数据包，并以某种方式编码其中的比特，使任意 5 个数据包都能用于重建原始数据。

安全性

IP 多播的另一个局限是缺少访问控制。任何人都可以加入一个组，任何人也都可以向任何组发送消息。如果你想实施访问控制（例如，只有付费用户才能观看体育比赛），就必须另外构建这项功能。

缺少访问控制会导致安全漏洞。恶意发送方可以向某个特定多播组泛洪数据包，使该组的所有成员不堪重负。请注意，这比单播方式下的相应攻击更有效，因为在单播情况下，恶意发送方必须分别向每个成员泛洪数据包。

加密等额外安全措施也很难实现。假设你通过向每个组成员提供一个共享密钥来加密多播消息。如果有人离开组，会怎么样？如果继续使用同一个密钥，该用户仍然知道这个密钥，仍能读取你的消息。一种方法是改用新密钥，但此时你需要一种方式，将这个新密钥安全地分发给余下的组成员。

实践中的 IP 多播

由于存在上述所有挑战，如今 IP 多播主要在单个域内使用，而不跨不同的域使用。

有些应用可能仍然希望跨多个网络进行组通信（例如多人游戏、视频会议）。许多应用没有依赖不支持跨网络通信的 IP 多播，而是为组通信实现了自己的定制解决方案。

例如，如果组足够小，应用可以实现一个中央中继服务器。组通信先通过单播发送到中央中继服务器，再由该服务器将消息单播给其他组成员。

或者，如果组足够小，朴素的单播方案（分别向每个组成员发送单独的单播数据包）也完全可以正常工作。

如果 IP 多播无法跨域工作，而定制解决方案又需要额外的实现与扩展工作，那么现代应用如何处理组通信？一种解决方案是使用覆盖多播（overlay multicast）。它是 IP 多播的一种替代方案，在第 7 层而非第 3 层实现网络功能。接下来我们将研究覆盖多播。
