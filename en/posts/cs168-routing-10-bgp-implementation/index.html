BGP 的实现与问题

边界路由器与内部路由器

到这里，我们已经对 BGP 如何在自治系统（AS）之间工作有了直观认识。本节将说明 BGP 在路由器层面究竟是如何实现的。在此过程中，我们还会说明 BGP 如何与前面介绍的域内路由协议交互。

到目前为止，我们的域间路由模型把整个 AS 当作单一实体，由它导入和导出路径。

然而在现实中，AS 包含许多由链路连接的路由器（和主机）。

为了真正实现 BGP，我们需要 AS 内部的所有路由器协同工作，从而表现得像一个节点。

在一个 AS 内，我们会把所有路由器分为两类。边界路由器（border router）至少有一条链路连接到另一个 AS 中的路由器。内部路由器（interior router）只与同一 AS 内的其他路由器相连。

只有边界路由器需要向其他 AS 通告路由。有时，我们把通告 BGP 路由的路由器称为 BGP 发言者（BGP speaker）。BGP 发言者需要理解 BGP 协议的语义和语法（如何读取和创建 BGP 通告、收到通告时该做什么，等等）。

外部与内部 BGP 会话

一个 BGP 会话（BGP session）由两台相互交换信息的路由器组成。

外部 BGP（external BGP，eBGP）会话发生在来自不同 AS 的两台路由器之间。eBGP 会话可用于在不同 AS 之间交换通告，并获知通往其他 AS 的路由。只有边界路由器参与 eBGP 会话（因为 eBGP 要求与另一个 AS 通信）。

相比之下，内部 BGP（internal BGP，iBGP）会话发生在同一 AS 内的两台路由器之间（它们不一定由一条链路直接相连）。更具体地说，如果一台边界路由器获知了一条新路由，它可以使用 iBGP 将这条新路由分发给 AS 中的其他路由器。这样，AS 中的所有路由器便能协调一致，并作为一个实体共同运作。边界路由器和内部路由器都会参与 iBGP 会话。

eBGP 和 iBGP 会话不同于内部网关协议（interior gateway protocol，IGP）。IGP 是部署在 AS 内、用于在 AS 内部路由数据包的域内路由协议（例如距离向量、链路状态协议）。

iBGP 和 IGP 很容易混淆。两者都在同一个 AS 内交换消息。然而，iBGP 是域间协议的一部分，帮助路由器获知通往其他 AS 的路径。IGP 是域内协议，帮助路由器获知通往同一 AS 内目的地的路径。

eBGP、iBGP 和 IGP 协同工作，从而建立从任意一台路由器到互联网中任意另一台路由器的路由（即使这些路由器位于不同的 AS 中）。

首先，每个 AS 运行 IGP，以获知同一 AS 内任意两台路由器之间的最低开销路径。

接下来，各 AS 运行 eBGP，彼此通告路由，以获知通往其他 AS 的路由。

最后，各 AS 运行 iBGP，使获知外部路由的路由器能够将该路由分发给同一 AS 内的所有其他路由器。

从 eBGP、iBGP 和 IGP 获知的路由可以用来向互联网中的任何地方发送数据包。如果目的地位于同一个 AS 内（相同的 IP 前缀），我们可以使用从 IGP 获知的路由来转发数据包。如果目的地位于另一个 AS 中（不同的 IP 前缀），我们可以回想 iBGP：它告诉了我们本 AS 中任何路由器发现的所有外部路由。利用 iBGP 的结果，我们可以确定哪一台边界路由器位于该外部路由上。然后，我们可以使用 IGP 将数据包转发给正确的边界路由器（该路由器随后会把数据包转发给下一个 AS）。

举一个具体例子，假设 E 想向 Z 发送数据包。首先，E 所在 AS 中的每台路由器都运行 IGP，获知所有内部路由。接着，AS#5 中的某台路由器使用 eBGP 通告一条通往 Z 的路由。此时，只有 G 知道自己能够到达 Z。最后，G 使用 iBGP 告诉其所在 AS 中的所有路由器，它能够到达 Z。

E 已经从 iBGP 得知，同一 AS 中的路由器 G 能够到达 Z。利用 IGP 路由，E 可以将数据包发送给 G（先转发给 F）。然后，G 可以使用从 eBGP 获知的路由将数据包发送给 Z。

通告一条通往外部目的地的路由的边界路由器，有时称为该目的地的出口路由器（egress router）。这台路由器可以帮助你的数据包离开本地网络，并转移到更接近目的地的其他网络。在上面的例子中，G 是目的地 Z 的出口路由器。

这些协议带来的一个结果是，每台路由器都有两张转发表。一张表将所有内部目的地（同一 AS）映射到下一跳，其中填充的是来自 IGP 的信息。另一张表将所有外部目的地映射到出口路由器（它知道通往外部目的地的路由），其中填充的是来自 eBGP 的信息。

请注意，在 eBGP 表中，出口路由器不一定是下一跳。出口路由器可能与本路由器相隔若干个本地跳，但我们会使用 IGP 到达该出口路由器。

我们已经看到 eBGP（路径向量、通告路由）和 IGP（距离向量或链路状态）是如何以算法形式实现的。那么 iBGP 是如何实现的呢？当一台边界路由器安装了一条通往某个目的地的新路由时，它必须通知 AS 中的其他路由器。一种简单的解决方案是，让边界路由器直接告诉 AS 中的每一台其他路由器。

这种解决方案相对简单，不过它要求每台边界路由器都与每一台其他路由器建立 iBGP 会话。在一个拥有 B 台边界路由器、总计 N 台路由器的网络中，该协议需要 BN 条 iBGP 连接，随着本地网络规模扩大，其扩展性可能较差。

注意：现实中还有其他组合域间路由器与域内路由器的方式。如果感兴趣，可以查阅“路由反射器（route reflector）”，不过本课程不会介绍它们。

AS 之间的多条链路：热土豆路由

到目前为止，在我们的 AS 图中，如果两个 AS 相连，我们会把它们画成只有一条链路（边）。实践中，由于一个 AS 实际上由许多路由器组成，两个 AS 之间可能通过多条链路连接。

在实践中，大型 AS 之间拥有多条链路可能很有用。例如，Verizon 和 AT&T 都是非常大的 AS，其基础设施遍布整个美国。假设这两个 AS 之间只有一条位于西海岸的链路。如果 Verizon 在东海岸的一台路由器与 AT&T 在东海岸的一台路由器想要通信，数据包就必须先在 Verizon 的网络中横跨整个国家，通过链路进入 AT&T 的网络，然后再次横跨整个国家才能到达目的地。

两个 AS 之间有多条链路，也意味着两台路由器之间可能存在多条经过相同 AS 的路径。在 AS 层面，这两条路径经过相同的 AS，我们先前的模型并不区分它们。然而，在更细致的模型中，两条路径都需要被导出，并且必须导入一条优选路由。

如果有两条路由，导入方 AS 会偏好哪一条？

带宽需要花钱，所以我更希望这些流量尽可能远地在他人拥有并付费的基础设施上传输，而尽可能少地使用我自己的基础设施。因此，橙色路径更受偏好。

更正式地说，导入方 AS 会收到两条通告：一条来自西侧路由器，另一条来自东侧路由器。

通过 iBGP，AS 内的每台路由器都能看到这两条通告。一条说出口路由器是西侧路由器，另一条说出口路由器是东侧路由器。每台路由器都必须决定导入哪一条通告。

让我们重点看路由器 E。借助 IGP，这台路由器可以算出到西侧出口路由器（F）的距离，以及到东侧出口路由器（I）的距离。由于西侧出口路由器（F）更近，经由西侧出口路由器（F）路由数据包将消耗该 AS 更少的带宽。因此，这台路由器会导入经过西侧出口路由器（F）的路径。另一台路由器，比如更靠近东侧出口路由器（I）的路由器，可能会决定导入另一条路径。

这种选择最近出口路由器的策略有时称为热土豆路由（hot potato routing）。我们希望数据包尽快离开自己的 AS，并尽快开始在别人的链路上传输。

路由器之间的多条链路：MED

如果一台路由器到两个可能的出口路由器距离相同，会怎样？

为了打破平局，导出方 AS 可以通告自己更偏好其中一条路由。

导出方 AS 偏好哪一条路由？同样，由于带宽需要花钱，导出方 AS 偏好粉色路径，因为它使用的自身带宽更少。在粉色路径的通告中，导出方 AS 可以额外说明“我希望你使用这条路径”；在橙色路径的通告中，导出方 AS 可以额外说明“我希望你避开这条路径”。

现在，到两个出口路由器距离相同的那台路由器，可以在 iBGP 通告中看到这些额外信息。

利用这些额外信息，该路由器可以选择粉色路径上的出口路由器，因为导出方 AS 更偏好这条路径。

导出通告中的这些附加信息称为多出口鉴别器（Multi-Exit Discriminator，MED）。从导出方的角度看，它指明了进入自己网络时所偏好的路由器。从导入方的角度看，它指明了离开自己网络并进入另一个 AS 网络时，另一个 AS 所偏好的路由器。

理解 MED 的另一种方式是：它表示经由这台路由器到目的地的距离。导出方可以说，“西海岸路由器距目的地 3 跳”，以及“东海岸路由器距目的地 12 跳”。数值较小的 MED 更受偏好，因为导出方希望尽可能少地使用自己的带宽。导出方宁愿使用自己的 3 条链路，也不愿使用自己的 12 条链路。

导入策略的优先级

在更细致的模型中，两个 AS 可以通过多条链路连接；这意味着除了 Gao-Rexford 规则之外，现在还有额外的导入策略规则。当你收到针对同一目的地的多条通告时，请按以下顺序根据这些平局判定规则选择路径：

  1. 使用 Gao-Rexford 规则。优先选择客户通告的路径，其次是对等方通告的路径，最后是提供商通告的路径。
  2. 如果多条路径具有相同的 Gao-Rexford 优先级（例如两条路径都来自客户），则选择较短的路径（经过较少 AS 的路径）。
  3. 如果多条路径长度相同，则选择具有更近出口路由器的路径（使用 IGP 求出到每台出口路由器的距离）。
  4. 如果多条路径到出口路由器的距离相同，则选择 MED 更低的路径（MED 包含在通告中）。
  5. 如果多条路径的 MED 相同，则任意打破平局（例如选择 IP 地址较小的路由器）。

请注意，最近出口路由器（热土豆路由）和 MED 常常相互矛盾。每个 AS 都希望将自己的带宽使用量降到最低，并希望数据包由其他 AS 的带宽承载。

作为导出方 AS，我希望数据包进入我的 AS 时尽可能靠近目的地。这意味着我希望导入方 AS 将数据包运送很远（到出口的路径很长）。

相比之下，作为导入方 AS，我希望尽可能少地运送数据包（到出口的路径很短）。这意味着我希望数据包进入另一个 AS 时尽可能远离目的地（迫使另一个 AS 完成所有工作）。

这种矛盾的一个结果是，互联网上的路径常常是不对称的。如果两台主机来回发送数据包，一个方向上的路径可能与另一个方向上的路径不同。

在这个例子中，对于向东传输的数据包，A 选择西侧出口路由器，并迫使 B 承载大部分路程的流量。在另一个（向西）方向上，B 选择东侧出口路由器，并迫使 A 承载大部分路程的流量。

从根本上说，BGP 允许这种行为，是因为每个 AS 都被赋予了设置自身策略的自主权（这里的策略就是热土豆路由）。

在实践中，有时 AS 会尝试实施更巧妙的策略，诱使其他 AS 将数据包承载得更远。或者，如果你支付额外费用，一个拥有更好带宽的 AS 可能会同意替你把流量承载得更远。

BGP 消息类型与路由属性

回想一下，协议必须规定语法和语义。具体来说，BGP 必须规定所发送和接收消息的结构。BGP 还必须规定路由器收到消息时应该做什么。

BGP 有四种不同的消息类型。打开（Open）消息可用于在两台路由器之间启动会话，使它们彼此通信。保活（KeepAlive）消息可用于确认会话仍然处于打开状态，即使最近没有发送消息。通知（Notification）消息可用于处理错误。我们不会进一步详细介绍前三种消息类型。

我们将重点介绍第四种也是最有意思的消息类型——更新（Update）。这些消息用于通告新路由、更改现有路由，或删除不再有效的路由。

Update 消息包含一个以 IP 前缀表示的目的地。该消息还包含路由属性（route attribute），可用于编码与该 IP 前缀对应的任何有用信息。路由属性是一组名称—值对，其中名称指明属性类型，值指明该属性的取值。一个与网络无关的属性示例可以是：颜色=红色，形状=三角形。属性名是颜色和形状，它们分别对应红色和三角形这两个值。

有些属性是 AS 本地的，只在 iBGP 消息中交换。另一些属性是全局的，可以在 eBGP 通告中发送。

BGP 有许多属性，但我们将重点介绍其中三个重要属性，它们用于编码导入路径时采用的不同平局判定条件。

本地优先级（LOCAL PREFERENCE）属性在特定 AS 内编码 Gao-Rexford 导入规则（最高优先级的平局判定条件）。AS 可以为更偏好的路由（例如来自客户的路由）分配较高的值，为较不偏好的路由（例如来自提供商的路由）分配较低的值。该属性是本地属性，只在 iBGP 消息中携带。该属性不会在 eBGP 通告中发送给其他 AS，因为其他 AS 不需要知道这个 AS 的偏好。

举例来说，假设路由器 E 收到来自 AS#7 的 eBGP 通告，并且路由器 A 知道 AS#7 是一个客户。那么，在 iBGP 消息中，路由器 E 可以将本地优先级设置为 3000（较大的数字）。现在，同一 AS 中的其他每台路由器都知道：路由器 E 可以通过 ASPATH 属性中的路径，以 3000 的本地优先级到达它所通告的目的地。

相比之下，如果路由器 D 收到来自 AS#79 的 eBGP 通告，而该 AS 是一个对等方，那么在 iBGP 消息中，路由器 D 可以设置一个较低的本地优先级值 1000，然后将这条路径（具有较低的本地优先级）分发给 AS 中的其他路由器。

本地优先级的数值是任意的，只有它们的相对排序才重要。在上面的例子中，这些数值可以是 300 和 100，而不是 3000 和 1000，其行为仍然相同。本地优先级数值通常由运营人员手动设置。

ASPATH 属性包含所通告路由沿途经过的 AS 列表（顺序相反）。该属性是全局属性，可以在 eBGP 通告中发送。

例如，一条通告会包含目的地的 IP 前缀（128.112.0.0/16），以及值为 [3, 72, 25] 的 ASPATH 属性。

导入路径时，ASPATH 是第二优先级的平局判定条件。如果两条通告的本地优先级相同（例如两者都来自客户），那么我们会选择较短的路径。ASPATH 告诉我们每条路径的长度，该长度以路径经过的 AS 数量衡量。

如果本地优先级和路径长度都相同，第三优先级的平局判定条件是到出口路由器的 IGP 开销。这个开销存储在路由器的本地转发表中（例如，本地距离向量协议会存储到同一 AS 中每台其他路由器的开销）。

MED 属性编码导出方 AS 的偏好。等价地说，该属性表示从导出路由器到目的地的距离（数值越小越受偏好）。

例如，如果这两个 AS 之间有两条链路，导出方 AS 的两台边界路由器都会通告一条路径。ASPATH 和目的地相同，因为在这两种情况下，通往目的地的 AS 路径都相同。然而，西侧路由器所包含的 MED 属性值会低于东侧路由器。这表示：在可能的情况下，请通过我的西侧路由器转发前往该目的地的数据包（数值较低），因为这台路由器离目的地更近。

如果本地优先级、路径长度以及到出口路由器的距离都相同，第四优先级的平局判定条件就是每条通告中的 MED 数值。

BGP 的问题

BGP 没有内置的安全保证。恶意 AS 可能撒谎并通告一条通往某个目的地的路由，即使该 AS 无法到达那个目的地。恶意 AS 也可能通告一条开销非常低的目的地路由，即使这条低开销路由实际上并不存在。这可能诱使其他 AS 通过该恶意 AS 路由数据包，使攻击者能够丢弃或修改经过该恶意 AS 的数据包。这些攻击称为前缀劫持（prefix hijacking）。人们正在积极研究如何使用密码学保护 BGP，但此类协议尚未得到广泛部署。

BGP 在选择路径时，策略优先于最低开销。此外，由于 BGP 用 AS 数量来衡量路径长度，这个路径长度可能会产生误导（例如，一条所通告路径上的某个 AS 可能包含 2 台路由器，也可能包含 200 台路由器）。这可能导致数据包不总是选择最低开销路径，也使我们很难推断互联网的性能。有些人可能会把这些归为问题，不过它们也可以看作有意为之的设计权衡。BGP 的设计者有意识地选择了优先考虑策略并隐藏 AS 内部拓扑，代价则是牺牲性能。

BGP 的实现很复杂。还有许多我们没有介绍的细微实现细节。即使在已经介绍的主题中，本地优先级或 MED 数值等某些配置也必须由运营人员手动设置，而错误配置可能导致错误路径在整个网络中传播。BGP 配置错误常常会造成互联网中断，人们也正在积极研究用于验证 BGP 是否正确配置的工具。

BGP 需要某些假设（所有人都遵循 Gao-Rexford 规则、AS 图构成层次结构、不存在提供商—客户环路），才能保证可达性和收敛性。如果这些假设不成立（例如某个 AS 选择了违反 Gao-Rexford 的自定义策略），BGP 可能产生不稳定行为，导致路由永远无法收敛，或者出现环路和死胡同。
