域间路由模型

域间路由

回想一下前面讲过的内容，路由是在由多个网络构成的网络中执行的。我们已经了解了可用于实现域内路由（intra-domain routing）的距离向量协议和链路状态协议，它们使数据包能够在本地网络内部传送。

在本节中，我们将建立一个模型，以便定义域间路由协议（inter-domain routing protocol），这类协议可以在不同的本地网络之间传送数据包。我们还会看到域间路由协议与域内路由协议如何结合起来，使数据包能够发送到任意网络中的任意主机。

定义自治系统

我们可以通过定义 自治系统（autonomous system，AS） 来形式化本地网络这一概念。自治系统由一个或多个本地网络组成，并且这些网络全部由同一个运营者管理。例如，在 Google 这样的公司内部，可能有一个供员工计算机使用的本地网络，还有另一个供数据中心使用的本地网络，但这两个网络都由同一家公司控制。运营者可以部署单一的域内路由协议，在这些本地网络中任意机器之间传送消息。有时，术语 域（domain） 会被非正式地用来指代 AS，不过这个术语也用于其他协议，因此只要可能，我们都会使用 AS 这一称呼。

为了思考自治系统之间的数据包路由，我们可以忽略 AS 内部所有单独的路由器和主机，把 AS 当作一个单一实体。然后，我们可以画出一张图，其中每个节点代表一个 AS，两个 AS 之间的边表示它们之间存在连接。这张图有时称为域间拓扑（inter-domain topology）或 AS 图（AS graph）。

自治系统简史

在现实中，一个名为互联网号码分配机构（Internet Assigned Numbers Authority，IANA）的组织管理着互联网中所有现存自治系统的全球列表。要成为一个 AS，你必须向该组织注册并获得唯一的自治系统号（autonomous system number，ASN）。

趣闻：早期，IANA 由 Jon Postel 一人手工管理。这意味着，世界上任何想要注册新 AS 的人都必须请求他的批准。

如今，自治系统的数量已经超过 90,000 个，其中美国拥有的 AS 数量居各国之首。

趣闻：加州大学伯克利分校的 ASN 是 25；考虑到 ASN 数量如此之多，这个编号低得相当惊人。这反映出加州大学伯克利分校在互联网历史的很早时期（20 世纪 80 年代）就获得了 ASN。

AS 的类型

回想一下，在为域内路由建立网络模型时，我们区分了终端主机和路由器。对于域间路由，我们也会进行类似的区分，定义两类 AS。

末梢自治系统（stub autonomous system）存在的唯一目的，是为其本地网络中的主机提供互联网连接。末梢 AS 只代表该 AS 内部的主机发送和接收数据包，不会在不同 AS 之间转发数据包。它们类似于域内路由模型中的终端主机：终端主机只发送和接收自己的数据包，不会转发其他人的数据包。

现实中的末梢 AS 包括非互联网企业（例如，为员工提供网络连接的银行）或大学（例如，为学生和员工提供网络连接的加州大学伯克利分校）。这些组织不负责承载来自其他组织的互联网流量。世界上绝大多数 AS 都是末梢 AS。

相比之下，中转自治系统（transit autonomous system）会代表其他 AS 转发数据包。中转 AS 可以通过接收并转发数据包，在两个不同的 AS 之间承载该数据包。

中转 AS 对应现实中业务包含向其他组织出售互联网连接服务的公司。现实中的中转 AS 包括 AT&T 和 Verizon，你可以向这些公司付费，请它们为你提供互联网连接。AT&T 等一些中转 AS 是全球性的，在世界各地都拥有基础设施。另一些则可能只覆盖特定地区，例如 Sonic 这家互联网服务提供商只转发进出加利福尼亚州的流量。

请注意，中转 AS 仍然可以包含发送和接收自身数据包的终端主机。尽管如此，中转 AS 仍类似于域内路由模型中的路由器：路由器接收其他用户的数据包，并代表用户转发这些数据包。

这些讲义将采用末梢 AS 与中转 AS 的模型，不过现实中的分类有时没有如此明确。例如，Google、Microsoft 和 Amazon 等大型科技公司控制着规模庞大的 AS，它们承载的流量与中转 AS 一样多（甚至可能更多）。由于它们的主要作用是承载进出自身服务的流量（例如，接收 Google 搜索请求并发送搜索结果），所以它们可以被归类为末梢 AS。不过近年来，这些公司也开始提供 AS 之间的流量承载服务，因此也可以说它们同样属于中转 AS。

域间拓扑由商业关系定义

在域间拓扑中，如果两个 AS 交换流量，我们就在它们之间画一条边。是什么促使现实中的两个组织（例如一家本地银行和 Verizon）同意交换流量？AS 图中的边由 AS 之间现实的商业关系定义。

一对 AS 之间可能存在两种关系。

一对 AS 可能处于客户—提供商关系中。在现实中，客户（customer） 付费购买服务，而 提供商（provider） 则收取费用并提供连接。例如，本地银行 AS 可以作为客户，向提供商 Verizon 支付互联网服务费用。

一对 AS 也可能处于对等（peer）关系中。两个对等 AS 通常向彼此发送大致等量的流量。在现实中，两个 AS 可以通过公司之间签署法律合同，同意成为对等方。通常，只要两个方向发送的流量大致相等，双方就会约定不为连接服务相互付费。

带商业关系的 AS 图

我们可以通过向 AS 图添加箭头，把这些关系画进图中。有向边从提供商指向客户。无向边连接两个对等方。请注意，图中可以同时包含有向边和无向边（并非所有边都需要箭头）。

图中的末梢 AS 只充当客户。它们拥有入边，表示谁为它们提供连接。然而，它们没有任何出边，因为它们不向其他方提供连接。

相比之下，图中的中转 AS 是提供商。它们向外的箭头表明，它们正在向其他组织出售连接服务。

请注意，箭头的方向并不能说明数据包的发送方向。事实上，即使沿着一条有向边，数据包也可以双向发送。客户向提供商付费，通常是为了获得向互联网其他部分发送数据包和从中接收数据包的能力。

AS 图是无环的

客户—提供商关系图是无环的。图中不存在完全由有向边组成的环。

这种无环性质源于环在现实世界中的含义。在现实中，一个环意味着 A 向 B 付费，B 向 C 付费，然后 C 又向 A 付费；让钱从某人流出后又回到这个人自己手里并不合理。此外，这个环还意味着 A 向 C 提供服务，C 向 B 提供服务，而 B 又向 A 提供服务。让某人为自己提供连接同样不合理。

打个比方，设想你向加州大学伯克利分校支付课程学费，随后加州大学伯克利分校向加州大学系统支付课程费用，然后加州大学系统又向你支付课程费用。这种商业关系完全说不通！

请注意，无环性质只适用于客户—提供商关系。对等关系形成环并没有问题。例如，A-B、B-C 和 C-A 全部互为对等方是可以的。它们彼此之间都不转账，因此不会产生定义不清的商业关系。

提供商层级与一级 AS

图是无环的，这带来了一个结果：我们可以形成一个提供商层级。换句话说，我们可以排列这些节点，使所有箭头都指向下方。末梢 AS 位于底部，提供商位于顶部。服务从较高的节点流向较低的节点。较低的节点则向上向较高的节点付费。

在层级的最顶端，是一级自治系统（Tier 1 autonomous system），它们没有提供商（没有入边）。每个一级 AS 都与其他所有一级 AS 建立了对等关系。

这一层级结构带来的一个结果是：每个非一级 AS 都至少有一个提供商（入边）。这在现实中很合理，因为你必须向某个主体付费，请它为你提供连接。

在这个层级中，从任意 AS 出发，沿着提供商链向上走，最终总会到达一个一级 AS。这在现实中同样合理。所有一级 AS 彼此互为对等方，正是整个互联网能够连通的原因（而不是出现两个互不连通的子图，代表两个彼此分离、只能与各自一半中的主机通信的互联网）。为了保证存在一条通往图中其他每个 AS 的路径，每个 AS 都必须有一条向上的路径，并最终到达一个一级 AS。

现实中的一级 AS 包括 AT&T 和 Verizon（总部位于美国）、France Telecom 和 Telecom Italia（总部位于欧洲），以及 NTT Communications（总部位于日本）。现实中大约有 20 个一级或接近一级的 AS。这些一级 AS 通常拥有横跨多个大洲的基础设施（例如海底电缆）。

AS 图的层级结构由现实中的商业和政治动机定义。理论上，我们可以画出一张树状 AS 图，让单个一级 AS 位于根节点，并向每个末梢 AS 提供服务。然而，这意味着由单一的现实实体控制全世界的互联网接入，出于政治原因，这可能并不理想。

基于策略的路由

回想一下，在域内路由中，我们的目标是找到有效（无环且无死路）并且良好（成本最低）的路径。

在域间路由中，我们仍然希望路径有效。然而，与域内路由不同的是，域内路由中的路由器彼此之间没有什么特殊差异，而每个自治系统都有自己的商业目标，也与其他 AS 存在各自的关系（例如客户、提供商、对等方）。因此，我们需要重新定义“良好”，使其反映 AS 在现实中的商业目标和偏好。

为了让每个 AS 都能以符合其现实目标的方式承载流量，我们的路由协议将允许每个 AS 设置自己的策略。随后，协议计算出的路径应当正确遵守每个 AS 的策略。

理论上，AS 可以设置自己喜欢的任何策略，不过也确实存在标准惯例（我们接下来会讨论）。以下是 AS 可能设置的一些策略示例：

  • “我不想通过自己的网络承载 AS#2046 的流量。”（定义我将如何处理来自其他 AS 的流量。）
  • “我希望自己的流量由 AS#10 而不是 AS#4 承载。”（定义其他 AS 应当如何处理我的流量。）
  • “除非绝对必要，否则不要让我的流量经过 AS#54。”
  • “工作日我偏好 AS#12，周末我偏好 AS#13。”（策略可以随时间变化！）

路由协议并不关心 AS 为什么会有这些偏好。也许我拒绝承载来自 AS#2046 的流量，是因为它是一家竞争公司，但协议不需要知道这一点。

到目前为止，我们的最低成本路由协议无法支持这些策略。最低成本是一个全局最小化问题，其中每台路由器都在尝试解决同一个问题。相比之下，在基于策略的路由（policy-based routing）中，每个 AS 只关心自己的策略，并不存在一个所有人共同协作解决的全局问题。

路由策略的 Gao-Rexford 规则

尽管我们的路由协议允许每个 AS 设置自己喜欢的任意策略，但在实践中，大多数 AS 都会按照一些称为 Gao-Rexford 规则（Gao-Rexford rules）的标准惯例设置策略。这些惯例基于这样一个假设：现实中的组织喜欢赚钱，不喜欢亏钱。

AS 通常遵循两条宽泛的规则。第一，当一个 AS 可以在多条路由中进行选择时，它会优先把数据包转发给利润最高的下一跳。具体而言，AS 偏好下一跳是客户的路由。如果没有这样的路由，AS 会偏好下一跳是对等方的路由。只有在没有更好路由、不得不如此时，AS 才会选择下一跳是提供商的路由。

这一原则决定了 AS 所选择的路由。你可以把这一原则看作距离向量协议中路径选择的偏好版本。我不会选择自己知道的最短路由，而会选择下一跳能让我赚钱（客户最佳）、帮我省钱（没有客户时选择对等方），并避免让我亏钱（没有客户或对等方时选择提供商）的路由。

第二，AS 只有在获得报酬时才会承载流量。AS 没有动力从事无偿劳动。这一原则决定了 AS 愿意参与哪些路径。你可以把这一原则看作距离向量协议中通告路径的一种限制更严格的版本。我不会向每个邻居通告路由、允许任何人通过我转发数据包，而只会通告那些让我通过转发数据包获得报酬的路由。

第二条原则带来的一个结果是：作为一个 AS，我所承载的流量应当来自客户，或者发往客户。换句话说，对于任何一条经过我的路由，我的邻居中必须有一个是客户。

下面逐一讨论所有具体情形。

我的两个邻居都是客户的路由是好的，因为两个客户都会为我转发数据包而付费。

类似地，一个邻居是客户、另一个邻居是对等方的路由也是好的，因为尽管对等方不向我付费，客户会付费。

一个邻居是客户、另一个是提供商的路由也是好的。乍看之下，这条路径似乎不好，因为客户向我付费，而我随后又向提供商付费。这笔交易难道不会让我赚不到钱，甚至亏钱吗？确实可能如此，但如果我们不参与这些路由，就会成为一个无法为客户提供服务的无用 AS。AS 的职责是为用户提供连接，而参与这些客户—AS—提供商路由，能够解锁更多通向互联网其余部分的路由。

我的两个邻居都是对等方的路由是不好的，因为双方都不会为我转发数据包而付费。

更一般地说，对等方不会在其他对等方之间提供中转。从层级结构的角度考虑，一条路径不应在同一个层级上连续经过多跳。

一个邻居是对等方、另一个是提供商的路由同样不好，因为同样没有任何一方向我付费来转发数据包。

更一般地说，如果一个 AS 拥有对等链路，那么该链路只会承载往返于它自身客户的流量。换句话说，当数据包经由这条对等链路到达 B 时，B 唯一有利可图的选择就是把数据包转发给客户（而不是提供商或另一个对等方）。类似地，来自客户的数据包可以经由对等链路转发（客户付费），但来自提供商和对等方的数据包不能经由对等链路转发（没有人付费）。

类似地，我的两个邻居都是提供商的路由是不好的，因为我必须向双方付费来转发数据包，却没有任何人向我付费做这件事。

Gao-Rexford 规则示例

选择路由的策略（客户最佳，提供商最差），以及通告路由的策略（只通告并参与我的邻居中有一个是客户的路由），将用于我们修改后的协议，以计算遵守每个 AS 策略的路由。我们还没有说明如何计算路由，但给定一条路由后，我们可以检查它是否满足这两项策略。

在这个例子中，假设 D（一个末梢 AS）中的一台计算机想与 E（另一个末梢 AS）中的一台计算机通信。D 和 E 可能想要交换消息（记住，箭头代表客户/提供商关系，而不是数据包的方向）。

流量的一条可能路径是 D、B、A、C、E（从 E 到 D 的消息则沿相反方向）。

在这条路径中，谁向谁付费？由于流量沿 D-B 链路发送，客户（D）必须向提供商（B）付费。类似地，E 必须向 C 付费，而 B 和 C 都必须向 A 付费。

中转 AS A、B 和 C 会同意通告并参与这条路由吗？让我们检查它们各自的邻居。

A 在这条路径上的两个邻居都是客户，所以 A 能够赚钱，并认为这条路径很好。

B 的邻居分别是客户（D）和提供商（A）。B 从客户（D）那里赚钱，因此认为这条路径很好。（请记住，一个客户邻居和一个提供商邻居组成的路径是好的，即使 AS 的净利润为 0，因为它们能带来更广泛的连接。）

类似地，C 至少有一个客户邻居（E），所以它也认为这条路由很好。

B 和 C 也许会选择建立对等关系，而不再同时向 A 付费，这会使 AS 图发生变化：

现在，流量的另一条可能路径是 D、B、C、E。此时 D 仍需向 B 付费，E 仍需向 C 付费。然而，B 和 C 不再需要向 A 付费，而且彼此之间也不付费（对等关系）。

同样，我们可以检查这条路径上的中转 AS，也就是 B 和 C，是否会同意通告并参与这条路由。B 的邻居分别是客户（D）和提供商（C）。B 从客户（D）那里赚钱，因此认为这条路径很好。类似地，C 至少有一个客户邻居（E），所以 C 也认为这条路径很好。

我们刚刚推导出，可以使用两条良好路径从 D 向 E 发送消息。现在，B 必须决定通过路径 B-A-C-E 或路径 B-C-E 进行转发。B 应该选择哪一条路径？根据第一条原则，B 偏好利润最高的路径（而不是最短路径）。在 B-A-C-E 中，下一跳是提供商 A（我们必须向它付费）；在 B-C-E 中，下一跳是对等方 C（无需付费）。因此，B 会选择经过 C 的路径，最终得到路径 D-B-C-E。

注意：有了额外的对等关系，B 和 C 似乎都能省钱，那么为什么不是每个 AS 都建立对等关系来省钱呢？在现实中，建立链路还需要安装物理基础设施（例如在地下铺设电缆），因此在 AS 之间建立新关系需要付出成本，以换取更便宜的路由；两者之间存在权衡。

路由无谷

更一般地说，AS 图中的路径总是无谷的（valley-free）。

从层级结构的角度考虑，如果一条路径包含经由对等链路的横向一跳，那么紧接着的下一跳就必须向下通往客户。下一跳不能再次横向（两个邻居都是对等方），也不能向上通往提供商（邻居分别是对等方和提供商）。

从层级结构的角度考虑，如果一条路径包含从提供商到客户的向下一跳，那么紧接着的下一跳必须继续向下通往它的某个客户。下一跳不能再次横向（邻居分别是提供商和对等方），也不能向上（两个邻居都是提供商）。

如果一条向下链路之后必须跟着另一条向下链路，那么我们可以得出结论：一旦路径中出现向下链路，后续所有链路也必须向下。“谷”是指一条先向下、随后掉头开始向上的路径。路径不能包含谷，因为一旦开始向下，就必须一路向下到达目的地。

总而言之，以下是我们推导出的规则（不过，与其死记这些规则，不如从遵守 AS 金钱偏好的角度理解它们）：

  • 一条向上链路之后可以接对等链路、向下链路或另一条向上链路。（如果上一跳向我付钱，我很乐意把数据包转发给任何人。）
  • 一条对等链路之后只能接向下链路。（如果上一跳不向我付钱，我就需要让下一跳成为向我付钱的客户。）
  • 一条向下链路之后只能接向下链路。（如果上一跳是我正在付钱的提供商，我就需要让下一跳成为向我付钱的客户。）

这些规则意味着路由总是无谷且单峰的。一条路由可以从 0 条或多条向上攀升的链路开始。最终，它会到达唯一的峰顶，并经过 0 条或 1 条对等链路。随后，路由必须开始一路向下到达目的地（不能再横向或向上移动）。

路径不能有谷（先向下再掉头向上）。此外，除峰顶之外，路径在任何位置都不能横向移动。一旦横向移动，就必须掉头向下。不能继续横向或向上行进。

AS 希望拥有自主权和隐私

设计计算域间路由的协议时，我们的协议应尊重每个 AS 的自主权和隐私。

AS 希望拥有自主权（autonomy），也就是能够自由选择自己的任意策略，而无须与其他 AS 协调，也无须担心协议允许哪些策略。在实践中，策略通常遵循我们所述的金钱原则，但协议不应强迫 AS 遵循任何特定策略。

AS 也希望拥有隐私（privacy）。AS 不希望必须明确地告诉网络中的其他方自己的偏好和策略。例如，一个 AS 不应需要明确地告诉所有人，它的邻居是对等方、客户还是提供商。这反映了现实中的商业策略。作为一家公司，你可能不愿向竞争对手透露有关客户和提供商的信息。

请注意，我们对隐私的定义是：AS 不应需要明确地公开其策略。在实践中，AS 仍然需要与网络的其余部分协调，就穿越网络的路径达成一致，因此一定程度的信息泄露不可避免。确实存在逆向工程技术，可以追踪数据包通过网络时所走的路由。

例如，网络中的其他方不可避免地能够发现一个数据包采用的路由。然而，我们的协议不应迫使一个 AS 告诉全世界“相比另一条路径，我更喜欢这条路径”。我们也不应迫使 AS 披露其提供商、对等方和客户分别是谁。
