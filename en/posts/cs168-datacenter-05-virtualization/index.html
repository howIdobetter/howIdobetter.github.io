虚拟化和封装

物理数据中心限制

数据中心以固定且结构化的方式组织。相同的服务器被组织成机架，并且机架以某种固定的拓扑排列。这种方法有一些好处。例如，它为我们提供了一种分配分层地址的自然方法。

然而，当我们考虑如何在数据中心托管应用程序时，数据中心的固定组织有一些缺点。假设 Google 推出了一项新服务，他们希望将其托管在现有的数据中心中。如果我们将该应用程序直接放置在物理服务器上，则必须有人为该应用程序物理安装一台具有自己的 IP 地址的新服务器。如果服务扩展，可能需要安装更多服务器。如果服务器出现故障，我们就必须等待有人来修复它。这里的关键问题是改变物理基础设施很困难，但我们经常想要添加新主机、扩展现有主机以及快速、频繁地移动主机。

将应用程序放置在物理服务器上也会带来扩展问题。假设谷歌的新服务非常轻量级，但需要专用服务器（例如出于安全原因）。我们必须将整个物理服务器分配给这个轻量级服务，并且服务器的大部分计算能力都将未被使用。

这种方法也存在路由问题。假设我们想将服务移动到数据中心大楼的不同部分（例如，因为大楼的一部分正在进行维护）。首先，必须有人实际移动建筑物中的服务器。此外，在我们的分层地址模型中，我们需要为该服务分配一个与其新物理位置相对应的新 IP 地址。理想情况下，应用程序更愿意保留相同的地址，无论其数据中心位置如何。

虚拟化

我们可以利用虚拟化来解决这些问题，赋予应用程序更大的灵活性，同时保持数据中心刚性的物理结构。 虚拟化允许我们在物理服务器内运行一个或多个虚拟服务器。

虚拟服务器给应用程序带来了它们正在专用物理机上运行的错觉。然而，实际上，多个虚拟服务器可能运行在同一台计算机上。当应用程序尝试与硬件（例如磁盘、网卡）交互时，它实际上是在与软件中的管理程序交互。虚拟机管理程序为每个虚拟应用程序提供与真实硬件相同的接口。虚拟机管理程序本身运行在实际的物理硬件上，并且可以将应用程序请求（例如磁盘写入、网络数据包发送）转发到硬件级别。

通过虚拟化，如果我们有一个新的应用程序，我们可以要求虚拟机管理程序为此应用程序启动一个新的虚拟机。虚拟机管理程序在软件中运行，因此无需在物理数据中心安装任何新服务器。同样，我们可以完全通过软件将主机移动到不同的物理机器。

虚拟化允许多个应用程序共享一台物理服务器。应用程序可以彼此分离，并且可以由不同的人管理。这使我们能够更有效地使用数据中心中的计算资源。这也使我们能够在数据中心拥有更多主机。例如，具有 40 台服务器的单个机架可能有 40 多个终端主机。

虚拟交换机

物理服务器有一个网卡和一个IP地址，但我们需要给每个虚拟机一个错觉，它有自己专用的网卡和地址。此外，交换机现在可能有多个虚拟机连接到单个物理端口。

为了管理同一台物理机上的多个网络连接，服务器需要虚拟交换机。该虚拟交换机在服务器上的软件中运行（它不是物理路由器），并执行与真实交换机相同的操作（例如转发数据包）。每个虚拟机都连接到虚拟交换机，而虚拟交换机则连接到网络的其余部分。

注意：交换机通常在专用硬件上运行，以最大限度地提高效率。虚拟交换机可以在通用 CPU 上的软件中运行，因为它们只需要支持一些虚拟机（比交换机通常处理的容量更低）。

底层和覆盖网络

通过虚拟化，我们现在可以在物理服务器上运行虚拟主机。与物理服务器不同，虚拟主机可以快速创建、关闭和更改。

虚拟机不一定使用与物理服务器相同的寻址方案。物理服务器 IP 地址由物理数据中心拓扑（例如 Pod、机架）定义。相比之下，虚拟机 IP 地址通常由一些现实生活中的层次结构（例如国家、组织）定义。特别是，单个物理服务器上的虚拟主机不一定都具有相同的 IP 前缀，因此我们不能使用相同的聚合技巧来扩展。

如果我们试图天真地扩展我们的路由方案以支持虚拟机，我们的转发表将变得非常大、非常快。以前，我们可以这样聚合：“蓝色 Pod 中的所有服务器都具有相同的 IP 前缀，并且它们的下一跳都是 R2。”现在，该蓝色容器中的服务器可以包含数百个虚拟主机，所有主机都具有不同的 IP 地址（没有公共前缀）。我们需要为每个虚拟主机一个单独的转发条目。此外，如果虚拟主机移动到不同的物理机（保持相同的 IP 地址），路由协议将必须重新发现到该虚拟主机的路径。我们能否找到一种方法来避免扩展数据中心以支持每个虚拟机地址？

这里的关键问题是我们现在有两种不同的寻址系统，一种用于虚拟主机，一种用于物理主机。两种寻址方案都在 IP 层工作，但在 IP 层内，我们现在需要考虑两个抽象子层。

底层网络处理物理机之间的路由。底层网络包含数据中心基础设施，例如架顶交换机和主干交换机。底层网络可以很好地扩展，因为我们使用物理数据中心拓扑定义分层地址。

覆盖网络存在于物理拓扑（底层）之上，它只考虑虚拟机之间的路由。在实践中，每个虚拟机通常只需要与网络中的少数其他虚拟机进行通信。因此，覆盖网络可以很好地扩展，因为虚拟机不需要了解所有其他虚拟机。

理想情况下，我们希望这两层分别考虑寻址。底层网络不需要知道虚拟主机地址（否则，它的扩展性会很差）。同样，覆盖网络不需要了解数据中心中的每个物理服务器（每个虚拟机只需要了解其他几个虚拟机）。

如果我们没有告诉底层网络有关虚拟主机地址的信息，那么如果数据中心交换机收到以虚拟 IP 作为目的地的数据包，它将在其转发表中查找，找不到任何虚拟 IP，并丢弃该数据包。我们需要某种方法来弥合覆盖层（虚拟思考）和底层（物理思考）之间的差距。

封装

为了统一覆盖层和底层，我们可以使用与设计互联网时相同的分层和标头策略！

到目前为止，我们已经将 IP 视为单层，每个数据包都有一个 IP 标头，它理解 IP 寻址系统。

现在我们有两个 IP 子层和两个不同的 IP 寻址系统，我们可以在数据包中引入一个额外的标头。例如，我们可以有两个 IP 标头，其中一个标头了解覆盖网络，另一个标头了解底层网络。或者，我们可以将原始的 IP 标头用于底层网络，并为覆盖网络引入一种新类型的标头（与 IP 不同）。

现在，我们的数据包路由策略可以结合覆盖网络和底层网络。假设VM A想要向VM B发送数据包。

  1. VM A 创建一个具有单个 IP 标头的数据包，其中包含 B 的虚拟 IP 地址。（请记住，A 考虑的是覆盖层，并且不知道底层物理 IP 地址。）VM A 将此数据包转发到虚拟交换机（在 A 的物理服务器上）。
  1. 虚拟交换机读取报头，获知B 的虚拟IP 地址。然后，虚拟交换机查找B的虚拟IP地址对应的物理服务器地址。 （我们还没有描述如何做到这一点。）

虚拟交换机添加一个附加的外部标头，其中包含 B 的物理服务器地址。添加标头有时称为封装。

此时，数据包有两个标头。内部标头（高层、覆盖层，由 VM A 添加）包含 B 的虚拟 IP 地址，外部标头（较低层、底层，由虚拟交换机添加）包含 B 的物理服务器地址。

虚拟交换机根据物理服务器地址将此数据包转发到下一跳交换机。

  1. 数据包通过底层网络发送。数据中心中的每个交换机仅查看外部标头（底层、物理服务器地址）来决定如何转发数据包。 （请记住，数据中心交换机根据底层进行思考，并且不知道覆盖虚拟 IP 地址。）
  1. 最终数据包到达目的物理服务器的虚拟交换机。虚拟交换机查看外部标头（底层）并注意到目标物理服务器地址是其自身。

虚拟交换机移除外部标头，露出内部标头。删除外部标头有时称为解封装。

最后，虚拟交换机读取内部标头（覆盖）。这告诉虚拟交换机数据包应该转发到物理服务器上的哪个虚拟机。

在这个过程中，封装让我们能够考虑在两个不同层进行路由。底层能够使用物理服务器地址路由数据包，而无需考虑覆盖。同样，覆盖层中的虚拟机能够发送和接收数据包，而无需考虑如何转发底层中的数据包。虚拟交换机通过将虚拟机地址转换为物理服务器地址并添加和删除额外的底层标头来桥接两层。

带封装的转发表

我们应该在转发表中安装哪些条目来支持封装路由？

虚拟机应安装一条默认路由，将每个数据包转发到物理机上的虚拟交换机。

虚拟交换机需要实现一些额外的功能来桥接两层。特别是，当看到虚拟地址时，应该与相应的物理地址进行封装（添加外层）。转发表包含该服务器上的任何虚拟机可能想要与之通信的每个目标虚拟机的条目。我们可以支持这种规模，因为我们假设虚拟机不需要与数据中心中的所有其他虚拟机通信。与标准路由算法不同，我们不需要任意到任意的路由（我们不需要到每个其他虚拟机的路径）。

虚拟交换机还需要额外的规则来解封装数据包。如果外部（底层）数据包目的地是交换机本身，则应解封装（删除外部标头）并将数据包传递到内部标头中的 VM 地址。此规则随着服务器上虚拟机的数量而扩展，服务器上的虚拟机数量通常足够小，易于管理。

添加这个功能很难吗？幸运的是，虚拟交换机是通过软件实现的，因此添加此功能只需要编写代码（不需要额外的硬件）。但在实践中，封装非常常见，有时甚至会在硬件中实现。

数据中心中的交换机的工作方式与引入虚拟化之前完全相同。转发表仅包含物理服务器地址，我们知道这些可以通过基于物理拓扑的聚合技巧进行扩展。

多租户和专用网络

数据中心由单个操作员管理，但不同的组织可能在该数据中心内运行应用程序。例如，Google 运行的数据中心可能有一些由 Gmail 运行的虚拟服务器，而其他虚拟服务器则由 Google 地图运行。这种在一个数据中心托管多个服务的方法称为多租户。

云提供商还使用数据中心为客户提供虚拟机。例如，亚马逊网络服务（AWS）和谷歌云平台（GCP）允许用户在数据中心启动虚拟机，做他们想做的事情，并在完成后销毁虚拟机。

多租户的一个问题是，我们并不总是希望不同的租户能够相互通信。例如，如果客户请求虚拟机，他们可能无法连接到数据中心中的所有其他虚拟机。

另一个问题是，数据中心的租户在选择地址时不会相互协调。例如，假设我们的数据中心有两个租户：百事可乐和可口可乐。每个租户都会创建自己的专用网络，并在其中为虚拟机分配内部 IP 地址。专用网络仅供数据中心内的主机相互通信，永远不会从公共 Internet 联系这些主机。由于网络是私有的，因此两个租户都可以使用同一专门分配的私有范围中的地址（RFC 1918 地址）。 Pepsi 的专用网络可能有一个 IP 地址为 192.0.2.2 的 VM，而 Coke 的专用网络可能有另一个 IP 地址为 192.0.2.2 的 VM。 （实际上，我们使用私有范围来重用 IPv4 地址，因为我们已经用完了它们。）

从每个租户的角度来看，这都不是问题。 Pepsi 的 192.0.2.2 永远不会与 Coke 的 192.0.2.2 通信，并且这两个主机都无法访问全球 Internet。然而，这对于数据中心来说是一个问题。如果我们使用基于目标的转发，并且看到目标为 192.0.2.2 的数据包，我们不知道该地址指的是哪个虚拟机。

实践中出现重复 IP 地址的原因有两个。首先，数据中心通常无法控制租户分配给其虚拟机的地址。其次，在 IP 中，标准做法是对专用网络使用特定范围，这通常会导致重复地址。

多租户封装

我们可以再次使用封装的思想来解决这个问题。我们可以添加一个新标头，其中包含一个虚拟网络 ID，用于标识特定租户（例如，百事可乐的 ID 为 1，可口可乐的 ID 为 2）。这个新标头不包含转发和路由信息，但它提供了额外的上下文。现在，如果物理服务器具有用于多个租户的虚拟机，它可以将数据包传递到正确的虚拟网络。

当虚拟交换机收到数据包并解开外部（底层）标头时，它会查看我们的新标头来决定该数据包适用于哪个租户。然后，它查看覆盖标头，将数据包转发到属于正确租户的特定虚拟机。

堆叠封装

我们可以多次使用封装的思想，添加多个新的标头来支持虚拟化和多租户。

首先，虚拟机创建一个带有虚拟 IP 目标的标准 TCP/IP 数据包。

在第一个封装步骤中，我们添加一个虚拟网络标头，它告诉我们哪个租户发送了这个数据包。这有助于我们消除使用相同地址的两个租户的歧义，并且还可以防止数据包被发送到不同的租户。

在第二个封装步骤中，我们添加一个底层网络标头，它告诉我们与虚拟 IP 目的地对应的物理服务器地址。

当我们堆叠封装时，抽象层仍然存在。底层网络不需要知道多个租户位于同一数据中心。底层网络仅查看物理服务器地址的最外层标头，并相应地转发数据包。

解封装步骤以相反的顺序进行。目标服务器上的虚拟交换机收到带有两个额外标头的数据包。

在第一个解封装步骤中，我们删除外部底层标头。由于数据包已到达目标物理服务器，因此不再需要这样做。

在第二个解封装步骤中，我们使用虚拟网络标头来决定我们应该考虑哪一组虚拟机。物理服务器可能有多个租户的虚拟机，这有助于缩小范围到单个租户。

最后，我们使用最里面的 IP 标头将数据包发送到正确虚拟网络中的正确 VM。

注意：使用封装时，我们在读取五元组（IP、端口和协议）以跨多个路径进行负载平衡数据包时必须小心。幸运的是，现代路由器硬件擅长解析数据包以了解相关标头位于数据包中的位置，即使插入了额外的标头也是如此。

实际上，存在许多不同的封装协议。我们可以使用 IP-in-IP 来支持两个 IP 标头（一个用于覆盖，一个用于底层）。

MPLS 是一个简单的标头，用于添加标识服务（例如虚拟网络、租户）的标签。这可用于添加多租户封装。

随着数据中心变得越来越流行，许多其他协议（例如 GRE、VXLAN 和 GENEVE）也被开发出来。其中大多数通过 IP 工作，因此这些自定义协议是内部覆盖标头，而常规 IP 是外部底层标头。
