docker配置

在根目录（比如你克隆仓库之后得到了 cmu15-445这个目录，点进去即可，或者说和 build同目录），然后创建目录 .devcontainer ，在这个目录下面创建文件 devcontainer.json 和 Dockerfile。

devcontainer.json配置如下：

{
  "name": "BusTub Dev Env",
  "build": {
    "dockerfile": "Dockerfile"
  },
  "customizations": {
    "vscode": {
      "extensions": ["ms-vscode.cpptools", "ms-vscode.cmake-tools"]
    }
  },
  "remoteUser": "root"
}

Dockerfile配置如下：

FROM ubuntu:22.04

ENV DEBIAN_FRONTEND=noninteractive
RUN apt-get update && apt-get install -y \
    build-essential \
    cmake \
    clang-14 \
    clang-format-14 \
    clang-tidy-14 \
    gdb \
    git \
    libgtest-dev \
    libreadline-dev \
    python3 \
    zlib1g-dev \
    && rm -rf /var/lib/apt/lists/*

RUN update-alternatives --install /usr/bin/clang clang /usr/bin/clang-14 100 && \
    update-alternatives --install /usr/bin/clang++ clang++ /usr/bin/clang++-14 100

WORKDIR /bustub

Task #2 - Disk Scheduler

std::promise

std::promise的头文件为 future，它像一个一次性的单槽管道。

std::promise<int> box;
std::future<int> receipt = box.get_future(); // 只能用一次get_future()

box.set_value(100); // 只能放一次
int value = receipt.get(); // 同时也只能用一次get

std::optional

std::optional的头文件为 optional，它像一个只有一个格子的盒子。

初始化：

std::optional<int> opt_empty; // 初始化为空
std::optional<int> opt_value = 10;
auto opt_none = std::nullopt; // 明确表示为空

检测是否有值：

if (opt_value.has_value()) { /* ... */ }
if (opt_value) { /* ... */ } // 隐式转换为`bool`，最常用。

获取值：

int v1 = opt_value.value(); // 如果为空，抛出 std::bad_optional_access 异常
int v2 = *opt_value; // 类似于指针解引用，如果为空则为未定义。
int v3 = opt_value.value_or(0); // 如果为空，返回值默认为0

值的修改和替换：

std::optional<std::string> name;
name.emplace("Bustup"); // 原地构造，避免多余拷贝
name.reset(); // 清空内容，使其变为 nullopt

Channel的实现

仅仅是建立一个临界资源的存取，这里应该是没有上限的（因为用的是队列），它是线程安全的。

使用：

bustup::Channel<std::optional<DiskRequest>> resquest_queue_;
resquest_queue_.Put(); // 存
resquest_queue_.Get(); // 放

Task #3 - Buffer Pool Manager

RAII(Resource Acquisition IS Initialization)

RAII的核心思想是：将资源的生命周期和对象的生命周期绑定

  • 获取资源：在对象的构造函数中进行
  • 释放资源：在对象的析构函数中自动进行

移动语义

移动语义允许我们将资源（如内存、所有权）从一个对象“移动”到另一个对象，而不是进行昂贵的拷贝。

  1. 右值引用(T&&)：表示一个即将销毁的对象（临时变量）
  2. std::move：强制将一个左值转换为右值，从而出发移动构造函数

移动构造函数：

class MyBuffer {
private:
    int *data;
    size_t size;
public:
    MyBuffer(MyBuffer&& other) noexcept
        : data(other.data), size(other.szie) {
        other.data = nullptr;
        other.size = 0;
    }
}

RAII的简单应用

一个简单的文件包装器，实现RAII不止于移动语义：

class FileHandler {
private:
    FILE *handle;

public:
    FilerHandler(const char *filename) {
        handle = fopen(filename, "r");
        if (!handle) throw std::runtime_error("File open failed");
    }
    FileHandler() {
        if (handle) fclose(handle);
    }

    // 禁止拷贝，放置多个对象维护同一个 handle 导致重复关闭
    FileHandler(const FileHandler&) = delete;
}

结合移动语义的RAII：


class ResourceManager {
private:
    int *data;
public:
    ResourceManager(int size) : data(new int[size]) {
        std::cout << "Get Resource" << '\n';
    }

    ~ResourceManager() {
        if (data) {
            delete[] data;
            std::cout << "Release Resource" << '\n';
        }
    }

    ResourceManager(ResourceManager&& source) noexcept : data(source.data) {
        source.data = nullptr;
        std::cout << "Move Resource\n";
    }

    ResourceManager(const ResourceManage&) = delete;
};

ResourceManager createResource() {
    ResourceManager temp(100);
    return temp; // 触发移动语义，资源从 temp 转移到调用者，没有多余的内存申请
}

为什么说PageGuard是一个RAII对象，系统永远不需要手动对页面的读写锁进行加锁和解锁操作？

因为一个RAII对象必须从一开始就获取所有资源，在销毁时消除所有资源，因此在一开始就会获取锁，在离开作用域之后就会放开锁。

FrameHeader

包含成员变量：

  • frame_id_ : 帧的唯一索引，是 frame_id_，并且是常量
  • rwlatch_ : 读写锁，修改时必须独占，是 std::shared_mutex
  • pin_count_ : 引用计数，当前有多少个线程正在使用这个页面，是 std::atomic<size_t>
  • is_dirty_ : 脏标记，记录内存中的数据是否被修改过，是 bool
  • data_ : 实际数据存储。是一个 std::vector<char> ，大小通常是4KB

方法：

  • GetData() : 返回指向数据的常量指针
  • GetDataMut() : 返回指向数据的指针
  • Reset() : 用0填充数据，强制修改 pin_count_ 为0，is_dirty_设置为 false

ReadPageGuard

包含成员参数：

  • page_id_ : 正在保护的页面号，是page_id_t
  • frame_ : 保存着被保护页的帧，是std::shared_ptr<FrameHeader>
  • replacer_ : 指向驱逐算法的共享指针，是std::shared_ptr<LRUKReplacer>
  • bpm_latch_ : 指向缓冲池锁的共享指针，是std::shared_ptr<std::mutex>
  • is_valid_ : 读保护的有效位，是bool

WritePageGuard

包含成员参数：

  • page_id_ : 正在被保护的页面号，是page_id_t
  • frame_ : 保存着被保护页的帧，是std::shread_ptr<FrameHeader>
  • replacer : 指向驱逐算法的共享指针，是std::shared_ptr<LRUKReplacer>
  • bpm_latch_ : 指向缓冲池锁的共享指针，是std::shared_ptr<std::mutex>
  • is_valid_ : 读保护的有效位，是bool

BufferPoolManager

构造参数说明：

  • num_frames : 缓冲池大小
  • disk_manager : 磁盘管理器
  • k_dist : LRU-K 替换策略中的 后向 k 距离
  • log_manager : 日志管理器

包含成员变量：

  • num_frames_ : 缓冲池大小，为size_t
  • next_page_id : 下一个待分配的页面ID，为std::atomic<page_id_t>
  • bpm_latch_ : 缓冲池管理器互斥锁，为std::shared_ptr<std::mutex>
  • frames_ : 帧头部集合，为std::vector<std::shared_ptr<FrameHeader>>
  • page_table_ : 页表，为std::unordered_map<page_id_t, frame_id_t>
  • free_frames_ : 空闲帧集合，为std::list<frame_id_t>
  • replacer_ : 置换器，这里是 LRU-K 算法，为std::shared_ptr<LRUKReplacer>
  • disk_scheduler_ : 磁盘调度器，为std::unique_ptr<DiskScheduler>
  • log_manager : 日志管理器，为LogManager *
