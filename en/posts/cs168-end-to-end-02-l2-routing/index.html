第 2 层路由 (STP)

带以太网的第 2 层网络

到目前为止，我们已经展示了第 2 层协议在连接有多台计算机的单个链路上运行，但我们可以引入多个链路并完全使用第 2 层构建网络。数据包可以转发，机器甚至可以运行路由协议，所有这些都专门使用第 2 层 MAC 地址。

我们在 IP 层运行的路由协议也可以在第 2 层运行，但缺点是我们无法聚合 MAC 地址。 IP 地址是根据地理位置分配的，但 MAC 地址是根据制造商分配的，因此没有明确的方法来聚合它们。这个缺点就是我们无法仅用第 2 层构建全球互联网的原因。

如果单个本地网络中有多个链路，我们必须确保如果有人广播消息，第 2 层的任何交换机都会将数据包从所有传出端口转发出去。

在具有多个链路的第 2 层网络中，组播变得更加复杂。需要额外的协议，稍后讨论（在专题部分）。

Bonjour/mDNS 是 LAN 上有用的多播示例，它是 Apple 开发的一种协议。在此协议中，所有 Apple 设备（例如 iPhone、iPad、Apple TV）都经过硬编码以加入本地网络上的特殊组。如果您的 iPhone 想要找到附近的设备来播放音乐（例如 Apple TV、Apple 扬声器或 HomePod 或其他名称），iPhone 可以向群组多播一条消息，询问是否有人可以播放音乐。该组中的设备还可以多播响应，说“我是 Apple TV，我可以播放音乐”。有趣的是，该协议实际上还使用多播组中的 DNS 来发送 SRV 记录，该记录将每台机器映射到其功能。

历史记录：在现代互联网中，我们说过术语“路由器”和“交换机”是可以互换的。现在我们有了二层网络的概念，我们可以说交换机只工作在第一层和第二层，而路由器工作在第一层、第二层和第三层。

如果您回到包装和解开标头的图片，我们假设每个路由器都会解析数据包至第 3 层，并通过 IP 将数据包转发到下一个路由器。但是，如果我们有一个具有多个链路的第 2 层网络，交换机只需将数据包传递到第 2 层，然后通过以太网将数据包转发到下一个交换机。

如今，几乎所有交换机都实现了第 3 层，这就是我们交替使用这些术语的原因。从历史上看，以太网早于互联网，这就是交换机和路由器之间存在区别的原因。

二层网络拓扑

就像我们在路由单元中看到的那样，我们可以使用许多不同的拓扑来连接本地网络中的计算机。

我们可以使用单个链接来连接所有计算机，但这效率很低。我们只有一个链路的带宽可供使用。另外，每个人都需要等待轮到发送消息，如果两台计算机同时发送消息，可能会发生冲突。

我们还可以使用全网状网络，它为每对主机提供专用链接，但很难扩展。

就像在第 3 层一样，我们可以引入交换机，通过拓扑将数据包转发到最终目的地。但是，就像在第 3 层一样，这引入了路由问题，交换机需要决定将数据包转发到哪里。

在本节中，我们将探讨一些专为本地第 2 层网络设计的路由协议。我们还将看到一些阻碍这些协议扩展并用于全球第 3 层网络的挑战。

洪泛转发

最幼稚的转发方法是淹没您收到的每个数据包。当交换机收到数据包时，它会将数据包从每个端口发送出去。

作为一个轻微的优化，我们不需要将数据包从接收数据包的端口发送回。

这种幼稚的方法有两个主要问题：

1.浪费带宽。数据包的副本不必要地发送到不需要该数据包的交换机和主机。

  1. 洪泛会导致数据包循环并淹没网络。

学习开关

让我们从第一个问题开始：洪泛数据包浪费带宽。

为了解决这个问题，我们希望填充交换机的转发表，以便它们能够将数据包直接转发到目的地，而不是向各个方向洪泛数据包的副本。

我们可以运行路由算法来填充转发表，但更简单的方法是使用学习交换机。

假设您是路由器 R2。您没有有关完整网络拓扑的任何信息，并且您的转发表为空。北、南、东、西都有港口。

你看到一个数据包从你西边的港口传来。包裹上写着：“从 A，到 B。”从这个数据包中，你可以推断出 A 一定在你的西边。

现在，您可以向转发表添加一个条目：A 的数据包应转发到西方。

这是学习开关背后的关键思想。当您收到传入数据包时，您会得到有关发送者所在位置的线索。您可以使用该信息填充发件人的转发条目。

请注意，传入的数据包不会告诉您有关收件人所在位置的任何信息。在上面的示例中，当您从西港收到“从 A，到 B”时，这不会告诉您有关 B（收件人）在哪里的任何信息。相反，您填充 A 的转发表，以便 A 未来的数据包可以转发到西方。

当您收到更多传入数据包时，您可以开始用更多条目填充转发表。如果您收到的数据包的目的地不在转发表中，您仍然可以通过将数据包从所有端口（传入端口除外）泛洪来转发该数据包。

例如，当您从西端口收到“From A，To B”时，您还没有B的转发表。因此，您应该将该数据包从所有端口（除西端口外）转发出去。

注意：无需将数据包从传入端口（例如西边）发送回，因为之前的交换机/主机（例如您的西边）已经拥有数据包的副本并转发了它（这就是它到达您的方式）。如果您再次将数据包发送回来，之前的交换机/主机只会再次做出相同的转发决策（要么再次泛洪，要么再次转发回给您），并且这种重复转发并不能帮助数据包到达目的地。

总结起来，学习开关有两个规则需要遵循：

  1. 当您收到传入数据包时，更新转发表以将发送者与传入端口关联起来。

  2. 如果目的地在您的转发表中，则将数据包转发到正确的下一跳。否则，将数据包从除传入端口之外的所有端口泛洪出去。

这是学习开关实际操作的示例。考虑这个网络拓扑。所有交换机都是学习交换机，它们的转发表为空。

A 向 B 发送数据包。A 将数据包转发至 R1。

R1 看到从端口 1 传入的数据包“从 A，到 B”。因此，A 必须朝向端口 1。R1 将此映射添加到其转发表中。

R1 不知道 B 在哪里，因此 R1 将这个数据包从所有端口（除了传入端口）泛洪出去。

R2 和 R4 都接收“从 A 到 B”数据包。他们现在都知道 A 在哪里，并将 A 的映射添加到他们的转发表中。他们都不知道 B 在哪里，因此他们将数据包从所有端口（除了传入端口）泛洪出去。

R3 和 R5 都收到“From A, to B”数据包。他们现在都知道 A 在哪里，并将 A 的映射添加到他们的转发表中。他们都不知道 B 在哪里，因此他们将数据包从所有端口（除了传入端口）泛洪出去。

C 收到“从 A 到 B”数据包。 C 检查报头并意识到它不是该数据包的预期接收者，因此 C 丢弃该数据包。

B 收到“从 A 到 B”数据包。 B检查包头并意识到自己是接收者，因此B成功接收并处理该数据包。

接下来，假设 B 向 A 发送一个数据包。首先，B 将数据包转发给 R3。

R3 收到“从 B，到 A”数据包。这为 R3 提供了有关 B 在哪里的线索，因此 R3 将 B 的映射添加到其转发表中。此外，R3 注意到 A 在其转发表中，因此 R3 可以沿着下一跳将数据包转发到 A（而不是洪泛数据包）。

R2 收到“从 B，到 A”数据包。这允许 R2 将 B 的映射添加到其转发表中。 R2 在其转发表中查找并发现 A 的条目，因此它将数据包沿着下一跳转发到 A。

R1 收到“从 B，到 A”数据包。这允许 R1 将 B 的映射添加到其转发表中。 R1 在其转发表中查找并发现 A 的条目，因此它将数据包沿着下一跳转发到 A。

随着发送的数据包越多，转发表中添加的条目就越多，发生的洪泛就越少。

我们需要添加的最后一项功能：安装转发表条目后，我们为其分配一个 TTL。如果 TTL 过期，则删除该条目。这使得被破坏的路由（例如，因为链路、主机或交换机发生故障）过期。例如，如果 B 在上例中离开网络，则 TTL 将确保 B 的所有转发表最终都会过期。

STP 动机：循环

回想一下，洪泛有两个问题：它浪费带宽，并且环路可能使网络不堪重负。学习交换机解决了第一个问题，但没有解决环路问题。

要了解原因，请考虑这种带有环路的拓扑。假设所有交换机都是学习交换机，并且所有转发表都是空的。 A 尝试向 B 发送数据包，并将数据包转发到 R1。

R1 没有 B 的条目，因此它将数据包洪泛到 R2（和 R3）。

R2 没有 B 的条目，因此它将数据包洪泛到 R4。

R4 没有 B 的条目，因此它将数据包洪泛到 R3。

R3 没有 B 的条目，因此它将数据包洪泛到 R1。

R1 没有 B 的条目，因此它将数据包洪泛到 R2，并继续循环。

与此同时，数据包的副本也在另一个方向上循环传播：R1 最初洪泛到 R3，然后洪泛到 R4，然后洪泛到 R2，R2 洪泛到 R1，R1 洪泛到 R3，继续循环。

在整个过程中，交换机为A安装了转发条目，但它们从未获得B的任何条目，因此无限循环永远无法解决。没有人拥有 B 的转发条目，因此每个人在收到数据包时都会洪泛该数据包。

这个问题有时被称为广播风暴，因为网络正被广播流量淹没。

我们如何解决这个问题呢？理想情况下，我们希望“删除”冗余链路，以便拓扑中没有环路。然后，学习切换方法将正常工作，不会出现广播风暴。

注意：另一种解决方案可能是为每个数据包添加 TTL 字段，以便数据包在转发过多后就会过期。不幸的是，以太网报头没有TTL字段，所以这个解决方案无法实现。

注意：另一种解决方案可能是丢弃数据包（如果您以前见过它们）。这需要将某种时间戳或唯一 ID 附加到每个数据包。同样，以太网报头没有相应的报头字段，因此该解决方案也无法实现。

STP：选举根

生成树协议 (STP) 帮助我们禁用链路，以便生成的拓扑没有环路。这将帮助我们避免广播风暴。

请注意，主机不参与此协议。路由器将协同工作以禁用链路并消除拓扑中的环路。因此，在描述此协议时我们将忽略主机。

STP 如何决定禁用哪些链路？让我们首先以网络的全局视角来解决这个问题。然后，我们将考虑交换机如何在没有网络全局视图的情况下交换消息来实现这一目标。

STP的第一步是选举一个根交换机，如下：

每个交换机都分配有一个 ID，其中包含优先级值（由网络运营商手动设置）和交换机的 MAC 地址。

比较两个交换机时，优先级较低的交换机具有较低的 ID。如果优先级相同，则 MAC 地址较低的交换机具有较低的 ID。

根交换机是ID最小的交换机。

如果网络运营商想要选择特定的根，他们可以通过手动设置各个交换机的优先级来实现。或者，操作员可以将所有交换机优先级保留为默认值，这将导致 MAC 地址最低的交换机被选为根交换机。对于这些注释，我们不会讨论哪个根最好；我们会讨论哪个根最好。重要的是其中一台路由器被明确选择为根。

STP：港口国

现在我们有了根交换机，我们将把每台交换机上的每个端口分类为以下三种状态之一：

  1. 指定端口： 这些是远离根的端口（即它们通向距根更远的地方）。

  2. 根端口： 有一个或多个端口指向根（即它们通向更靠近根的某处）。在这些端口中，沿着到根的最低成本路径的端口是根端口。

  3. 阻塞端口： 指向根的所有端口（不是根端口（到达根的最佳方式））都是阻塞端口。

以下是港口国实际行动的一些例子。假设ID按照路由器标签排序。这意味着R1 具有最低的ID，因此它被选为根交换机。

根交换机 (R1) 上的所有端口都指向远离根的端口，因此它们都是指定端口。

R2 有两个端口。其中只有一条指向根，因此这一定是通往根的最佳路径。因此，R2 的朝上端口被标记为根端口。

R2 上的另一个端口远离根，因此 R2 的朝下端口被标记为指定端口。

R6 有三个端口。朝下的端口远离根，因此它是指定端口。

在 R6 处，到 R4 和 R3 的端口均指向根。但是，到 R3 的端口提供到根的最低成本路径（成本 2），而到 R4 的端口提供到根的较差路径（成本 3）。因此，我们将通往 R3 的端口标记为根端口（到达根的最佳路径），将通往 R4 的端口标记为阻塞端口（指向根，但不是最佳路径）。

有时，我们有平局，有两种最好的方法可以找到根源。

例如，在 R4 处，到 R2 的端口和到 R3 的端口都指向根，并且它们都提供到根的成本 2 路径。如果出现平局，我们会说具有较低 ID 的下一跳是到达根的更好路径。这使得到 R2 的端口成为根端口，而到 R3 的端口成为阻塞端口。

有时，我们会有一个链接指向距根同样远的地方。

例如，R4 距根的距离为 2，并且它具有到 R5 的链接，R5 距根的距离也是 2。同样，我们将使用路由器 ID 作为决胜局。如果该链路通向更高 ID 的路由器，我们会说该链路指向远离根的位置。如果链路通向较低 ID 的路由器，我们会说该链路指向根。在此示例中，R4 的右侧端口远离根（通向距离相同但 ID 更高的某个位置），因此它是指定端口。另一方面，R5 的左侧端口指向根（通向距离相同但 ID 较低的某个地方），因此它要么是根端口，要么是阻塞端口。

STP：禁用链接

现在每个端口都已分配了一个状态（指定端口、根端口或阻塞端口），我们准备从网络拓扑中删除环路。

要消除环路，每个交换机只需假装其被阻止的端口不存在即可。换句话说，不要从该端口发送任何用户数据，也不要从该端口接收任何用户数据。

（注意：我们在这里指定用户数据，因为 STP 数据包仍然可以从被阻止的端口发送和接收。这将允许 STP 在拓扑发生变化时重新启用被阻止的端口。）

如果我们停止沿阻塞端口发送用户数据，则任何具有阻塞端口的链接最终都将被禁用。

为什么这有效？我们从具体交换机的角度来思考一下。您的根端口是您到达根的最佳方式。您被阻止的端口也指向根，但它们不是到达根的最佳路径。这意味着被阻止的端口实际上创建了一条通往根的冗余（但更糟糕）路径，因此我们应该禁用该链接。

您可能担心的一个问题是：您阻止了一个端口，但其他人需要使用该禁用的链接将数据包转发给您，然后将其转发到根？幸运的是，这永远不会发生。请记住，您被阻止的端口指向根（即您距离根更远，而另一侧比您更靠近根）。因此，如果另一侧（较近）的交换机将数据包转发给您（较远），它们会将数据包转发到远离根的地方。这意味着我们可以安全地阻止此端口并禁用此链接，而不必担心其他交换机尝试使用该链接作为其根路径的一部分。

相比之下，指定链路无法安全地禁用，因为它们远离根（即另一侧的交换机比您距离根更远）。另一侧的交换机实际上可能想将数据包转发给您，因为您正沿着它们通往根的最佳路径。幸运的是，这也不是问题。尽管您无法安全地禁用此链接，但您可以依靠另一端的开关来禁用该链接（如果他们不需要）。另一侧的交换机比您更远，因此如果这是到 root 的最佳路径（即根端口），他们将保留此链接，或者如果这不是到 root 的最佳路径（即被阻止的端口），他们将禁用此链接。

通过这种策略，每个链接仅被一侧禁用。较远的一侧会问这样的问题：我是否使用此链接作为 root 的最佳路径？如果是，则该链路的端口为根端口。如果不是，则该链接的端口是被阻止的端口。

较近的一侧始终将此链路的端口设为指定端口。这具有将禁用的决定留给另一侧的效果。这很好，因为较近的一侧不知道较远的一侧是否会使用此链接作为其根的最佳路径。

STP：指定端口

旁注：为什么我们称它们为指定端口？到目前为止，我们已经绘制了每个链接都连接两台计算机的网络，但请记住，有时我们可以拥有连接多台计算机的链接。

假设连接两个交换机的链路还连接有许多主机。如果这些主机想要发送或接收数据，它们会将数据发送到指定端口，而不是阻塞端口。 （被阻止的端口不会接收任何用户数据。）这可确保他们的数据仅通过一条路径到达目的地。如果数据同时发送到指定端口和阻塞端口，则数据可能会通过两条路径到达目的地，从而形成环路。

考虑到这一点，指定端口的另一个等效解释是：链路上的主机应向指定端口发送数据以到达根（或生成树上的任何其他位置）。从交换机的角度来看，指定端口指向远离根的端口。从主机的角度来看，发送到指定端口使它们更接近根（或生成树上的任何其他位置）。

STP：BPDU 交换

我们现在知道如何使用 STP 禁用链路并删除网络拓扑中的环路。然而，到目前为止，我们的协议假设网络的全局知识。您需要全局视图来识别根，并确定端口是否指向根或远离根。

为了让交换机了解标记其端口所需的信息，交换机交换称为“桥接协议数据单元 (BPDU)”的消息。这些与我们在其他路由协议中交换的控制平面路由消息几乎相同，但具有一个奇特的名称。请注意，这些控制平面消息与数据平面用户数据包（我们转发的实际数据）不同。

当协议开始时，每个交换机都认为根是它自己，并且根（本身）的成本是0。

当协议运行时，每个交换机都会跟踪它认为的根是什么，以及到达该根的最知名路径（以及该路径的成本）。

当您发送 BPDU 时，您会包含两条信息：您认为根是谁，以及您距离根有多远。例如，BPDU 可能会说：“根是 R2，我可以以 7 的成本到达 R2。”

当您收到 BPDU 时，您会检查它是否有任何“更好”的信息。 BPDU 可能会更好，原因有两个：

  1. BPDU 中的根具有较低的 ID。这意味着你发现了更好的根。您应该放弃当前的根和成本，而采用新的根和通往新根的路径。

  2. BPDU 中的根是相同的，但 BPDU 提供了到根的更好路径。您应该采用新的根路径。

root 的成本的计算就像我们在距离向量协议中所做的那样。例如，假设你的邻居告诉你“根是 R2，我可以用成本 7 到达 R2”。那么您到根的成本就是您到邻居的直接链路成本，加上邻居到根的成本（如广告中指定的）。

当您更新状态（您认为 root 是谁，或者您最了解的 root 成本）时，您应该向邻居发送 BPDU，通知他们您的新状态。

一旦协议收敛，状态就会为每个交换机提供足够的信息来标记其所有端口。您知道到达根的最佳路径，因此可以将相应的端口标记为根端口。

你的邻居也都告诉你他们离根有多远。如果邻居说他们更远，那么您可以将相应的端口标记为指定端口。如果邻居说他们更近（但他们不在通往根的最佳路径上），那么您可以将相应的端口标记为阻塞端口。

BPDU 定期交换，因此如果网络拓扑发生变化，协议可以适应并为新拓扑找到生成树（即禁用链路）。

STP：BPDU 交换示例

路由器并行发送和接收 BPDU 交换，因此不存在发送第一个 BPDU 的特定路由器。在此示例中，我们将显示发送的 BPDU 的子集。

R3 的初始状态表示：根 R3 距离为 0。 R3 的第一个通告将此状态发送给其邻居。

R1 听到这个广告。 R1当前认为根是R1，广告提供的根是R3。所通告的根更差（ID 更高），因此 R1 拒绝该通告。

R6 听到这个广告。 R6当前认为根是R6，而该广告提供了R3的根。通告的根​​更好，因此 R6 接受此通告。 R6 的更新状态显示：根 R3 距离 1。注意：成本是从 0（来自 R3 的通告中的成本）加上 1（到 R3 的链路成本）计算的。

此时，R6 已更新其状态，因此它将向其邻居发送广告（本演示中未显示）。

一段时间后，R1 向其邻居发送通告，其中包含其状态：根 R1 为 0 距离。

R2 听到了这个广告。通告的根​​ (R1) 比当前最知名的根 (R2) 更好，因此 R2 接受此通告。 R2 的更新状态显示：根 R1 距离 1。

同样，R3 听到此通告并接受它，因为通告的根 (R1) 比当前最知名的根 (R3) 更好。 R3 的更新状态显示：根 R1 距离 1。

R2 和 R3 已更新其状态，因此它们将各自向其邻居发送广告。

R4 听到 R2 发来的广告。通告的根​​ (R1) 比当前最知名的根 (R4) 更好，因此 R4 接受此通告。 R4 的更新状态显示：根 R1 距离 2。注意：此成本是通过将 1（来自 R2 的通告成本）加上 1（到 R2 的链路成本）相加来计算的。

R6 听到 R3 发来的广告。通告的根​​ (R1) 比当前最知名的根 (R3) 更好，因此 R6 接受此通告。 R6 的更新状态显示：根 R1 距离 2。注意：R6 的旧状态表示 R3 距离为 1，新状态表示 R1 距离为 2。尽管新状态具有更高的距离，但它仍然更好，因为新状态具有更好的根。

R4 和 R6 已更新其状态，因此它们将以其更新的状态向其邻居发送广告。我们将首先展示 R4 的广告，然后再回到 R6（再次强调，所有这些在现实中都是并行发生的）。

R5 听到 R4 发来的广告。通告的根​​ (R1) 比当前最知名的根 (R5) 更好，因此 R5 接受此通告。 R5 的更新状态表示：根 R1 距离 3（距离广告 2，加上到 R4 的链路成本 1）。

R6 也听到了 R4 的广告。通告的根​​（R1）与当前最知名的根（R1）相同，因此我们需要检查成本。接受广告将产生 2 的成本（来自广告），再加上 1（来自到 R4 的链路成本），总共 3。当前最著名的成本是 2。因此，R6 拒绝广告（3 比 2 更差）。

R5 已更新其状态，因此它将向其邻居发送具有更新状态的通告。

R7 听到 R5 的广告。通告的根​​ (R1) 比当前最知名的根 (R7) 更好，因此 R7 接受此通告。

R7 已更新其状态，并将向其邻居发送通告，但此处未显示该通告（R6 会拒绝它，因为 root 成本更差）。

继之前之后，R6 向其邻居发送通告，R7 接收该通告。通告的根​​（R1）与当前最知名的根（R1）相同，因此我们需要检查成本。接受广告将产生 2 的成本（来自广告），再加上 1（来自到 R6 的链路成本），总共 3。当前最著名的成本是 4。因此，R7 拒绝广告（3 比 4 更好），并将其状态更新为 root 成本为 3（而不是 4）。

路由器继续定期相互交换通告。此演示中并未显示所有广告，但最终协议将收敛，所有路由器都会知道根是 R1。此外，所有路由器都会知道它们对根的成本。

一旦所有路由器都知道它们到商定的根的成本，它们就可以交换定期通告。这允许路由器了解其邻居的根成本值，进而允许路由器将端口分配为 DP、RP 或 BP。
